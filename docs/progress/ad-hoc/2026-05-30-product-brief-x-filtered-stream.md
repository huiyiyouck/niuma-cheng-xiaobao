## 1. 背景

平台当前通过 X API timeline 接口轮询指定账号。轮询可以补抓推文，但无法在推文发布后立即更新平台；轮询频率越高，请求成本和限流压力越大。

本方案将指定账号采集改为：

```text
Filtered Stream 实时监听 + 可配置的低频 timeline 补偿抓取
```

实时流负责秒级接收新推文；补偿抓取负责覆盖服务器重启、网络断开或 X API 短暂异常期间可能遗漏的推文。

## 2. 产品目标

1. 指定账号发布原创推文后，平台能够通过共享的 X Filtered Stream 长连接及时接收并进入现有处理链路。
2. 不收录回复和转推。
3. 每个 X/Twitter Source 独立配置补偿抓取周期。
4. 补偿抓取周期默认 24 小时，可在管理页面修改。
5. 实时流和补偿抓取复用同一去重逻辑，重复推文不会重复生成新闻卡片。

## 3. 用户故事

### US-1：实时跟踪指定账号
作为平台管理员，我希望添加一个或多个 X 账号后，平台自动监听这些账号的新原创推文，以便新闻卡片快速更新。

### US-2：按 Source 配置补偿周期
作为平台管理员，我希望为每个 X Source 单独设置补偿抓取周期，以便重要账号提高补漏频率，普通账号保持较低请求成本。

### US-3：自动恢复
作为平台管理员，我希望服务重启或网络中断后，后台自动恢复 Stream 规则和长连接，不需要登录 X Developer Console 手工重新配置。

## 4. 产品方案

### 4.1 共享长连接

后台维护一个共享 Filtered Stream 长连接，集中监听所有启用的 `x_twitter` 账号追踪 Source。一个账号对应一条规则：

```text
from:OpenAI -is:retweet -is:reply
```

规则 tag 使用平台命名空间：

```text
niuma:x-user:openai
```

### 4.2 收录范围

| 内容类型 | 是否收录 |
|----------|----------|
| 原创推文 | 是 |
| 回复 | 否 |
| 转推 | 否 |
| 引用推文 | 是，作为账号发布的原创内容进入处理链路 |

### 4.3 补偿抓取

每个 X Source 的绑定策略保留调度周期，但页面文案调整为“补偿抓取周期”：

```json
{
  "schedule": {
    "every_seconds": 86400
  }
}
```

- 默认值：`86400` 秒，即 24 小时。
- 配置粒度：每个 X Source 独立设置。
- 页面输入单位：小时。
- 最小值：1 小时。
- 保存时转换为秒写入 `fetch_policy.schedule.every_seconds`。

### 4.4 自动规则同步

后台每 30 秒读取一次启用状态和账号配置。发现账号新增、编辑、启用、停用或解绑后，自动同步 X Streaming Rules 并在需要时重连。

管理员不需要在 X Developer Console 手工维护规则。

## 5. 页面调整

管理页中的 X Source 编辑区域增加或调整以下字段：

| 字段 | 说明 | 默认值 |
|------|------|--------|
| 跟踪账号 | 指定账号用户名，支持多个 | 无 |
| 补偿抓取周期（小时） | Stream 断线或服务重启后的补漏频率 | 24 |

辅助文案：

> 新推文会通过实时流立即接收。该周期仅用于网络中断或服务重启后的补偿抓取。

RSS 等非 X Source 继续显示“抓取周期”，不改变原有语义。

## 6. 验收标准

### 6.1 实时监听
- [ ] 启用 `user_timeline` 模式的 X Source 后，后台自动生成 `from:{username} -is:retweet -is:reply` 规则。
- [ ] 指定账号发布原创推文后，平台通过 Stream 接收并写入 `raw_items`。
- [ ] 推文进入现有 LLM 处理队列并生成新闻卡片。
- [ ] 回复和转推不会进入平台。

### 6.2 规则同步
- [ ] 新增、编辑、启用、停用或解绑 X Source 后，规则在 30 秒内自动同步。
- [ ] 服务重启后自动重新同步规则并恢复 Stream。
- [ ] 多个 Source 跟踪同一账号时，只维护一条 X Streaming Rule，但推文可分发到对应绑定。

### 6.3 补偿抓取
- [ ] 每个 X Source 默认补偿周期为 24 小时。
- [ ] 管理页允许修改周期，保存后只影响当前 Source 绑定。
- [ ] 补偿抓取与 Stream 重复收到同一推文时，只生成一条 `raw_items` 和一条新闻卡片。

### 6.4 异常处理
- [ ] Stream 网络中断后自动指数退避重连。
- [ ] HTTP `429` 时延迟重连并记录日志。
- [ ] HTTP `401/403` 时生成告警并暂停 Stream 重试，避免无效请求循环。
- [ ] 单条 Stream 数据解析失败时记录日志，长连接继续处理后续数据。

## 7. 范围边界

### 本次包含
- `user_timeline` 模式指定账号实时监听。
- 原创推文收录。
- 每个 X Source 独立补偿周期配置。
- 自动规则同步、自动重连、现有入库链路复用。

### 本次不包含
- 关键词搜索模式实时化。
- 回复、转推收录开关。
- X Developer Console 内规则的可视化管理页面。
- 多服务器同时运行多个 Stream Manager 的高可用选主。

## 8. 风险和依赖

1. X API 套餐必须支持 Filtered Stream 和规则管理接口。
2. X API 的连接数、规则数和计费策略可能变化，部署前需再次核实当前套餐。
3. 当前工作区中 `server/` Node.js 主线文件处于未提交删除状态。实施前必须先确认并恢复正确的 Node.js 工作区。
4. 第一阶段按单实例部署设计。若未来水平扩容，需要增加数据库锁或独立 Stream 服务，保证全局只有一个活跃长连接。

## 9. 结论

推荐采用一个共享 Filtered Stream 长连接，并保留每个 X Source 独立配置的低频 timeline 补偿抓取。默认补偿周期为 24 小时。

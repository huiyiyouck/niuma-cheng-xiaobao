## 3. 功能需求

### 3.1 Source 管理

#### 3.1.1 数据模型（保留两层结构）

不新增表，在现有 `sources` 和 `channel_sources` 基础上调整字段。

**字段变更策略：**
- 将现有 `name` 字段重命名为 `display_name`（`ALTER TABLE sources RENAME COLUMN name TO display_name`）
- 移除 `UNIQUE(type, name)` 约束——在用户自主创建 Source 的模型下，同类型同名 Source 应被允许
- 若保留原有 `name` 字段（不重命名），则 `name` 的赋值逻辑为"等于 display_name"（向后兼容），新增 `display_name` 作为用户可见名称

`sources` 表变更后字段：

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `display_name` | VARCHAR(200) | NOT NULL | 用户可见的来源名称（由原 `name` 重命名而来） |
| `status` | VARCHAR(20) | NOT NULL, DEFAULT 'unverified' | unverified / verified / active / error |
| `source_url` | TEXT | NULLABLE | 用户添加的原始 URL（用于自动识别类型） |
| `last_verified_at` | TIMESTAMPTZ | NULLABLE | 最近一次验证时间 |
| `verify_error` | TEXT | NULLABLE | 最近一次验证失败的错误信息 |

状态流转：
```
unverified → (触发验证) → verified → (启用) → active
                ↓                        ↓
            (验证失败)              (抓取持续失败)
                ↓                        ↓
            unverified                 error
```

`sources.config`（JSON 字段，已有）：保留，存各类型特定的配置（如 Twitter 的 search_query、usernames）。

`channel_sources` 表不变（已有 fetch_policy、sub_channel_id、enabled 等字段）。

#### 3.1.2 Source 类型自动识别

系统根据用户提供的 URL 自动识别 Source 类型：

| URL 特征 | 识别类型 |
|---------|---------|
| 包含 `x.com` 或 `twitter.com` | `x_twitter` |
| URL 路径匹配已知 RSS 模式（`/feed/`、`/rss/`、`/feeds/posts/default`），或后缀为 `.xml`/`.rss`/`.atom`，或 HTTP 响应头 Content-Type 为 `application/rss+xml` / `application/atom+xml` | `rss` |
| 包含 `github.com` | `github_trending` |
| 包含 `huggingface.co/papers` | `hf_daily_papers` |
| 包含 `news.ycombinator.com` | `hacker_news` |
| 包含 `semanticscholar.org` | `semantic_scholar` |

> **Content-Type 检测说明**：当 URL 后缀无法判定时，后端发起 HEAD/GET 请求检查响应头 Content-Type。超时时间为 3 秒；超时或请求失败时标记类型为 `unknown`（用户可手动指定）。

自动识别结果在创建表单中展示，用户可手动修改类型。手动指定的类型优先于自动识别。

> **v0.2 实际可用类型**：当前仅 X/Twitter 类型的抓取器代码存在。其他类型虽然可以创建，但抓取时会报错提示"该 Source 类型的抓取器尚未实现"。后续迭代按需逐个添加抓取器。

> **Source type 枚举共享**：`x_twitter`、`rss`、`github_trending`、`hf_daily_papers`、`hacker_news`、`semantic_scholar`、`unknown` 等类型值建议在共享模块（如 `app/schemas.py`）中定义为 `SourceType` 枚举，API、Worker、前端共用，避免字符串散落各处。

#### 3.1.3 Source CRUD API

新增 Source 管理接口（独立于频道空间绑定）：

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/v1/sources` | 列出所有 Source（支持按 status、type 筛选） |
| POST | `/v1/sources` | 创建 Source（body：display_name、source_url、type 可选） |
| GET | `/v1/sources/{id}` | 获取 Source 详情（含验证状态、关联的频道空间） |
| PUT | `/v1/sources/{id}` | 更新 Source 配置（display_name、source_url、config、type） |
| DELETE | `/v1/sources/{id}` | 删除 Source（级联删除关联的 channel_sources、raw_items、processed_news、source_states） |

创建 Source 的请求体：
```json
{
  "display_name": "OpenAI Twitter",
  "source_url": "https://x.com/OpenAI",
  "type": null,
  "config": {}
}
```

- `type` 为 null 时，后端根据 `source_url` 自动识别
- `type` 非 null 时，以手动指定的值为准
- `config` 为自由 JSON，各类型的校验在 Worker 抓取时进行（与 v0.1 策略一致）

> **删除级联范围**：删除 Source 时受影响的数据——`channel_sources`（CASCADE）、`tasks`（CASCADE via channel_sources）、`raw_items`（CASCADE via channel_sources）、`processed_news`（CASCADE via raw_items）、`source_states`（CASCADE via channel_sources）。此操作为硬删除，不可恢复，前端需二次确认。实现前需验证以上所有 FK 的 ON DELETE 行为，设计文档中应列出当前 schema 的 FK 级联状态并补充缺失的 CASCADE。

#### 3.1.4 Source 验证机制

验证是一个手动触发的**测试抓取**，用于确认 Source 配置正确、可正常获取数据。

**架构路径**：验证端点在 API 层实现。API 层直接导入并使用各 Source 类型的抓取函数（如 `worker/fetch_x_twitter.py` 中的独立抓取函数 `execute_fetch(config, max_items)`）。这要求将 Worker 中每种 Source 类型的 HTTP 调用 + 解析逻辑抽取为 `worker/fetch_*.py` 中的可复用异步函数，API 和 Worker 共用。架构师在设计文档中确认抽取方案。

**验证流程：**
- 端点：`POST /v1/sources/{source_id}/verify`
- 行为：以该 Source 的配置执行一次限量抓取（每次验证最多拉取 5 条），将结果返回给前端展示，不入库 `raw_items` 表
- 如果 Source 尚未绑定到任何频道空间，验证时使用临时参数（如 max_items=5）
- 如果 Source 已绑定到频道空间，使用该绑定的 fetch_policy（但 max_items 仍限制为 5）
- 验证成功：更新 `sources.last_verified_at`，返回抓取结果的预览数据。**不自动修改 status**——用户需查看预览后手动决定是否标记为 verified
- 验证失败：更新 `sources.verify_error`，返回错误信息

**超时处理：**
- 外部服务调用超时时间为 15 秒
- 超时时返回 `{"status": "error", "error": "外部服务超时（15s）", "items": []}`
- 前端在验证期间展示 loading 状态，不阻塞其他操作（弹窗内 spinner，可关闭弹窗取消等待）

端点响应（成功时）：
```json
{
  "status": "ok",
  "items": [
    {
      "source_item_id": "...",
      "source_item_url": "...",
      "title": "...",
      "content_preview": "...前 200 字符或全文（取较小者）",
      "published_at": "..."
    }
  ],
  "total_fetched": 5
}
```

端点响应（失败时）：
```json
{
  "status": "error",
  "error": "X API 返回 401：Bearer Token 无效",
  "items": []
}
```

标记验证通过：
- 端点：`POST /v1/sources/{source_id}/mark-verified`
- 将 `status` 从未验证改为 `verified`
- 幂等性：对已 `verified` 或 `active` 的 Source 调用此端点，直接返回 200（不报错，不重复操作）。对 `error` 状态的 Source 调用此端点视为异常（状态机不允许从 error 直接跳到 verified），返回 409
- 此时 Source 可以被启用（enable）参与正式抓取

**Source 编辑后验证状态规则：**
- 修改 `source_url` 或 `type`：自动将 status 重置为 `unverified`，清空 `last_verified_at` 和 `verify_error`
- 仅修改 `display_name` 或 `config`：维持现有 status 不变

#### 3.1.5 Source 列表响应

`GET /v1/sources` 返回的每条 Source 包含：

```json
{
  "id": "uuid",
  "display_name": "OpenAI Twitter",
  "source_url": "https://x.com/OpenAI",
  "type": "x_twitter",
  "status": "active",
  "config": {...},
  "last_verified_at": "2026-05-24T10:00:00Z",
  "verify_error": null,
  "channel_spaces": [
    {
      "channel_space_id": "uuid",
      "channel_space_name": "AI 资讯",
      "sub_channel_id": "uuid",
      "sub_channel_name": "AI 前沿",
      "enabled": true
    }
  ],
  "created_at": "2026-05-24T09:00:00Z"
}
```

> **channel_spaces 说明**：一个 Source 可以绑定到多个频道空间（尽管当前 `UNIQUE(channel_space_id, source_id)` 约束同一空间只能绑一次）。`channel_spaces` 数组列出所有绑定关系及子频道信息，未绑定时为空数组。

#### 3.2.3 数据库清理

执行数据清空脚本，清理 Source 和新闻相关数据表（**保留** `channel_spaces` 和 `sub_channels` 表数据）：

```sql
-- 按外键依赖顺序清空（channel_spaces 和 sub_channels 保留）
DELETE FROM processed_news;
DELETE FROM raw_items;
DELETE FROM tasks;
DELETE FROM source_states;
```

**首次使用引导**：数据清空后，前端应处理空状态场景：
- 新闻列表页：提示"暂无新闻，请先在管理页添加信息来源"

#### 3.3.7 统计卡片真实数据

| 统计项 | 数据来源 | 计算方式 |
|--------|---------|---------|
| 总新闻数 | `processed_news` 表 | `COUNT(*) WHERE channel_space_id = ?` |

### 阻断项（必须在 R2 解决，否则无法进入设计阶段）

**#1 Source 验证机制的架构路径不明确（3.1.4）**

`POST /v1/sources/{source_id}/verify` 需要实际执行一次抓取（调用 X API 等外部服务）并同步返回结果。但当前架构中，抓取逻辑位于 Worker 进程（`worker/main.py` 的 `fetch_and_ingest`），API 层不持有抓取能力。PRD 未说明 API 如何触发测试抓取。

三种可行方案及影响：

| 方案 | 优点 | 缺点 |
|------|------|------|
| A. API 直接导入 fetch 函数（如 `parse_x_twitter`）并同步执行 HTTP 调用 | 同步返回，前端简单 | API 与 Worker 代码耦合；需将 HTTP 调用逻辑从 `fetch_and_ingest` 抽取为可复用函数 |
| B. API 创建临时 Task（type='verify'），Worker 执行后写回结果，API 轮询或前端轮询 | 与现有任务模型一致 | 异步模式，前端需轮询；延迟不可控 |
| C. Worker 暴露内部 HTTP 端点供 API 调用 | 解耦清晰 | 引入服务间通信，增加部署复杂度 |

**建议**：方案 A（抽取共享抓取函数）。将 `worker/main.py` 中每种 Source 类型的 HTTP 调用 + 解析逻辑抽取为 `worker/fetch_*.py` 中的独立异步函数（如 `fetch_x_twitter.execute_fetch(config, max_items)`），API 和 Worker 共用。这是改动最小且与现有模块边界最兼容的方案。

**请在 R2 中明确验证机制的架构路径。**

---

**#2 `sources.display_name` 与现有 `name` 字段的关系不清（3.1.1）**

当前 `sources` 表已有 `name` 字段（`TEXT NOT NULL`），且有 `UNIQUE(type, name)` 约束。PRD 新增 `display_name` 字段，但未说明：

- `name` 是保留、重命名还是废弃？
- 若 `name` 保留，用户创建的 Source 的 `name` 值从哪里来（自动生成？等于 display_name？）
- `uq_sources_type_name` 约束在用户自由创建 Source 的场景下是否仍然合理？（两个用户可能创建同类型同名 Source）

**建议**：
- 将 `name` 重命名为 `display_name`（而非新增字段），同时将 `UNIQUE(type, name)` 改为非唯一约束或直接移除——在用户自主创建 Source 的模型下，同类型同名不应被禁止。
- 若确实需要新增 `display_name` 而保留 `name`，请明确 `name` 的赋值逻辑和约束变更。

**请在 R2 中明确字段策略。**

### 中等问题（建议在 R2 中回应或确认，设计阶段处理）

**#3 Worker scheduler 未感知 Source status（3.1.1, 3.1.4）**

`scheduler_loop` 当前查询 `channel_sources WHERE enabled=true` 来创建 fetch 任务，未 JOIN `sources` 表。按 PRD 设计，Source 状态机为 `unverified → verified → active`，只有 `active` 状态的 Source 才应参与正式抓取。若 `channel_source.enabled=true` 但 `sources.status='unverified'`，Worker 仍会尝试抓取。

**建议**：在 R2 中明确 scheduler_loop 的过滤逻辑——是 JOIN sources 加 `status='active'` 过滤，还是在 `POST /v1/channel-sources/{id}` 启用时校验 Source 必须为 verified/active。两种方案二选一即可，但须在设计文档中明确。

---

**#4 数据全清范围：channel_spaces 也被清空（3.2.3）**

清理脚本中包含 `DELETE FROM channel_spaces`。用户故事 2.1.1 说的是"清空全部旧 Source 数据"，但清空 channel_spaces 意味着所有频道空间和子频道也被删除。用户可能期望保留频道空间结构（如"AI 资讯"空间及其子频道），仅清理 Source 和新闻数据。

**建议**：将 channel_spaces 和 sub_channels 的清理改为可选（默认保留频道空间结构），或将此决策明确标注为"重置到出厂状态"并在前端给用户选择权。

---

**#5 RSS 类型自动识别有大量漏判（3.1.2）**

URL 后缀匹配（`.xml`、`.rss`、`.atom`）遗漏了大多数实际 RSS feed 路径，如 `/feed/`、`/rss/`、`?feed=rss2` 等。Content-Type 检测需要发出 HTTP 请求，增加了 Source 创建的延迟。

**建议**：在设计文档中补充 Content-Type 检测的超时策略（如 3 秒超时）和失败降级行为（超时或请求失败时默认 RSS 还是标记为 unknown）。同时补充已知的 RSS 路径模式列表（`/feed/`、`/rss/`、`/feeds/posts/default` 等）。

---

**#6 Source 编辑后验证状态的处理缺失（3.1.3, 3.1.4）**

当用户编辑已 verified/active 的 Source 的关键配置（`source_url`、`type`、`config`）时，PRD 未定义是否应重置 status。例如：一个 active 的 X/Twitter Source 被修改了 `source_url` 指向完全不同的站点——其 status 仍为 active 显然不合理。

**建议**：在 R2 中补充规则：修改 `source_url` 或 `type` 时自动将 status 重置为 `unverified`，清空 `last_verified_at` 和 `verify_error`；仅修改 `display_name` 或 `config` 时维持现有状态。或者统一策略为"任何修改后需重新验证"。

---

**#7 日志查看器 `/v1/admin/logs` 的架构考虑（3.4.3）**

API 直接读取磁盘上的日志文件（`logs/api.log`、`logs/worker.log`），存在以下问题：
- 分页参数 `offset` 暗示从文件头顺序读取，但对于日志查看场景，用户通常想看最新日志（tail），从文件尾反向读取更实用
- 日志轮转期间（按天轮转），读取的文件可能被重命名，需处理文件不存在的情况

**建议**：在设计文档中补充：日志查询默认按时间倒序（最新在前），offset 语义改为"从末尾倒数第 N 行"；Worker 和 API 日志文件的发现机制（配置文件路径 vs 自动扫描 logs/ 目录）。

---

**#8 Source DELETE 级联遗漏 tasks 表（3.1.3）**

PRD 列出的级联删除范围为 `channel_sources → raw_items → processed_news` 和 `source_states`。但 `tasks` 表也通过 `channel_source_id` 外键关联 `channel_sources`（`ON DELETE CASCADE`）。删除 Source 后会通过 channel_sources 级联清理 tasks。

**建议**：在 3.1.3 的删除级联范围说明中补充 `tasks` 表，使文档完整。

### 阻断项（必须在 R2 解决，否则无法进入设计阶段）

**#1 Source 验证机制的同步调用超时和错误处理未定义（3.1.4）**

`POST /v1/sources/{source_id}/verify` 需要实际调用外部服务（如 X API）。外部 HTTP 调用可能耗时 5-30 秒甚至超时。PRD 未定义：

- API 端的 HTTP 调用超时时间（X API 可能在负载高时延迟显著）
- 超时后返回什么？（当前只定义了成功和"X API 返回 401"两种响应，没有超时场景）
- 前端在等待期间的 UI 状态（loading spinner 持续多久？是否需要"取消验证"按钮？）

**建议**：在 R2 中补充：超时时间（建议 15 秒）、超时响应格式（与验证失败一致，error 字段写明"外部服务超时"）、前端 loading 状态说明。

**理由**：这直接决定了 API 端实现验证端点的可用性。同步验证如果没有超时保护，一个慢响应可能阻塞整个 API worker。

---

**#2 数据全清后前端启动路径不明确（3.2.3 + 3.3）**

清空脚本删除了所有 channel_spaces 和 sub_channels（共 7 张表的全部数据）。这意味着前端首次加载时：

- 新闻列表页无任何 channel_space 可选
- 管理页无子频道可管理
- Source 管理页无频道空间可绑定
- 统计卡片全为 0

当前前端是否处理了"空状态"（empty state）？用户是否需要按特定顺序操作（先创建频道空间 → 子频道 → 添加 Source → 验证 → 绑定 → 启用）？PRD 未定义这个引导流程。

**建议**：在 R2 中补充"首次使用引导流程"——用户清空数据后第一步做什么、前端应展示什么引导提示。或者明确将 channel_spaces 和 sub_channels 从清空范围中移除（保留空间结构），与架构师 #4 的建议一致。

**理由**：没有引导流程，前端会呈现一片空白，用户不知道该从哪开始。这会影响"从零开始可控来源体系"的核心用户体验。

### 中等问题（建议在 R2 回应或确认，设计阶段处理）

**#3 删除旧抓取器后 Worker 调度代码中引用路径的清理顺序需明确（3.2.1 + 3.2.2）**

删除 5 个 `worker/fetch_*.py` 文件后，`worker/main.py` 中的 `import` 语句和 `if/elif` 分支需要同步移除。但当前 `worker/main.py` 的 `fetch_and_ingest` 函数中，不同 source type 的分发逻辑和文件导入是紧耦合的——删除文件而不删除 import 会导致 `ModuleNotFoundError`。

这是一个实现细节问题（不是设计问题），但需要在设计文档中明确清理步骤的顺序：先删分发分支 → 再删 import → 最后删文件，还是反过来。同时需要 grep 确认 `worker/` 和 `app/` 中没有其他文件引用这些被删除的模块。

**建议**：在设计文档中列出完整的引用路径清理 checklist，确保没有遗漏的 import。

---

**#4 Source type 枚举值的统一定义缺失（3.1.2）**

`x_twitter`、`rss`、`github_trending` 等类型值在以下位置使用：

- API 层：`POST /v1/sources` 的 type 字段校验
- Worker 层：`fetch_and_ingest` 的 source type 分发
- 前端：Source 创建表单的类型下拉、类型展示标签
- 数据库：`sources.type` 列、`uq_sources_type_name` 约束

当前这些值以硬编码字符串散落在各处，容易出现拼写不一致（如 `x_twitter` vs `x-twitter` vs `twitter`）。

**建议**：在 `app/schemas.py` 或新建 `shared/types.py` 中定义 `SourceType` 枚举，API 和 Worker 共享引用。前端用枚举值列表渲染下拉选项。与架构师 #12 的意见一致，从可实现性角度再次强调。

---

**#5 Source 删除的级联范围——需要验证现有 FK 约束（3.1.3）**

PRD 列出删除级联范围为 `channel_sources → raw_items → processed_news` 和 `source_states`。架构师 #8 补充了 `tasks` 表。但实现前需要验证：这些表的 FK 是否已定义 `ON DELETE CASCADE`？

以 `processed_news` 为例——它通过 `raw_item_id` 关联 `raw_items`。如果 `raw_items` 的外键是 `ON DELETE CASCADE`（对 channel_sources），那删除 source 时通过 channel_sources 可以级联到 raw_items，但 processed_news 的 `raw_item_id` FK 需要确认是否也有 CASCADE。

**建议**：设计阶段检查 `db/schema.sql` 中所有相关 FK 的 ON DELETE 行为，如有缺失则在迁移脚本中补充。这直接决定 DELETE Source 的实现是单条 SQL 还是需要手动逐表清理。

---

**#6 前端日志查看器的日志文件读取策略需明确（3.4.3）**

`GET /v1/admin/logs` 的 `offset` 参数按 PRD 语义是"从文件头顺序读取"。但对于日志查看场景，用户想看的是最新日志（等价于 `tail -f`），而不是从第 1 行开始翻页。

从实现角度，读取文件末尾比读取文件头更复杂：
- `limit=100 offset=0` 应该返回最新 100 条还是最旧 100 条？
- 自动刷新时如何避免重复返回相同日志行？

**建议**：与架构师 #7 一致——明确 offset 语义为"从末尾倒数第 N 行"，默认返回最新日志。自动刷新时使用 `(to, from)` 时间窗口参数避免重复。建议设计文档补充具体实现策略。

### 轻微问题（可在设计阶段或实现阶段处理）

**#7 前端管理页面功能密度高，建议标注优先级（3.3.3 + 3.3.8）**

Source 管理页面（3.3.8）和子频道管理（3.3.3）合计包含：列表、新增表单、验证弹窗、确认弹窗、编辑表单、内联重命名、拖拽排序、启用开关、绑定选择器、删除确认。这在前端是较大的工作量。

建议在 PRD 中标注哪些功能是本迭代 MVP（如：Source 增删改查 + 验证流程），哪些可以做简化版（如：内联重命名可以用弹窗编辑替代，拖拽排序可以先用数字输入 sort_order 替代）。不是要砍功能，而是让设计文档和开发排期有优先级参考。

**#8 验证端点中 content_preview 长度对短内容意义不大（3.1.4）**

推文仅 280 字符，"前 200 字截断"对其无实际效果。建议改为"前 200 字符或全文（取较小者）"，与架构师 #9 一致。

**#9 `mark-verified` 端点的幂等性未定义（3.1.4）**

与架构师 #10 一致。对已 verified/active 的 Source 调用 mark-verified 的行为未定义。建议定义为幂等（直接返回 200），实现成本低且前端无需判断状态。

**#10 子频道删除提示"关联新闻失去子频道归属"的行为需澄清（3.3.3）**

删除子频道时，"关联的新闻将失去子频道归属"的具体行为是什么？是 `processed_news.sub_channel_id` 被设为 NULL，还是新闻被一并删除？PRD 仅提到提示文字，未定义数据层行为。当前 schema 中 `fk_news_sub_channel` 可能是 `ON DELETE SET NULL` 或 `ON DELETE CASCADE`，两者用户体验差异很大。

**建议**：明确为 SET NULL（新闻保留，仅失去子频道标签），并在设计文档中确认 FK 行为与此一致。

### 可实现性总结

整体可实施。4 个板块（Source 重构、前端体验、日志系统、代码清理）的技术边界清晰，与 v0.1 代码基兼容。架构师识别的 2 个阻断项（验证架构路径、字段策略）从开发角度也构成实质阻塞。

开发侧额外识别的 2 个阻断项（验证超时处理、空状态引导路径）需要在 R2 中明确，否则实现阶段会陷入反复确认。中等和轻微问题共 8 项，可在设计阶段逐步细化，不阻塞设计阶段的启动。

**全栈开发评审结论**：❌需修改（R1），2 项阻断 + 4 项中等 + 4 项轻微，共 10 条意见。

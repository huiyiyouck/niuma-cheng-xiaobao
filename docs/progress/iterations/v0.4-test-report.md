## 缺陷详细分析

### #B1 🔴阻塞 — alerts.status 列缺失

**复现步骤**：
1. 启动服务，调用 `POST /v1/alerts/acknowledge-all`
2. 或调用 `PATCH /v1/alerts/:id` body=`{"status":"acknowledged"}`

**实际结果**：`{"detail":"column \"status\" does not exist"}`

**根因**：Drizzle schema (`server/src/db/schema.ts`) 已定义 `status` 列，但 `drizzle-kit push` 未执行到生产数据库。项目无独立的 SQL 迁移文件。

**临时修复**：直接执行 SQL：
```sql
ALTER TABLE alerts ADD COLUMN IF NOT EXISTS status VARCHAR(20) NOT NULL DEFAULT 'active';
```

**正式修复建议**：
1. 创建迁移文件 `server/drizzle/0001_v0.4_alerts_status.sql`
2. 在部署文档/启动脚本中加入自动迁移步骤
3. 或使用 Drizzle Kit 的 `push` 命令作为部署流程的一部分

### #B2 🔴阻塞 — Source 删除 FOR UPDATE + LEFT JOIN 不兼容

**复现步骤**：
1. 调用 `DELETE /v1/sources/:id`（该 Source 有 channel_sources 绑定）

**实际结果**：`{"detail":"FOR UPDATE cannot be applied to the nullable side of an outer join"}`

**根因**：`server/src/api/routes/sources.ts:163-168`：
```typescript
const { rows: bindings } = await client.query(
  `SELECT cs.channel_space_id, csp.name as channel_space_name, sc.name as sub_channel_name
   FROM channel_sources cs
   JOIN channel_spaces csp ON csp.id = cs.channel_space_id
   LEFT JOIN sub_channels sc ON sc.id = cs.sub_channel_id   ← 问题在这里
   WHERE cs.source_id = $1
   FOR UPDATE`,    ← FOR UPDATE 与 LEFT JOIN 可空侧冲突
  [source_id],
);
```

PostgreSQL 的 `FOR UPDATE` 不能锁定 LEFT JOIN 结果中为 NULL 的行。当 `cs.sub_channel_id` 为 NULL 时（Source 直接绑定到频道空间而非子频道），对应的 `sc` 行为 NULL，FOR UPDATE 无法锁定。

**修复方案**：使用 `FOR UPDATE OF cs` 仅锁定 `channel_sources` 表：
```sql
SELECT cs.channel_space_id, csp.name as channel_space_name, sc.name as sub_channel_name
FROM channel_sources cs
JOIN channel_spaces csp ON csp.id = cs.channel_space_id
LEFT JOIN sub_channels sc ON sc.id = cs.sub_channel_id
WHERE cs.source_id = $1
FOR UPDATE OF cs
```

或分两步查询：先 `SELECT ... FOR UPDATE` 锁定 channel_sources，再 JOIN 获取显示名称。

**影响范围**：`DELETE /v1/sources/:id` 完全不可用。`DELETE /v1/channel-sources/:id`（解绑）不受影响（解绑不需要 FOR UPDATE）。

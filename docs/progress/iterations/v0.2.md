1. **实现轮次偏多（5 轮）**：R4（verified 绑定按钮遗漏）和 R5（数据清理脚本遗漏）属于可避免的疏忽。如果开发在提交前对照 PRD 验收标准 + UI 规范逐项自检，这两项可在 R1 就位。

2. **PRD 初版质量**：R1 收到 23 条意见，其中 4 条阻断（验证架构、字段策略、超时处理、空状态引导）如果在 PM 内部预审时多花一轮自查，可以减少一轮外部 Review。
   - **建议**：PM 产出 PRD 初版后，先以"架构师视角"和"开发视角"各预审一遍，再提交外部 Review。
3. **跨文档一致性**：PRD 6.3 节与 3.2.3 节的数据清理范围矛盾直到 R2 才被发现（架构师发现）。多文档引用同一事实时，修改一处容易遗漏另一处。
   - **建议**：避免在多处重复同一事实。PRD 正文定义规则，附录/摘要引用章节号而非复制内容。
4. **一天内压缩全流程**：PRD 产出→R1→R2 定稿→设计产出→R1→R2 定稿→实现 R1-R5 全部在 2026-05-24 一天内完成。高强度压缩下质量保持不错，但 5 轮实现的"回头率"可能与疲劳有关。后续迭代如时间允许，建议至少跨两天。

**#2 `_source_out_legacy` 兼容适配器无效（轻微）**

`app/routes.py:59-63`：
```python
def _source_out_legacy(r: Source) -> SourceOut:
    d = _source_out_v2(r).model_dump()
    d["name"] = r.display_name
    return SourceOut.model_validate(d)
```
意图是让旧代码通过 `source.name` 仍能访问显示名称。但 `SourceOut`（`app/schemas.py:59-68`）仅定义了 `display_name` 字段，Pydantic v2 默认忽略 extra fields——`d["name"]` 在 `model_validate` 时被静默丢弃，返回的 `SourceOut` 对象没有 `name` 属性。

好在所有前端组件已统一使用 `display_name`（`SourceManager.vue`、`BindSourceModal.vue` 等），未发现任何 `source.name` 引用。当前不会触发 bug，但 `_source_out_legacy` 与 `_source_out_v2` 功能完全等价，`d["name"] = r.display_name` 是死代码。

建议两个方向之一（可在 R2 或后续迭代处理）：
- 简单方案：删除 `_source_out_legacy`，`list_channel_sources` 直接使用 `_source_out_v2`
- 如果需要真正兼容：在 `SourceOut` 中增加 `name` 属性（computed field 或 alias），但这会污染新 Schema

**架构层面的补充确认：**

**#A1 `GET /sources/detect-type` 路由位置正确（通过）**

新增的 `GET /sources/detect-type` 定义在 `GET /sources/{source_id}` 之前（routes.py 第 190 行 vs 第 198 行）。FastAPI 按定义顺序匹配路由，字面路径 `detect-type` 优先于参数化路径 `{source_id}`，不会发生路由劫持。`detect-type` 不是合法 UUID，即使顺序相反也会被 FastAPI 的类型检查拒绝而 fall through，但当前顺序是更安全的做法。

**#A2 多选 OR 逻辑的后端实现一致（通过）**

三个多选场景的后端处理方式一致且正确：

| 场景 | 前端传参 | 后端解析 | 查询方式 |
|------|---------|---------|---------|
| 子频道筛选 | `subChannelId: "uuid1,uuid2"` (string) | `split(",")` → UUID 列表 | `IN (...)` |
| 日志级别筛选 | `level: "INFO,ERROR"` | `split(",")` → set | `IN (...)` |
| 日志来源筛选 | `source: "api,worker"` | `split(",")` → 文件列表 | 多文件合并 |

三处均使用逗号分隔的字符串传递多值，解析逻辑一致，与 UI 规范 §2（子频道多选取并集）和 §6（Chip 多选 OR）定义匹配。

**#A3 后端 `sub_channel_id` 类型从 `UUID` 改为 `str` 的兼容性（通过）**

`list_news` 的 `sub_channel_id` 参数类型从 `Optional[uuid.UUID]` 改为 `Optional[str]`，以支持逗号分隔的多 UUID。旧前端（R2 及之前）传单个 UUID 字符串仍可正常解析——`split(",")` 后得到单元素列表，走 `== ids[0]` 分支。向后兼容，无破坏性。

## 2. 数据库变更

### 2.1 新增表：sub_channels

```sql
CREATE TABLE IF NOT EXISTS sub_channels (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  channel_space_id UUID NOT NULL REFERENCES channel_spaces(id) ON DELETE CASCADE,
  name VARCHAR(100) NOT NULL,
  sort_order INT NOT NULL DEFAULT 0,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  CONSTRAINT uq_sub_channels_space_name UNIQUE (channel_space_id, name)
);

CREATE INDEX IF NOT EXISTS ix_sub_channels_space_sort
  ON sub_channels(channel_space_id, sort_order);
```

### 2.2 ALTER TABLE：channel_sources

```sql
ALTER TABLE channel_sources
  ADD COLUMN IF NOT EXISTS sub_channel_id UUID REFERENCES sub_channels(id) ON DELETE SET NULL;
```

### 2.3 ALTER TABLE：processed_news

```sql
ALTER TABLE processed_news
  ADD COLUMN IF NOT EXISTS sub_channel_id UUID REFERENCES sub_channels(id) ON DELETE SET NULL;
```

### 2.4 新增索引

```sql
CREATE INDEX IF NOT EXISTS ix_processed_news_sub_published
  ON processed_news(channel_space_id, sub_channel_id, published_at DESC);

CREATE INDEX IF NOT EXISTS ix_raw_items_url
  ON raw_items(source_item_url)
  WHERE source_item_url IS NOT NULL;
```

### 2.5 级联行为说明

- 删除 Source → `ON DELETE CASCADE` 删除关联的 `channel_sources`、`raw_items`
- 删除 `raw_items` → `ON DELETE CASCADE` 删除关联的 `processed_news`
- 删除 `channel_sources` → `ON DELETE CASCADE` 删除关联的 `source_states`
- 删除子频道 → `ON DELETE SET NULL` 将 `channel_sources.sub_channel_id` 和 `processed_news.sub_channel_id` 置 NULL

### 2.6 迁移与回滚

迁移脚本：`db/migrations/v0.1.sql`（幂等，使用 IF NOT EXISTS / ADD COLUMN IF NOT EXISTS）

### 3.7 `app/schemas.py` — ChannelSourceUpdatePolicy 设计

**需求**：传 `null` 清除绑定，不传保持现状，传有效 UUID 更新绑定。

**Pydantic `Optional[UUID] = None` 的局限性**：JSON `null` 和字段缺失都会被解析为 `None`，无法区分"清除"和"不修改"两种语义。

**设计决策**：使用 `model_fields_set` 区分：

```python
class ChannelSourceUpdatePolicy(BaseModel):
    enabled: Optional[bool] = None
    fetch_policy: Optional[dict] = None
    sub_channel_id: Optional[uuid.UUID] = None  # 语义由路由层通过 model_fields_set 区分
```

在路由层实现时：

```python
if "sub_channel_id" in payload.model_fields_set:
```

#### POST /v1/channel-spaces/{id}/sub-channels — 创建子频道

```python
@router.post("/channel-spaces/{channel_space_id}/sub-channels", response_model=SubChannelOut)
async def create_sub_channel(
    channel_space_id: uuid.UUID,
    payload: SubChannelCreate,
    session: AsyncSession = Depends(get_session),
):
    # 验证 channel_space_id 存在
    exists = (await session.execute(
        select(ChannelSpace.id).where(ChannelSpace.id == channel_space_id)
    )).scalar_one_or_none()
    if not exists:
        raise HTTPException(status_code=404, detail="channel_space not found")

    obj = SubChannel(
        channel_space_id=channel_space_id,
        name=payload.name,
        sort_order=payload.sort_order,
    )
    session.add(obj)
    await session.commit()
    await session.refresh(obj)
    return SubChannelOut.model_validate(obj, from_attributes=True)
```

错误响应：
- name 为空字符串 → 422（Pydantic min_length 校验）
- name 为纯空格 → 422（AfterValidator `_not_blank` 校验）
- 同一空间下名称重复 → 409 `{"detail": "子频道名称 'XXX' 已存在"}`。实现方式：捕获 IntegrityError 后查询冲突名称，返回 409

## 5. X/Twitter 数据源

### 5.1 Source Config 结构

`type` = `x_twitter`，v0.1 不在 API 层校验 config 结构合法性（错误配置在 Worker 执行时报错）。

**搜索模式：**
```json
{
  "mode": "search",
  "search_query": "LLM OR GPT OR Claude"
}
```
API 请求的 `max_results` 直接取 `fetch_policy.budget.max_items_per_run`（≤100），无需在 config 中单独配置。请求量与入库量一致，不存在截断丢弃，因此 `since_id` 取最新推文 ID 是安全的。

**账号追踪模式：**
```json
{
  "mode": "user_timeline",
  "usernames": ["OpenAI", "AnthropicAI"],
  "max_results_per_user": 20
}
```
拉取所有用户后按 `published_at` 倒序排列，再截断至 `max_items_per_run`——确保时间最新的推文被优先保留。游标**仅在截断后更新**，被截断丢弃的推文对应的用户游标保持不变。

**环境变量**：`X_BEARER_TOKEN`（App-Only 认证）。

**搜索范围**：X Basic 套餐仅支持最近 7 天的 recent search。

### 5.2 游标设计

搜索模式：`{"since_id": "1760012345678900000"}`
- 每次请求带 `since_id`（上一批返回结果中最新推文 tweet_id），只拉 ID > since_id 的新推文
- 搜索模式 API 请求量 = max_items_per_run，无截断，用最新 ID 不会丢数据

账号追踪：`{"user_cursors": {"OpenAI": "176...", "AnthropicAI": "175..."}}`
- 每个用户名维护独立游标

### 5.3 实现文件：`worker/fetch_x_twitter.py`

#### 5.3.1 主入口

```python
TWEET_FIELDS = "created_at,public_metrics,author_id,text,entities,referenced_tweets"
USER_FIELDS = "name,username"
EXPANSIONS = "author_id"

async def parse_x_twitter(
    config: dict, cursor: dict, max_items: int,
) -> tuple[list[dict], dict]:
    """返回 (items, cursor_updates)"""
    mode = config.get("mode", "search")
    if mode == "search":
        return await _fetch_search(config, cursor, max_items)
    elif mode == "user_timeline":
        return await _fetch_user_timelines(config, cursor, max_items)
    else:
        raise RuntimeError(f"unsupported x_twitter mode: {mode}")
```

#### 5.3.2 搜索模式

```python
async def _fetch_search(config, cursor, max_items) -> tuple[list[dict], dict]:
    query = config.get("search_query")
    if not query:
        raise RuntimeError("x_twitter search mode requires config.search_query")

    async with httpx.AsyncClient(timeout=30, proxy=...) as client:
        params = {
            "query": query,
            "max_results": min(max_items, 100),
            "tweet.fields": TWEET_FIELDS,
            "user.fields": USER_FIELDS,
            "expansions": EXPANSIONS,
        }
        since_id = cursor.get("since_id")
        if since_id:
            params["since_id"] = since_id

        r = await client.get(url, headers=_auth_headers())
        _handle_auth_failure(r)  # 401/403 → 立即失败，不重试
        _handle_rate_limit(r)    # 429 → 抛出异常，触发退避重试
        r.raise_for_status()
        data = r.json()

    items = _parse_tweets(data)
    # 取最新 tweet_id 作为下次 since_id
    new_since_id = None
    if items:
        tweet_ids = [int(t["source_item_id"]) for t in items if t["source_item_id"].isdigit()]
        if tweet_ids:
            new_since_id = str(max(tweet_ids))

    cursor_updates = {"since_id": new_since_id} if new_since_id else {}
    return items[:max_items], cursor_updates
```

#### 5.3.3 账号追踪模式

```python
async def _fetch_user_timelines(config, cursor, max_items) -> tuple[list[dict], dict]:
    usernames = config.get("usernames")
    if not usernames:
        raise RuntimeError("x_twitter user_timeline mode requires config.usernames")

    max_per_user = config.get("max_results_per_user", 20)
    all_items: list[dict] = []
    user_cursors = dict(cursor.get("user_cursors", {}))
    user_items: dict[str, list[dict]] = {}  # 记录每个用户拉到的 tweets

    async with httpx.AsyncClient(timeout=30, proxy=...) as client:
        user_ids = await _resolve_usernames(client, usernames)

        for username, user_id in user_ids.items():
            if not user_id:
                continue
            # ... 拉取该用户 tweets ...
            tweets = _parse_tweets(data)
            user_items[username] = tweets
            all_items.extend(tweets)

    # 按 published_at DESC 排序后截断
    all_items.sort(key=lambda x: x.get("published_at") or datetime.min, reverse=True)
    retained = all_items[:max_items]
    retained_ids = {item["source_item_id"] for item in retained}

    # 仅更新被保留推文对应的用户游标
    updated_cursors = dict(user_cursors)
    for username, tweets in user_items.items():
        retained_tweets = [t for t in tweets if t["source_item_id"] in retained_ids]
        if retained_tweets:
            tweet_ids = [int(t["source_item_id"]) for t in retained_tweets if t["source_item_id"].isdigit()]
            if tweet_ids:
                updated_cursors[username] = str(max(tweet_ids))

    return retained, {"user_cursors": updated_cursors}
```

**关键设计决策**：游标更新发生在截断之后，只有被保留的推文才推进游标。被截断丢弃的推文下次继续拉取。

**空结果处理**：某用户本次返回 0 条新推文时（since_id 之后无新内容），`user_items[username]` 为空列表，`retained_tweets` 也为空，该用户的游标保持原值不变——下次继续从同一 since_id 查询。代码中 `updated_cursors = dict(user_cursors)` 保证了此行为。

#### 5.3.4 认证失败处理

Worker 层定义不可重试异常（`worker/main.py`）：

```python
class NonRetryableError(Exception):
    """不可重试的异常——Worker 捕获后直接标记 failed，不走重试队列。"""
```

`_handle_auth_failure` 抛出此异常：

```python
def _handle_auth_failure(response: httpx.Response):
    """401/403 立即创建 alert 并标记任务失败，不重试。"""
    if response.status_code in (401, 403):
        raise NonRetryableError(
            f"X API authentication failed (HTTP {response.status_code}). "
            "Check X_BEARER_TOKEN in .env"
        )
```

Worker `_run` 中的异常处理：
```python
try:
    await fn(...)
except NonRetryableError:
    # 403/429 等不可重试异常 → 创建 alert，标记 failed，不 requeue
    await _create_alert(conn, task["source_id"], "fetch_auth_failed", ...)
    await _finish_task(conn, task_id, status="failed")
except Exception:
    # 其他异常 → 退避重试
    await _requeue_task(conn, task_id, ...)
```

#### 5.3.5 辅助函数

`_resolve_usernames`：逐用户查询 user_id。404 静默跳过，非 404 错误抛出触发重试。全部用户名均不存在时，由 Worker 层创建 alert。

`_parse_tweets`：将 X API 响应解析为标准 item 格式，包含 tweet_id、text、author_username、author_name、created_at、public_metrics、entities、referenced_tweets。

## 6. LLM 处理增强

### 6.1 Prompt 变更

`worker/llm.py` 中 `translate_and_summarize` 的 prompt 替换为要求输出完整 JSON 结构（title、summary、bullets、tags、entities、importance_score、language）。

Prompt 模板：
```
把下面内容处理成中文新闻卡片，做翻译与深度摘要。
要求：
1) 如果原文是英文，翻译成中文
2) 输出中文标题(title)和中文摘要(summary)
3) summary 80-200 字
4) 提炼 3-5 个核心要点(bullets)，每个不超过 50 字
5) 生成 3-5 个分类标签(tags)
6) 识别关键实体(entities)，每个包含 name 和 type(person/org/product/tech)
7) 评估重要性评分(importance_score)，0-10 浮点数，考虑时效性、影响力、相关性
8) 只输出严格 JSON，不要输出多余文字
JSON 格式：{"title":"...","summary":"...","bullets":["..."],"tags":["..."],"entities":[{"name":"...","type":"..."}],"importance_score":7.5,"language":"zh"}
source_url: {source_url}

content:
{text}
```

### 6.2 输出校验函数

```python
def validate_llm_output(result: dict) -> dict:
    """校验并修正 LLM 输出，确保字段完整且类型正确。"""

    # title：必须为非空字符串
    title = result.get("title")
    if not isinstance(title, str) or not title.strip():
        result["title"] = "无标题"

    # summary：必须为非空字符串
    summary = result.get("summary")
    if not isinstance(summary, str) or not summary.strip():
        result["summary"] = result.get("title", "")

    # bullets：字符串数组，截断至 5 条
    bullets = result.get("bullets")
    if not isinstance(bullets, list):
        bullets = []
    result["bullets"] = [str(b) for b in bullets if b][:5]

    # tags：字符串数组，截断至 5 条
    tags = result.get("tags")
    if not isinstance(tags, list):
        tags = []
    result["tags"] = [str(t) for t in tags if t][:5]

    # entities：对象数组，过滤掉无 name 的项
    entities = result.get("entities")
    if not isinstance(entities, list):
        entities = []
    result["entities"] = [
        e for e in entities
        if isinstance(e, dict) and e.get("name")
    ]

    # importance_score：0-10 浮点数，clamp
    score = result.get("importance_score")
    try:
        score = float(score)
    except (TypeError, ValueError):
        score = 0.0
    result["importance_score"] = max(0.0, min(10.0, score))

    # language：有值时保留原值，缺失时默认 "zh"
    lang = result.get("language")
    if not isinstance(lang, str) or not lang.strip():
        result["language"] = "zh"

    return result
```

**关键设计决策**：`language` 字段保留 LLM 返回的原值，仅在缺失或非字符串时默认 `"zh"`。这与 v0.1 R1 设计（一律硬覆盖为 "zh"）不同，PRD R4 明确要求保留原值。

### 6.3 调用流程

`translate_and_summarize` 中：JSON 解析 → `_extract_first_json_object` 兜底 → `validate_llm_output` 校验修正 → 返回。

```
        ▼
```

## 9. 技术约束

### 9.1 LLM 调用期间连接管理（技术债务）

当前 `process_one` 在单次连接内完成"查询 raw_item → 去重检查 → 调用 LLM → INSERT"。LLM 调用可能耗时数秒，期间连接处于空闲等待状态。当前 `process_concurrency=1` 时不暴露此问题（连接池中有充足连接），但并发提高后可能耗尽连接池。

**v0.1 决策**：不在此迭代拆分连接。拆分需要在 `worker_loop` 的 `_run` 中分离"数据查询"和"LLM 调用 + 写入"两个阶段——改动范围较大且当前设置下无实际影响。此项作为技术债务记录，后续迭代中解决。

实现方向（供后续参考）：
```python
# worker_loop 中：
# 1. 获取连接 → 查询 raw_item + 去重 → 释放连接
# 2. 调用 LLM (无连接持有)
# 3. 获取连接 → INSERT processed_news → 释放连接
```

### 9.2 部署顺序

先执行数据库迁移（`db/migrations/v0.1.sql`），再部署新代码。迁移新增的 `sub_channel_id` 列为 nullable，旧代码不受影响；新代码 INSERT 语句引用该列，必须先于代码存在。

### 9.3 X/Twitter config 校验

v0.1 不在 API 层校验 Source config，错误配置（如 mode 缺失、search_query 缺失）在 Worker 执行时通过 RuntimeError 报错。后续迭代可在 API 层引入 per-type config schema 校验。

### 9.4 配置新增

`app/settings.py` 新增 `x_bearer_token: Optional[str] = None`。
`.env.example` 同步新增 `X_BEARER_TOKEN`（pydantic-settings 自动映射 UPPER_CASE → snake_case）。

## PM Review — R2 (2026-05-23)

### 整体评价

设计文档质量好。数据库、API、Worker、前端四层覆盖完整。PRD R4 的所有修订点均已同步到设计中（X/Twitter config 简化、since_id 策略、账号追踪排序截断、认证失败处理、language 校验修正、Pydantic null 清除、前端数据层适配）。数据流总览（第 8 节）对开发理解全局很有价值。

以下 5 个问题按严重度排列：

---

### 1. 设计不一致：第 7 节 vs 第 9.1 节

**位置**：第 7 节（process_one 变更）与第 9.1 节（LLM 调用期间不持有数据库连接）

**问题**：9.1 节明确要求"process_one 应在调用 translate_and_summarize 前释放数据库连接"，但第 7 节的实现代码仍是单次连接获取 → 全程持有 → 最后释放。两处描述矛盾。

**建议**：二选一——
- A：第 7 节给出具体的连接分离方案代码。示例：在 `worker_loop` 的 `_run` 中拆分两步——先获取连接查询 raw_item + 去重（释放），调用 LLM，再获取连接写入 processed_news。
- B：明确标注 current `process_concurrency=1` 下不暴露此问题，重构推迟至后续迭代，9.1 节改为"技术债务记录"而非"当前设计要求"。

**判定**：需要架构师选择一个方案并统一文档。

---

### 2. 子频道 name 纯空格校验盲区

**位置**：第 4.1 节 POST /sub-channels

**问题**：设计说"name 为空或纯空格 → 422（Pydantic 校验）"，但 `SubChannelCreate.name: str = Field(min_length=1, max_length=100)` 只能拒绝空字符串 `""`，无法拒绝 `"   "`（三个空格——满足 min_length=1）。纯空格名称能通过 Pydantic 校验到达数据库。

**PRD 要求**（4.1.2）："name 不可为空字符串或纯空格，否则返回 422"。

**建议**：在 `SubChannelCreate` 中新增一个 validator，使用 `name.strip()` 检查。或改用 `Field(min_length=1, pattern=r"\S")` 等方案。

**判定**：需要修复，否则 PRD 验收标准无法通过。

---

### 3. 认证失败异常无法区分"立即失败"和"可重试"

**位置**：第 5.3.4 节 `_handle_auth_failure` + Worker 异常处理

**问题**：`_handle_auth_failure` 抛出 `RuntimeError`，Worker 的 `_run` 函数 catch 所有 Exception 后统一走 `_requeue_task`。RuntimeError 没有特殊标记，Worker 无法区分"这是认证失败（不应重试）"和"这是网络超时（应该重试）"。

当前设计文档说"标记任务为 failed（不进入重试队列）"，但未给出 Worker 层如何识别这个异常的具体机制。

**建议**：在 Worker 层定义一个 `NonRetryableError(Exception)` 自定义异常类。`_handle_auth_failure` 抛出它。`_run` 中 catch 到 `NonRetryableError` 时直接 `_finish_task(status="failed")`，不走 `_requeue_task`。

**判定**：需要补充实现方案，否则 401/403 仍会浪费重试次数。

### 4. 账号追踪：单用户空结果时的游标行为未说明

**位置**：第 5.3.3 节 `_fetch_user_timelines`

**问题**：设计正确处理了"截断后仅更新保留推文的游标"。但场景"用户 A 返回 0 条（since_id 已存在），用户 B 返回 10 条"没有说明。此时 A 的游标应保持不变（本次无新推文），代码中 `updated_cursors = dict(user_cursors)` 会保留 A 的原始 since_id——行为正确但未文档化。

**建议**：在 5.3.3 的代码注释或说明中补充一句："某用户本次返回 0 条新推文时，该用户的游标保持原值不变（下次继续从同一 since_id 查询）"。

**判定**：小问题，不影响实现。开发时补充注释即可。

### 5. 前端 listNews 新增参数破坏现有调用方

**位置**：第 10.2 节 `api.ts`

更关键的是：新增参数后调用方在 v0.1 不需要传，但如果未来参数继续增加（v0.2 可能加排序、筛选），位置参数列表会越来越长。

**建议**：将 `listNews` 改为对象参数模式：
```typescript
export async function listNews(
  spaceId: UUID,
  opts?: { limit?: number; offset?: number; subChannelId?: UUID },
): Promise<ProcessedNews[]>
```
v0.1 调用方 `listNews(spaceId, { limit: 20, offset: 0 })` 无需改动即可兼容。

**判定**：建议采纳，但不是阻塞项。如果开发时间紧张，当前方案（第 4 个位置参数，默认 undefined）也可以工作。

### 2. PUT /sub-channels/{id} 更新逻辑中冗余的条件判断

**位置**：第 4.1 节 `update_sub_channel` 路由

**问题**：
```python
if not values:
    raise HTTPException(status_code=400, detail="at least one field required")
if values:  # 前面已 raise，此行永远为 True
    await session.execute(...)
```
`if not values` 之后已经 raise 了 400，后面的 `if values:` 永远是 True。这个冗余判断会让阅读代码的人困惑——怀疑是否有某个分支能绕过前面的 raise。

**建议**：移除 `if values:` 包装，直接执行 UPDATE。或改为 if/else 结构使意图更清晰：
```python
if not values:
    raise HTTPException(...)
await session.execute(
    update(SubChannel).where(SubChannel.id == sub_channel_id).values(**values)
)
await session.commit()
```

**判定**：轻微代码质量问题，不影响功能。开发实现时顺手修正即可。

```python
class NonRetryableError(Exception):
```

### 6. 去重 SELECT-then-INSERT 存在 TOCTOU 竞态

**位置**：第 7.4 节子频道内去重

**问题**：去重逻辑是先 SELECT 检查是否存在，再 INSERT。当两个并发 process task 处理不同 raw_item 但 URL 相同时，可能同时通过 SELECT 检查（都返回不存在），然后双双 INSERT 成功。`ON CONFLICT (raw_item_id) DO NOTHING` 无法阻止此竞态（因为两个 raw_item_id 不同）。

发生条件比较苛刻：
- 两个不同 Source 绑定了同一个子频道
- 它们几乎同时抓到了相同 URL 的内容
- 两个 process task 恰好并发执行

PRD 4.4.1 的"已知限制"已经接受了不同 Source 相同 URL 场景下的非确定性（先到先得），TOCTOU 竞态是该非确定性的一个子集——两条新闻都入库而非只保留一条。

**建议**：在当前设计的"已知限制"段（设计文档 7.4 末尾）追加一句话说明 TOCTOU 竞态的可能性。如果后续迭代需要严格去重，可通过添加唯一约束 `UNIQUE(channel_space_id, sub_channel_id, source_item_url)` 或使用 `INSERT ... WHERE NOT EXISTS` 解决。

**判定**：低概率、低影响，不阻塞 v0.1。建议在"已知限制"段补充说明，避免后续开发者踩坑。

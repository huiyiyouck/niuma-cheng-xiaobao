## 2. NewsPage（新闻浏览页）

## 6. LogViewer（日志查看器）

| 区域 | 视觉要求 |
|------|---------|
| 筛选区 | **Chip 按钮组**（非下拉框）：级别用彩色 Chip（DEBUG 灰、INFO 蓝、WARNING 橙、ERROR 红），来源用 Chip（API / Worker），时间范围下拉（开始时间 / 结束时间），关键字搜索输入框，右对齐自动刷新开关 + 刷新按钮 |

| 自动刷新 | 默认 10s 间隔，可开关，刷新时追加新条目不重复 |

## Review by 全栈开发 — R1

### 整体评估

UI Spec 覆盖了 PRD 全部 9 个前端需求点，mockup 文件质量高、视觉一致性好。但存在 3 项可实现性冲突和若干细节遗漏——如果带到实现阶段会导致返工或需要反复确认。

---

### 需修改（影响可实现性）

**#1 SubChannelManager 的空间选择器与全局 SpaceSelector 架构冲突（中等）**

§5 给 SubChannelManager 单独加了一个"频道空间下拉选择器"。但当前 AdminPage 架构是：顶部一个全局 SpaceSelector → 选中空间后所有子组件通过 `spaceId` prop 接收。如果 SubChannelManager 内部也有独立空间下拉，两个选择器共存、互相覆盖——用户在 SubChannelManager 里选了"财经"，全局 SpaceSelector 还在"AI 资讯"，其他 Tab（频道管理/Source管理）展示哪个空间的数据？

建议：SubChannelManager 复用全局 SpaceSelector 的选中值，不内嵌独立空间选择器。或明确说明两个选择器的联动关系（谁覆盖谁）。

---

**#2 LogViewer mockup 与 UI Spec 文本矛盾 + 筛选 Chip 多选语义未定义（中等）**

两处不一致：
- §6 表格写了"时间范围下拉"，但 mockup `admin-subchannel-logs.html` 里根本没有时间范围控件——只有级别 Chip + 来源 Chip + 搜索框 + 自动刷新。
- Mockup 中 4 个级别 Chip 全部可以同时高亮选中，来源也是 API+Worker 同时选中。但多选时查询逻辑是取并集（`level IN ('INFO','ERROR')`）还是交集？"同时显示 INFO 和 ERROR"语义合理，但需要有明确定义。

建议：统一 UI Spec 和 mockup（去掉或补上时间范围控件），明确筛选 Chip 单选还是多选、多选时后端查询用 AND 还是 OR。

---

**#3 SourceManager 操作列未区分 Source 状态（中等）**

§3 操作列固定为"编辑 / 验证 / 绑定 / 删除"。但 PRD 3.1.4 的状态机下不同状态可用操作不同：

| Source 状态 | 合理操作 |
|------------|---------|
| unverified | 验证、编辑、删除 |
| verified | 绑定、编辑、删除 |
| active | 编辑、删除 |
| error | 验证（重试）、编辑、删除 |

Mockup 也体现了这一点——error 状态行操作是"重试 | 编辑 | 删除"。但 UI Spec 正文没有区分，按字面实现所有 Source 都会展示"验证/绑定"按钮。

建议：补充操作列按 Source 状态动态展示的规则，或引用 PRD 3.1.4 状态机。

---

### 建议改进（细节补全）

**#4 标签颜色分配策略缺失（轻微）**

§2 标签 Chip "彩色半透明背景"，§7 TagChip "彩色半透明背景"。但一个新闻有 3-5 个标签，颜色如何分配？按内容哈希？按出现顺序轮转？按语义分类？

建议：给出 4-5 色调色板（蓝 `#e8f4fd/#3498db`、橙 `#fef3e2/#f39c12`、绿 `#e8f8e8/#27ae60`、紫 `#f3e8ff/#8e44ad`），明确分配方式（如"按标签首字母哈希循环轮转"）。

---

**#5 颜色体系未集中管理（轻微）**

颜色值散落在 §1-§7 各段，以 `#3498db` 为例出现于：统计数字、子频道 Pill 选中、标签 Chip 蓝、Source 类型 Badge 蓝、级别 Badge INFO、链接。如果调整主题色需改 6 处以上。

建议：在文档末尾增加色彩体系汇总表，列出主色（#1a1a2e）、强调蓝（#3498db）、成功绿（#2ecc71）、警告金（#f39c12）、错误红（#e74c3c）、紫色系（#9b59b6/#8e44ad）及其使用场景。

---

**#6 深色顶栏与 Vue Router 路由模型的关系未说明（轻微）**

§1 定义深色顶栏（品牌名 + "浏览/管理" Tab），但当前前端是 Vue Router 双路由（`/` → NewsPage，`/admin` → AdminPage），没有全局顶栏组件。需说明：
- 顶栏是新增全局 Layout 组件还是页面内实现？
- "浏览/管理" Tab 高亮如何与路由对应？
- 管理页内部 Tab 和顶栏"管理" Tab 是父子还是同级？

---

**#7 空状态 / 加载态 / 错误态覆盖不全（轻微）**

| 场景 | 当前状态 |
|------|---------|
| 统计卡片数据为 0 或加载失败 | 未定义（mockup 硬编码 1,247/32/5/3） |
| SourceManager 列表为空 | 未定义 |
| VerifyDialog 验证失败（status=error） | mockup 只展示成功态 |
| SubChannelManager 列表为空 | 未定义 |
| LogViewer 日志文件不存在 | 未定义 |
| 详情面板摘要为空 | 未定义 |

建议：各区域补充空状态/错误态的简要规范（1-2 句话即可）。

---

**#8 SourceManager 添加弹窗"自动识别 + 手动覆盖"交互流程未定义（轻微）**

§3 添加弹窗"类型（显示自动识别结果 + 可手动覆盖下拉）"。交互有两种理解：
- A：用户先输入 URL → 失焦后自动识别 → 展示结果 → 确认或手动改
- B：输入 URL 同时下方展示"识别中…"→ 完成后回填下拉框

建议：用 2-3 句话明确交互流程。

---

**#9 EntityBadge 未预知类型的降级策略缺失（轻微）**

§7 只定义 4 种实体颜色（人物/组织/产品/技术）。但 `ProcessedNews.entities` 的 type 来自 LLM 输出，可能有 `location`、`event`、`topic` 等未预知值。

建议：补充"未匹配类型降级为灰色"或类似的兜底策略。

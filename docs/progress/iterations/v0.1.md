#### 发现问题

**#1 架构一致性：`_handle_auth_failure` 在 user_timeline 和 resolve_usernames 中缺失**

- 位置：`worker/fetch_x_twitter.py`
- `_fetch_search` (line 60) 调用了 `_handle_auth_failure(r)`，但 `_fetch_user_timelines` (line 112) 和 `_resolve_usernames` (line 136) 未调用。三个函数都向 X API 发起认证请求（使用相同的 `_auth_headers()`），401/403 场景下行为不一致。
- 建议：在 `_fetch_user_timelines` 的 `client.get` 后、`_resolve_usernames` 的 `client.get` 后均添加 `_handle_auth_failure(r)` 调用。
- 严重度：中等（当前 user_timeline 模式如果遇到 401/403 会走 `r.raise_for_status()` → HTTPError → 被外层 except Exception 捕获 → 进入退避重试，浪费重试次数且不会创建 alert）

**#2 防御性编码：NonRetryableError alert 条件可考虑泛化**

- 位置：`worker/main.py:488-492`
- 当前 `except NonRetryableError` 仅在 `task_dict["type"] == "fetch"` 时创建 alert。实际场景下没问题（NonRetryableError 只从 fetch 路径抛出），但条件判断隐含了"只有 fetch 任务才会抛 NonRetryableError"的假设。
- 建议：后续迭代考虑移除 type 判断，或改为通用 alert 创建逻辑。
- 严重度：轻微（当前无实际影响，属未来扩展性考虑）

#### Review 结论

架构整体与设计文档一致。NonRetryableError 异常体系、LLM language 校验修正、前端数据层适配、PUT 路由简化均正确实现。#1（auth failure 缺失）需要在 R3 中补充，#2 可选。修完后实现阶段可定稿。

#### 发现问题

**#1 账号追踪游标更新时序与设计不一致**

- 位置：`worker/fetch_x_twitter.py` `_fetch_user_timelines`（L120-126）
- 设计文档 5.3.3 节明确要求：拉取所有用户推文 → 按 `published_at` DESC 排序 → 截断至 `max_items` → **仅对保留推文更新游标**。当前代码在逐用户拉取阶段即更新游标（L120-123），而后才执行 `all_items[:max_items]` 截断（L126）。
- 后果：被截断丢弃的旧推文对应的用户游标已被推进，下次抓取时这些推文将被永久跳过。
- 设计原文：
  > 游标更新发生在截断之后，只有被保留的推文才推进游标。被截断丢弃的推文下次继续拉取。
- 建议：重构 `_fetch_user_timelines`——先收集所有用户推文到 `user_items` 字典（不更新游标），排序截断得到 `retained_ids`，再仅对保留推文更新游标。与设计 5.3.3 节伪代码对齐。
- 严重度：中等（数据完整性风险，被截断推文永久丢失）

**#2 账号追踪模式缺少 `_handle_auth_failure` 调用**

- 位置：`worker/fetch_x_twitter.py` `_fetch_user_timelines`（L112）、`_resolve_usernames`（L136）
- 搜索结果已确认：`_fetch_search` 在 L60 正确调用了 `_handle_auth_failure(r)`，但 `_fetch_user_timelines` 的 per-user 请求和 `_resolve_usernames` 均未调用。
- 后果：401/403 不会被转换为 NonRetryableError，走 `r.raise_for_status()` → 普通 HTTPError → 退避重试队列。认证问题重试不可能恢复，浪费重试次数且不创建 alert。`_resolve_usernames` 中更为严重——401 时所有用户名被设为 `None`，静默返回空结果。
- 建议：在两处 API 调用后均添加 `_handle_auth_failure(r)`。
- 严重度：中等（认证失败在账号追踪模式下不可见）
- 注：架构师已发现此问题（架构师 R2 #1），PM 确认为需要修复。

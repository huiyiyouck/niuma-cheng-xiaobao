#### 发现问题

**#1 架构一致性：`_handle_auth_failure` 在 user_timeline 和 resolve_usernames 中缺失**

- 位置：`worker/fetch_x_twitter.py`
- `_fetch_search` (line 60) 调用了 `_handle_auth_failure(r)`，但 `_fetch_user_timelines` (line 112) 和 `_resolve_usernames` (line 136) 未调用。三个函数都向 X API 发起认证请求（使用相同的 `_auth_headers()`），401/403 场景下行为不一致。
- 建议：在 `_fetch_user_timelines` 的 `client.get` 后、`_resolve_usernames` 的 `client.get` 后均添加 `_handle_auth_failure(r)` 调用。
- 严重度：中等（当前 user_timeline 模式如果遇到 401/403 会走 `r.raise_for_status()` → HTTPError → 被外层 except Exception 捕获 → 进入退避重试，浪费重试次数且不会创建 alert）

**#2 防御性编码：NonRetryableError alert 条件可考虑泛化**

- 位置：`worker/main.py:488-492`
- 当前 `except NonRetryableError` 仅在 `task_dict["type"] == "fetch"` 时创建 alert。实际场景下没问题（NonRetryableError 只从 fetch 路径抛出），但条件判断隐含了"只有 fetch 任务才会抛 NonRetryableError"的假设。
- 建议：后续迭代考虑移除 type 判断，或改为通用 alert 创建逻辑。
- 严重度：轻微（当前无实际影响，属未来扩展性考虑）

#### Review 结论

架构整体与设计文档一致。NonRetryableError 异常体系、LLM language 校验修正、前端数据层适配、PUT 路由简化均正确实现。#1（auth failure 缺失）需要在 R3 中补充，#2 可选。修完后实现阶段可定稿。

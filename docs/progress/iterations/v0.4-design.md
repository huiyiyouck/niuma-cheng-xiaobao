### 3.3 端点详细设计

#### 3.3.1 PUT `/v1/channel-spaces/:id`

- **Schema**：`ChannelSpaceUpdate`
- **逻辑**：
  1. 查询 `channel_spaces WHERE id = $1`，不存在返回 404
  2. 若 `name` 变更，校验唯一性（`SELECT id FROM channel_spaces WHERE name = $1 AND id != $2`），冲突返回 409 `{ detail: "频道空间名称已存在" }`
  3. `UPDATE channel_spaces SET ... WHERE id = $1 RETURNING *`
  4. 返回更新后的 `ChannelSpace` 对象
- **鉴权**：非 GET → adminGuard 自动拦截
- **错误码**：404（不存在）、409（名称冲突）

#### 3.3.2 GET `/v1/channel-spaces/:id/delete-preview`

- **逻辑**：对 7 张关联表分别计数。其中 `source_states` 通过 `JOIN channel_sources` 间接关联（`source_states` 表不直接持有 `channel_space_id`），完整 SQL：

  ```sql
  -- 6 张直接关联表（有 channel_space_id 列）
  SELECT COUNT(*) FROM sub_channels    WHERE channel_space_id = $1;
  SELECT COUNT(*) FROM channel_sources WHERE channel_space_id = $1;
  SELECT COUNT(*) FROM raw_items       WHERE channel_space_id = $1;
  SELECT COUNT(*) FROM processed_news  WHERE channel_space_id = $1;
  SELECT COUNT(*) FROM tasks           WHERE channel_space_id = $1;
  SELECT COUNT(*) FROM alerts          WHERE channel_space_id = $1;

  -- source_states：通过 channel_sources 间接关联
  SELECT COUNT(*) FROM source_states ss
  JOIN channel_sources cs ON cs.id = ss.channel_source_id
  WHERE cs.channel_space_id = $1;
  ```
- **响应**：
  ```json
  {
    "channel_sources": 5,
    "sub_channels": 3,
    "raw_items": 120,
    "processed_news": 80,
    "source_states": 5,
    "tasks": 2,
    "alerts": 1
  }
  ```
- **鉴权**：GET 端点，adminGuard 当前仅拦截 alerts 路径和写操作，本端点无需鉴权（与现有 GET `/v1/channel-spaces` 一致）
- **性能**：7 次 COUNT 查询，每张表有 `channel_space_id` 索引，单次 < 20ms

#### 3.3.3 DELETE `/v1/channel-spaces/:id`

- **逻辑**：
  1. 确认空间存在，不存在返回 404
  2. `DELETE FROM channel_spaces WHERE id = $1`
  3. 数据库外键 `ON DELETE CASCADE` 自动级联删除 7 张表
  4. 返回 204 No Content
- **鉴权**：DELETE → adminGuard 自动拦截
- **安全性**：前端应先调用 `delete-preview` 展示影响面，用户二次确认后再调此端点

#### 3.3.4 DELETE `/v1/channel-sources/:id`

- **逻辑**：
  1. 查询 `channel_sources WHERE id = $1`，不存在返回 404
  2. `DELETE FROM channel_sources WHERE id = $1`
  3. 数据库外键 `ON DELETE CASCADE` 自动清理 `source_states` 和 `tasks`
  4. 返回 204 No Content
- **鉴权**：DELETE → adminGuard 自动拦截
- **注意**：此操作仅删除绑定关系，不删除 `sources` 表记录，也不删除已入库的 `raw_items`/`processed_news`

#### 3.3.5 PUT `/v1/channel-spaces/:space_id/sub-channels/reorder`

- **Schema**：`SubChannelsReorder`
- **逻辑**：
  1. 确认 `channel_spaces` 存在，不存在返回 404
  2. 开启事务
  3. 对 `items` 中每个 `{ id, sort_order }`：先校验 `id` 属于当前 `space_id`（`SELECT id FROM sub_channels WHERE id = $1 AND channel_space_id = $2`），任一无效则回滚返回 400 `{ detail: "子频道不属于当前空间" }`
  4. `UPDATE sub_channels SET sort_order = $1 WHERE id = $2`（批量循环，或使用 `CASE WHEN` 单条 UPDATE）
  5. 提交事务，返回 `{ ok: true }`
- **鉴权**：PUT → adminGuard 自动拦截
- **幂等**：重复调用不产生副作用

#### 3.3.6 PATCH `/v1/alerts/:id`

- **Schema**：`AlertStatusUpdate`
- **逻辑**：
  1. 查询 `alerts WHERE id = $1`，不存在返回 404
  2. 校验状态转换合法性（§4.2 状态机）：
     - `acknowledged`：当前必须是 `active` → 更新
     - `resolved`：当前必须是 `active` 或 `acknowledged` → 更新
     - 其他（如当前已是 `resolved`）：返回 422 `{ detail: "已解决的告警不可再次修改状态" }`
  3. `UPDATE alerts SET status = $1 WHERE id = $2 RETURNING *`
  4. 返回更新后的 `Alert` 对象（含新增 `status` 字段）
- **鉴权**：PATCH + `/v1/alerts` → adminGuard 自动拦截
- **错误码**：404（不存在）、422（状态转换非法）

#### 3.3.7 POST `/v1/alerts/acknowledge-all`

- **Schema**：`AlertsAcknowledgeAll`
- **逻辑**：
  1. 构建 WHERE：`status = 'active'` + 可选 `channel_space_id`
  2. `UPDATE alerts SET status = 'acknowledged' WHERE status = 'active' [AND channel_space_id = $2]`
  3. 返回 `{ updated: N }`（实际更新的行数）
- **鉴权**：POST + `/v1/alerts` → adminGuard 自动拦截
- **幂等**：重复调用只影响仍为 `active` 的告警，已确认的告警不重复计数

### 3.4 现有端点增强

#### 3.4.1 GET `/v1/channel-spaces/:id/news` — 新增搜索参数

- **Schema 扩展**：`NewsQuery` 新增 `q: z.string().optional()`
- **逻辑**：当 `q` 参数存在且非空时，在 WHERE 子句中追加：
  ```sql
  AND (pn.title ILIKE '%' || $q || '%' OR pn.summary ILIKE '%' || $q || '%')
  ```
- **组合行为**：搜索条件与 `sub_channel_id` 筛选取 AND 交集
- **性能**：PRD 已注明数据量 < 5 万条时可接受全表扫描。无需额外索引（v0.4 范围外）

#### 3.4.2 DELETE `/v1/sources/:id` — 新增绑定检查

- **响应增强**：删除前查询 `channel_sources` 绑定列表，在错误响应或成功响应中附加 `affected_bindings`：
  ```json
  {
    "affected_bindings": [
      { "channel_space_id": "uuid", "channel_space_name": "财经快讯", "sub_channel_name": "美股" }
    ]
  }
  ```
- **并发安全**：在事务中 `SELECT ... FOR UPDATE` 锁定 `channel_sources` 行，然后 `DELETE`。若在用户确认期间绑定已被其他操作清理，`FOR UPDATE` 获取不到行，正常返回成功
- **鉴权**：DELETE → adminGuard 自动拦截

## 4. 核心流程

### 4.1 RSS Fetcher

**文件**：`server/src/worker/fetchers/rss.ts`

**依赖**：[`rss-parser`](https://www.npmjs.com/package/rss-parser)（npm 周下载 ~200K，成熟稳定），TypeScript 类型内置。

**接口契约**（与 `registry.ts` 的 `Fetcher` 接口对齐）：

```typescript
// rss.ts 核心结构
import Parser from "rss-parser";
import { register } from "./registry.ts";
import { NonRetryableError } from "../errors.ts";

const parser = new Parser({
  timeout: 15000,
  headers: { "User-Agent": "NiumaNewsBot/0.4" },
});

async function fetchRSS(
  config: Record<string, unknown>,
  cursor: Record<string, unknown>,
  maxItems: number
): Promise<[any[], Record<string, unknown>]> {
  const feedUrl = config.source_url as string;
  if (!feedUrl) throw new NonRetryableError("RSS fetcher requires source_url in config");

  let feed: Parser.Output<{ [key: string]: any }>;
  try {
    feed = await parser.parseURL(feedUrl);
  } catch (err: any) {
    // HTTP 4xx → NonRetryable（feed 不存在）
    // 解析失败 → NonRetryable（不是有效 RSS/Atom）
    throw new NonRetryableError(`RSS fetch/parse failed: ${err.message}`);
  }

  const lastPubDate = cursor.lastPubDate as string | undefined;
  const items: any[] = [];

  for (const entry of feed.items || []) {
    const pubDate = entry.pubDate || entry.isoDate || undefined;
    // 游标过滤：仅处理比 lastPubDate 更新的条目
    if (lastPubDate && pubDate) {
      if (new Date(pubDate) <= new Date(lastPubDate)) continue;
    }
    const guid = entry.guid || entry.link || undefined;
    if (!guid) continue; // 跳过无标识的条目

    items.push({
      source_item_id: guid,
      url: entry.link || undefined,
      published_at: pubDate ? new Date(pubDate).toISOString() : undefined,
      content: {
        title: entry.title || "",
        summary: entry.contentSnippet || entry.content || "",
        author: entry.creator || undefined,
        categories: entry.categories || [],
        source_name: feed.title || "",
      },
    });
  }

  // 按 published_at 升序排列（先发布的在前，游标取最新的）
  items.sort((a, b) => {
    const da = a.published_at ? new Date(a.published_at).getTime() : 0;
    const db = b.published_at ? new Date(b.published_at).getTime() : 0;
    return da - db;
  });

  const retained = items.slice(0, maxItems);
  const newCursor: Record<string, unknown> = {};
  if (retained.length > 0) {
    const latest = retained[retained.length - 1];
    if (latest.published_at) {
      newCursor.lastPubDate = latest.published_at;
    }
  }

  return [retained, newCursor];
}

function renderRSSForLLM(content: Record<string, unknown>): string {
  const title = content.title as string || "";
  const summary = content.summary as string || "";
  const source = content.source_name as string || "";
  const author = content.author as string || "";

  const parts: string[] = [];
  if (source) parts.push(`[来源: ${source}]`);
  if (author) parts.push(`作者: ${author}`);
  parts.push(title);
  if (summary) parts.push(summary);
  return parts.join("\n\n").trim();
}

// 注册到 registry
register({
  type: "rss",
  fetch: async (cfg, cursor, maxItems) => {
    const [items, cursorUpdates] = await fetchRSS(cfg, cursor, maxItems);
    return { items, cursorUpdates };
  },
  renderForLLM: renderRSSForLLM,
});
```

**注册**：`server/src/worker/fetchers/index.ts` 新增一行：
```typescript
import "./rss.ts";
```

**validate 策略**：PRD 指定"调用 fetch() 空跑判断"。实现方式：在 `source-detector.ts` 或 `VerifyDialog` 调用的验证接口中，对 RSS 类型调用 `find("rss").fetch(config, {}, 1)` — 成功即验证通过，`NonRetryableError` 即验证失败。暂不扩展 `Fetcher` 接口。

**游标机制**：`source_states.cursor`（jsonb）存储 `{ lastPubDate: "ISO8601" }`。与 X/Twitter Fetcher 的 `{ since_id: "..." }` 结构隔离，各自维护独立的 key。Dispatcher 通过 `find(type)` 路由到正确的 Fetcher，Cursor 合并逻辑不变（`{ ...row.cursor, ...cursorUpdates }`）。

**非 Retryable 场景**：HTTP 404 → `NonRetryableError`；XML 解析失败 → `NonRetryableError`。

### 4.2 告警状态机

```
                Worker 创建
                    │
                    ▼
               ┌─────────┐
               │ active   │  (红色 #e74c3c)
               └────┬─────┘
                    │ PATCH status="acknowledged"
                    │ 或 POST /acknowledge-all
                    ▼
               ┌──────────────┐
               │ acknowledged │  (橙色 #f39c12)
               └──────┬───────┘
                    │ PATCH status="resolved"
                    ▼
               ┌──────────┐
               │ resolved  │  (灰色 #94A3B8)
               └──────────┘   ← 终态，不可回退

回退路径：
  acknowledged ──(告警复现)→ active （Worker 端逻辑，见 §4.2.1）
```

**状态转换规则（后端校验）**：

| 当前状态 | 允许目标 | 不允许 |
|---------|---------|--------|
| `active` | `acknowledged`、`resolved` | — |
| `acknowledged` | `resolved` | `active`（API 层禁止；Worker 告警复现时可回退，见下方） |
| `resolved` | —（终态） | `active`、`acknowledged` |

**API 层校验伪代码**：
```typescript
function validateAlertTransition(current: string, target: string): void {
  if (current === "resolved") {
    throw new HttpError(422, "已解决的告警不可再次修改状态");
  }
  if (current === "acknowledged" && target === "active") {
    throw new HttpError(422, "不允许通过 API 将已确认告警回退为 active");
  }
  // active → acknowledged ✓
  // active → resolved ✓
  // acknowledged → resolved ✓
}
```

#### 4.2.1 Worker 端告警复现回退

当 Worker 检测到已 `acknowledged` 的告警再次发生时（如连续失败数重新增长），允许将 status 从 `acknowledged` 回退为 `active`。此操作由 Worker 的 `monitor.ts` 执行，不经过 API：

```typescript
// scheduler.ts 或 monitor.ts 中
await pool.query(
  "UPDATE alerts SET status = 'active' WHERE type = $1 AND channel_space_id = $2 AND status = 'acknowledged'",
  [alertType, channelSpaceId]
);
```

### 4.3 频道空间删除流程

```
前端                                    后端
  │                                       │
  │  GET /v1/channel-spaces/:id/          │
  │  delete-preview ──────────────────►   │
  │                                       │ 7 次 COUNT 查询
  │  ◄── { channel_sources: 5, ... }     │
  │                                       │
  │  展示 Modal：影响统计 +                │
  │  "输入空间名称确认删除"                 │
  │                                       │
  │  DELETE /v1/channel-spaces/:id ────►  │
  │                                       │ DELETE ... CASCADE
  │  ◄── 204 No Content                   │  (sub_channels, channel_sources,
  │                                       │   raw_items, processed_news,
  │  Toast "已删除"                        │   source_states, tasks, alerts)
```

**事务安全**：依赖 PostgreSQL 外键 `ON DELETE CASCADE`，单条 DELETE 语句原子完成。无需应用层事务包装。

### 4.4 Source 解绑流程

```
  DELETE /v1/channel-sources/:id ────►  DELETE channel_sources WHERE id = $1
                                        → CASCADE DELETE source_states
                                        → CASCADE DELETE tasks (channel_source_id)
                                        → raw_items/processed_news 保留
                                        → sources 表保留
                                        ◄── 204
```

与删除 Source 的区别：删除 Source（`DELETE /v1/sources/:id`）会级联删除 `channel_sources` 和 `raw_items`；解绑仅删除绑定关系。

### 4.5 搜索流程

```
  GET /v1/channel-spaces/:id/news?q=关键词&sub_channel_id=id1,id2&sort=published_desc
  ──► SELECT ... FROM processed_news pn
      JOIN raw_items ri ON ri.id = pn.raw_item_id
      WHERE pn.channel_space_id = $1
        AND (pn.title ILIKE '%' || $2 || '%' OR pn.summary ILIKE '%' || $2 || '%')
        AND (
          -- 多子频道：逗号分隔 → 转为数组后 ANY 匹配；无参数时不限制
          $3::uuid[] IS NULL
          OR (ri.sub_channel_id IS NOT NULL AND ri.sub_channel_id = ANY($3::uuid[]))
        )
      ORDER BY
        CASE $4
          WHEN 'score_desc' THEN pn.importance_score END DESC,
        CASE $4
          WHEN 'score_asc'  THEN pn.importance_score END ASC,
        pn.published_at DESC
      LIMIT $5 OFFSET $6
```

**多子频道处理**（与现有 `news.ts` 路由逻辑一致）：
- 前端传参：`sub_channel_id=uuid1,uuid2`（逗号分隔，UI 规范 §2.3 已定义多选）
- 后端解析：`const subIds = req.query.sub_channel_id ? req.query.sub_channel_id.split(",").filter(Boolean) : null`
- SQL 参数：`$3::uuid[]`（PostgreSQL 数组类型，`ANY()` 匹配）
- 无参时传 `NULL`，条件分支跳过子频道筛选

**搜索条件**：`q` 参数使用参数化查询（`$2`），避免 SQL 注入。`ILIKE` 前后通配符实现模糊匹配。

搜索与子频道筛选、排序自由组合（AND 交集）。

### 4.6 日志文件路径修复

**问题**：`admin-logs.ts` 的 `LOG_FILES` 硬编码了无后缀文件名，winston-daily-rotate-file 轮转后旧日志改名，导致历史日志无法读取。

**修复**：

```typescript
// admin-logs.ts 中 LOG_FILES 改为动态解析
import fs from "node:fs";
import path from "node:path";

const LOG_DIR = path.resolve("logs");

function getLogFiles(): { name: string; path: string }[] {
  const files: { name: string; path: string }[] = [];
  const entries = fs.readdirSync(LOG_DIR).filter(f => f.endsWith(".log"));

  for (const f of entries) {
    const match = f.match(/^(api|worker)(?:-(\d{4}-\d{2}-\d{2}))?\.log$/);
    if (match) {
      const [, source, date] = match;
      const label = date ? `${source} (${date})` : `${source} (今天)`;
      files.push({ name: label, path: path.join(LOG_DIR, f) });
    }
  }
  return files.sort((a, b) => b.path.localeCompare(a.path)); // 最新在前
}
```

**读取策略**：
- 当天：`api.log` / `worker.log`（winston 当前写入目标，一定存在）
- 历史：`api-YYYY-MM-DD.log` / `worker-YYYY-MM-DD.log`
- 文件列表按日期降序排列

3. 现有 Vue 3 生态（如 `vue-toastification`）普遍采用此模式
4. Modal（`modal.confirm()` 返回 Promise）同样适合模块级单例

**实现架构**：

**`useToast.ts` 设计**：

```typescript
// 模块级响应式状态（单例）
import { reactive } from "vue";

interface ToastItem {
  id: number;
  type: "success" | "error" | "info";
  message: string;
}

const state = reactive<{ toasts: ToastItem[] }>({ toasts: [] });
let nextId = 0;

export function useToast() {
  function show(type: ToastItem["type"], message: string) {
    const id = nextId++;
    state.toasts.unshift({ id, type, message }); // 新 toast 在上方
    setTimeout(() => remove(id), 2500);
  }
  function success(msg: string) { show("success", msg); }
  function error(msg: string) { show("error", msg); }
  function info(msg: string) { show("info", msg); }
  function remove(id: number) {
    const idx = state.toasts.findIndex(t => t.id === id);
    if (idx >= 0) state.toasts.splice(idx, 1);
  }
  return { toasts: state.toasts, success, error, info, remove };
}
```

**`ToastContainer.vue`**：渲染 `state.toasts` 列表，应用 §5.1 UI 规范样式（动画、颜色、堆叠）。

**`useModal.ts` 设计**：

```typescript
import { reactive } from "vue";

interface ModalState {
  visible: boolean;
  title: string;
  body: string;
  confirmText?: string;
  danger?: boolean;
  loading?: boolean;
  resolve?: (value: boolean) => void;
}

const state = reactive<ModalState>({
  visible: false, title: "", body: "",
});

export function useModal() {
  function confirm(title: string, body: string, opts?: { confirmText?: string; danger?: boolean }): Promise<boolean> {
    return new Promise((resolve) => {
      Object.assign(state, {
        visible: true, title, body,
        confirmText: opts?.confirmText || "确认",
        danger: opts?.danger || false,
        loading: false,
        resolve,
      });
    });
  }
  function close(accepted: boolean) {
    state.visible = false;
    state.resolve?.(accepted);
  }
  return { state, confirm, close };
}
```

**`ModalContainer.vue`**：渲染 Modal UI（遮罩、对话框、按钮），通过 `state` 绑定。

### 5.4 HTTP 客户端增强

现有 `frontend/src/lib/http.ts` 使用 `requestJson<T>(url, init?)` 函数模式（无拦截器/axios 式 API 客户端）。v0.4 需在 `requestJson` 中扩展写操作的 admin token 注入。

**修改方案**：

```typescript
// frontend/src/lib/http.ts — 在现有代码基础上修改

export type HttpMethod = "GET" | "POST" | "PUT" | "PATCH" | "DELETE";

export class HttpError extends Error {
  status: number; bodyText?: string;
  constructor(status: number, message: string, bodyText?: string) {
    super(message); this.status = status; this.bodyText = bodyText;
  }
}

// Admin token 解析（仅在需要时计算）
function getAdminToken(): string {
  // Vite 环境变量（构建时注入）优先，localStorage fallback（开发便利）
  return import.meta.env.VITE_ADMIN_TOKEN || localStorage.getItem("__admin_token__") || "";
}

// 需要鉴权的 HTTP Method
const WRITE_METHODS = new Set<HttpMethod>(["POST", "PUT", "PATCH", "DELETE"]);

export async function requestJson<T>(
  url: string,
  init?: { method?: HttpMethod; body?: unknown },
): Promise<T> {
  const method = init?.method || "GET";
  const headers: Record<string, string> = { "Content-Type": "application/json" };

  // Admin token 注入（仅写操作）
  if (WRITE_METHODS.has(method)) {
    const token = getAdminToken();
    if (token) headers["x-admin-token"] = token;
  }

  const res = await fetch(url, {
    method,
    headers,
    body: init?.body ? JSON.stringify(init.body) : undefined,
  });

  if (!res.ok) {
    const text = await res.text().catch(() => "");
    throw new HttpError(res.status, `HTTP ${res.status}`, text);
  }
  return (await res.json()) as T;
}
```

**变更说明**：
- 在现有 `requestJson` 函数内追加 token 注入逻辑，不改变函数签名（`api.ts` 中 20+ 调用方无需修改）
- `getAdminToken()` 优先级：`VITE_ADMIN_TOKEN` 环境变量（生产）→ `localStorage`（开发 fallback）
- Token 仅对 POST/PUT/PATCH/DELETE 附加，GET 请求不携带
- 安全边界已在 PRD 3.2.6 记录

## 7. 风险和取舍

### 7.1 已知风险

| 风险 | 可能性 | 影响 | 缓解 |
|------|--------|------|------|
| `rss-parser` 对非标准 RSS feed 解析失败 | 中 | 部分 RSS 源不可用 | 先支持 RSS 2.0 + Atom 主流格式；解析失败抛 NonRetryableError 不阻塞队列 |
| ILIKE 全表扫描性能 | 低 | 搜索慢 | v0.4 数据量 < 5 万条可接受；PRD 已规划 v0.6+ 全文搜索 |
| Admin Guard 扩展影响现有部署 | 低 | 升级后管理端点不可用 | `ADMIN_PROTECT_READS` 默认 false，需主动开启 |
| 频道空间删除不可逆 | 低 | 误删 | 前端二次确认（输入名称），后端可考虑软删除（v0.5+） |

### 7.2 取舍决策

| 决策 | 选择 | 替代方案 | 理由 |
|------|------|---------|------|
| RSS 解析库 | `rss-parser` | 手写 XML 解析 | 成熟稳定，TypeScript 类型内置，减少维护负担 |
| Toast/Modal 注入 | 模块级单例（方案 B） | provide/inject（方案 A） | 命令式调用场景更自然，组件树解耦 |
| 搜索实现 | PostgreSQL ILIKE | pg_trgm + GIN 索引 | v0.4 数据量小，简单方案优先；已规划升级路径 |
| Admin Guard 扩展 | 默认 false 的 feature flag | 直接启用 | 保持向后兼容，让用户自主选择 |
| 频道删除 | 硬删除 + 前端确认 | 软删除 | 减少复杂度；v0.5+ 可升级 |
| RSS validate | fetch 空跑判断 | 扩展 Fetcher 接口增加 `validate()` | 保持接口简洁，设计阶段评估后再决定 |

### 7.3 未覆盖事项（后续迭代）

- Fetcher 接口 `validate?()` 可选成员（设计阶段评估后再决定，PRD 3.2.4 已注明）
- 全文搜索索引（pg_trgm / Elasticsearch）→ v0.6+
- 频道空间软删除 → v0.5+
- 自动刷新日志 → v0.5+（v0.4 手动刷新）

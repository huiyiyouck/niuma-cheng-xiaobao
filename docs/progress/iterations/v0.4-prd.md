## 2. 用户故事

### 2.1 UI 重构

**US-2.1.1** 作为产品所有者，我希望管理页面不再使用四个 Tab 切换，而是以频道空间为唯一视口——信息源卡片列表为主体、子频道管理为辅助抽屉，以便在一个视图中完成所有管理操作。

**US-2.1.2** 作为产品所有者，我希望系统日志从管理页的 Tab 中独立出来，作为与"浏览""管理"平级的全局顶级导航项，因为日志不隶属于任何频道空间。

**US-2.1.3** 作为产品所有者，我希望所有弹窗（删除确认、编辑、验证等）使用自定义 UI 而非浏览器原生 `alert()` / `confirm()`，以保持视觉一致性。

**US-2.1.4** 作为产品所有者，我希望看到统一的 Toast 通知（成功/错误/信息，右上角滑入，2.5 秒自动消失），替代页面内嵌的错误文字。

### 2.2 功能闭环

**US-2.2.1** 作为产品所有者，我希望能够重命名和删除频道空间。当前只能创建和列出，建错名称后无法修改。

**US-2.2.2** 作为产品所有者，我希望能够将信息源从频道解绑，而不是只能删除整个 Source。删除 Source 会导致所有频道中的绑定级联消失，风险太高。

**US-2.2.3** 作为产品所有者，我希望添加 RSS 类型的 Source 后系统能真正抓取，而不是报错"抓取器未实现"。RSS 是最基础的信息源协议。

**US-2.2.4** 作为产品所有者，我希望告警列表中的告警可标记为"已处理"，以便区分新告警和历史告警。

**US-2.2.5** 作为产品所有者，我希望生产环境配置 ADMIN_TOKEN 后，前端的写操作不会全部 403 失败。

### 2.3 功能增强

**US-2.3.1** 作为产品所有者，我希望在浏览页通过关键词搜索新闻（标题+摘要模糊匹配），以便快速找到特定内容。

**US-2.3.2** 作为产品所有者，我希望在删除 Source 时系统明确提示该 Source 已绑定了哪些频道空间，并阻止意外删除（或给出清晰的级联影响说明）。

**US-2.3.3** 作为产品所有者，我希望管理员鉴权覆盖日志查询端点（当前任何人可访问），以及有个开关可以控制是否对读取操作也要求鉴权。

**US-2.3.4** 作为产品所有者，我希望列表加载时展示骨架屏而非"加载中…"文字，以提供更好的加载体验。

**US-2.3.5** 作为产品所有者，我希望快速切换子频道筛选或排序方式时，不会因为瞬间大量请求导致页面抖动。

#### 3.2.4 RSS Fetcher 实现

新增 `server/src/worker/fetchers/rss.ts`，支持：

**接口对接**（与现有 Fetcher 注册表 `server/src/worker/fetchers/registry.ts` 对齐）：
- `type` 标识值：`"rss"`（与 `sources.type` 枚举值对齐；X/Twitter 使用 `"x_twitter"`，RSS 使用 `"rss"`）
- `fetch()`：标准抓取逻辑
- `renderForLLM()`：LLM 格式化
- **validate 策略**：调用 `fetch()` 空跑判断（能成功获取 feed 并解析出条目 = 可验证通过）。暂不扩展 Fetcher 接口增加可选 `validate` 方法，由设计阶段评估是否为接口增加 `validate?()` 可选成员。

**功能行为**：
- 标准 RSS 2.0 和 Atom 格式解析
- HTTP 请求获取 XML 内容，超时 15 秒，设置 User-Agent
- 提取字段映射：`<title>`、`<link>`、`<description>`、`<pubDate>` / `<published>` / `<updated>`
- `source_item_id` 使用 `<guid>` 或 `<link>`（fallback）
- 游标机制：利用 `source_states.cursor`（jsonb）存储最近成功抓取文章的 `pubDate`，下次抓取时只取更新的（与现有 `x_twitter` Fetcher 的游标设计一致）
- 注册到 registry（`fetchers/index.ts` 增加 `import "./rss.ts"`）

非 Retryable 场景：HTTP 404（feed 不存在）、XML 解析失败（内容不是有效 RSS/Atom）

#### 3.2.5 告警已读状态

- `alerts` 表新增 `status` 字段：`VARCHAR(20) NOT NULL DEFAULT 'active'`（值：active / acknowledged / resolved）

**状态转换规则**：
- 允许：`active → acknowledged`、`acknowledged → resolved`、`active → resolved`（可跳过 acknowledged）
- 允许回退：`acknowledged → active`（告警复现时回退为 active）
- 不允许：`resolved → active` 或 `resolved → acknowledged`（已解决告警不得回退，如需重新告警应由 Worker 生成新告警记录）
- `POST /v1/alerts/acknowledge-all` 仅影响 `status = 'active'` 的告警

- 新增端点：

| 方法 | 路径 | 说明 |
|------|------|------|
| PATCH | `/v1/alerts/:id` | 更新告警状态（body: `{ status }`），校验状态转换合法性 |
| POST | `/v1/alerts/acknowledge-all` | 批量标记所有 active 告警为 acknowledged（可选按 channel_space_id 过滤） |

- 前端告警列表展示状态标签（颜色区分：active 红色、acknowledged 橙色、resolved 灰色），支持单条或批量标记

#### 3.2.6 前端写操作鉴权

`frontend/src/lib/http.ts` 修改：

- 对 POST/PUT/PATCH/DELETE 请求自动附加 `x-admin-token` header
- admin token 来源策略（按环境区分）：
  - **生产环境**：使用 `VITE_ADMIN_TOKEN` 环境变量（构建时注入，不可被前端 JS 读取修改）

#### 3.3.6 Toast/Modal 全局注入方式

Toast（`useToast`）和 Modal（`useModal`）作为全局单例，有两种实现路径：

- **方案 A**：`App.vue` 根节点 `provide` 注入单例实例，子组件 `inject` 获取
- **方案 B**：模块级响应式状态（module-level reactive state），`useToast()` / `useModal()` 直接导入

两种方案对组件使用方式无影响（都是 `const toast = useToast()`），由设计阶段评估后选择。此外：

## 7. 风险与缓解

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|---------|
| RSS 格式多样性导致解析失败 | 高 | 部分 RSS 源无法抓取 | 先支持 RSS 2.0 + Atom 两种最广泛的标准格式；解析失败抛出 NonRetryableError |
| 管理页重构工作量大、前端开发周期长 | 中 | 延迟 v0.4 交付 | 原型已完整确认交互，开发可 1:1 对照实现 |
| 频道空间 DELETE 级联影响数据量大 | 低 | 用户误删重要数据 | 删除前 API 返回受影响数据统计；前端二次确认 + 输入空间名称确认 |
| 前端防抖可能导致用户感知延迟 | 低 | 点击后 300ms 才刷新 | 300ms 为常用值，用户基本无感 |
| RSS Fetcher HTTP 请求可能超时或被屏蔽 | 中 | 特定 feed 源无法抓取 | 15s 超时 + User-Agent 设置 + 重试机制 |

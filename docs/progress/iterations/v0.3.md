**#1 Fetcher 注册表形同虚设（架构阻断）**

技术评估 §5.2 要求 Dispatcher 通过 `registry.find(type)` 查找 Fetcher，新增数据源只写一个文件+register，不动核心逻辑。`registry.ts` 接口和 `register()/find()` 已写好，`x_twitter.ts` 也调了 `register()` 注册自己——但 **`dispatcher.ts` 根本没 import `find`**。三处 hardcode source_type：

| 位置 | 当前代码 |
|------|---------|
| `dispatcher.ts:134` `fetchAndIngest()` | `if (row.source_type === "x_twitter")` → 直接调 `xt.parseXTweets()` |
| `dispatcher.ts:198` `processOne()` | `if (row.source_type === "x_twitter")` → 直接调 `xt.renderTextForLLM()` |
| `sources.ts:213` `verifyFetch()` | `if (sourceType === "x_twitter")` → `await import("../../worker/fetchers/x_twitter.ts")` |

新增 RSS Fetcher 要改 3 个文件，和 Python 版一样的问题。

**修复方向**：
```typescript
// dispatcher.ts
import { find } from "./fetchers/registry.ts";

// fetchAndIngest: 替换 hardcode
const fetcher = find(row.source_type);
if (!fetcher) throw new NonRetryableError(`未注册的 Source 类型：${row.source_type}`);
const fetchResult = await fetcher.fetch(row.source_config, row.cursor || {}, maxItems);

// processOne: 替换 hardcode
const fetcher = find(row.source_type);
const text = fetcher ? fetcher.renderForLLM(content) : JSON.stringify(content);

// sources.ts verifyFetch: 替换 hardcode + dynamic import
const fetcher = find(sourceType);
if (!fetcher) throw new NonRetryableError(`未注册的 Source 类型：${sourceType}`);
const fetchResult = await fetcher.fetch(config, {}, 5);
```

**#2 Worker 模块未拆分**

技术评估 §5.1 要求拆为 `scheduler.ts / dispatcher.ts / processor.ts / monitor.ts / reclaim.ts` 独立文件。当前全部塞在 `dispatcher.ts`（415行），函数名上按逻辑分了段但物理上是同一文件——和 Python `main.py` 602 行同样的问题。

---

**#3 DB Schema 缺两个索引**

DDL (`db/schema.sql`) 有，Drizzle schema (`src/db/schema.ts`) 漏了：

| 索引名 | 定义 | 影响 |
|--------|------|------|
| `ix_processed_news_sub_published` | `processed_news(channel_space_id, sub_channel_id, published_at DESC)` | 子频道筛选+排序查询走不到索引 |
| `ix_raw_items_url` | `raw_items(source_item_url) WHERE source_item_url IS NOT NULL` | URL 去重查询（`processOne` 用到）走不到索引 |

---

**#4 source-detector HTTP HEAD 请求缺 scheme 校验（建议）**

`source-detector.ts:72` — `fetch(sourceUrl, ...)` 对用户输入的 URL 没有 scheme 白名单。`file://` 等协议可能触发 SSRF。建议加：`if (!/^https?:\/\//i.test(sourceUrl)) return SourceType.unknown;`

---

**#5 news.ts 排序参数未用 Zod 校验（建议）**

`news.ts:12` 直接读 `req.query.sort`，但 `NewsQuery` schema 已定义 `z.enum(["published_desc", "score_desc", "score_asc"])` 校验。当前 if/else + default 兜底不会出错，但与其他路由不一致（其他路由都用 Zod parse）。

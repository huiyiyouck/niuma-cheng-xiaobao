## 1. 概述

本文档是 v0.3 迭代"后端技术栈迁移 Python(FastAPI) → Node.js"的技术预研评估。主要回答两个问题：
1. 能不能迁？
2. 怎么迁？

v0.3 方向由 PM 在 2026-05-27 提出，动机：统一前后端语言（全栈 JS/TS），单进程单端口部署，降低上下文切换成本。

### 2.3 Worker 当前架构

```
worker/main.py（602行，4个协程并发运行）：

  scheduler_loop ─── 每5秒扫 channel_sources
    ├─ 查 fetch_policy.schedule.every_seconds 判断是否到抓取时间
    ├─ JOIN sources WHERE status='active'
    └─ 创建 fetch task 写入 tasks 表

  worker_loop ─── 信号量控制并发（fetch=2, process=1）
    ├─ 抢 fetch task → fetch_and_ingest()
    │   └─ source_type == "x_twitter" → parse_x_twitter()
    │       └─ 其他类型 → NonRetryableError("尚未实现")
    └─ 抢 process task → process_one()
        └─ source_type == "x_twitter" → render_text_for_llm()
            → translate_and_summarize() → 写 processed_news
            → pg_notify('news_processed') → WebSocket 推前端

  zero_new_monitor_loop ─── 每1h 检查零新增 → 创建告警
  reclaim_stale_running_loop ─── 每30s 回收超时 task
```

**当前问题**：
- `fetch_and_ingest` 中 hardcode if/else 按 source_type 分发，新增数据源要改核心逻辑
- Scheduler、Fetcher、Processor 全耦合在一个文件
- `pg_notify → WebSocket` 链路增加不必要的复杂度（前端刷新页面就能拿到数据）

## 4. 目标架构

### 4.1 整体拓扑

```
┌─ Node.js 单进程 ──────────────────────────────────────┐
│                                                         │
│  Fastify HTTP Server                                    │
│  ├─ /v1/*        REST API（Zod → Drizzle → PG）       │
│  ├─ /assets/*    前端静态文件                           │
│  └─ /            SPA fallback (index.html)             │
│                                                         │
│  ┌─ Worker（同进程，setInterval 驱动）───────────────┐  │
│  │                                                    │  │
│  │  Scheduler ──→ 创建 fetch task                    │  │
│  │  Dispatcher ──→ 按 source_type 分发到 Fetcher     │  │
│  │  Processor ──→ LLM 处理 → 写 processed_news      │  │
│  │  Monitor ────→ 零新增告警                         │  │
│  │  Reclaim ────→ 超时 task 回收                     │  │
│  │                                                    │  │
│  └────────────────────────────────────────────────────┘  │
│                         ↕                                │
│                PostgreSQL (Supabase)                     │
└─────────────────────────────────────────────────────────┘
```

### 4.2 技术选型

| 层 | 选型 | 理由 |
|----|------|------|
| HTTP 框架 | **Fastify** | 与 FastAPI 结构最像（router→route→schema），直译成本最低；性能最高；TypeScript 原生 |
| 数据校验 | **Zod** | 与 Pydantic 模式等价，`z.object` + `.refine` 对应 `BaseModel` + `validator` |
| ORM | **Drizzle** | JSONB 原生支持好；Supabase 兼容（PgBouncer 友好）；类 SQL 写法不学新 DSL |
| PG 驱动 | **pg** (node-postgres) | Drizzle 底层依赖，Worker 直连也用它 |
| 日志 | **winston** + DailyRotateFile | 等价 Python `TimedRotatingFileHandler` + JSON 格式 |
| Worker 方式 | **同进程 setInterval** | 全是网络 IO（X API + LLM API + PG），无 CPU 密集，Node.js 事件循环处理毫无压力 |
| WebSocket | **砍掉** | 见 4.3 节 |
| Redis/BullMQ | **不引入** | 当前 PG Task Queue 够用，不增加运维复杂度 |

### 4.3 砍掉 WebSocket

当前链路：

```
Worker LLM写完 → pg_notify → API收到通知 → WebSocket推前端
```

前端展示的数据本来就从 API 读数据库来。WebSocket 的唯一价值是"新新闻入库，前端秒级看到"，但用户刷新页面或切换 Tab 时调 API 一样能拿到。当前阶段没有真正的实时需求（如多用户协作、告警弹窗）。

**砍掉后减少的代码：**

| 文件 | 行数 | 动作 |
|------|------|------|
| `app/pg_notify.py` | 57 | 删除 |
| `app/ws_manager.py` | 40 | 删除 |
| `app/main.py` WS端点 | ~20 | 删除 |
| **合计** | **~117** | |

## 5. Worker 可扩展重构方案

### 5.1 文件结构

```
worker/
  scheduler.ts       ← 调度：扫 channel_sources，创建 fetch task
  dispatcher.ts      ← 分发：按 source_type 找 Fetcher 执行
  fetchers/
    registry.ts      ← Fetcher 注册表
    x_twitter.ts     ← X/Twitter 抓取
    rss.ts           ← 未来扩展
  processor.ts       ← 处理：调 LLM → 写 processed_news
  llm.ts             ← LLM 调用封装
  monitor.ts         ← 零新增告警
  reclaim.ts         ← 超时 task 回收
  index.ts           ← 入口，启动所有循环
```

### 5.2 Fetcher 插件接口

```typescript
// worker/fetchers/registry.ts

interface FetchResult {
  items: RawItem[];        // 抓到的数据
  cursorUpdates: Cursor;   // 游标更新（如 X API 的 since_id）
}

interface Fetcher {
  /** source_type 标识，需与 sources.type 字段匹配 */
  type: string;
  /** 执行一次抓取 */
  fetch(config: SourceConfig, cursor: Cursor, maxItems: number): Promise<FetchResult>;
  /** 将原始 content 渲染为给 LLM 的文本 */
  renderForLLM(content: Record<string, unknown>): string;
}

// 注册表：{ "x_twitter": XTwitterFetcher, "rss": RssFetcher, ... }
const registry = new Map<string, Fetcher>();

export function register(f: Fetcher) { registry.set(f.type, f); }
export function find(type: string): Fetcher | undefined { return registry.get(type); }
```

**新增数据源 = 写一个文件 + 调 `register()`**。不动 dispatcher、scheduler、processor。

```typescript
// worker/fetchers/rss.ts  ← 未来新增 RSS
import { register } from './registry';

register({
  type: 'rss',
  async fetch(config, cursor, maxItems) { /* RSS 抓取逻辑 */ },
  renderForLLM(content) { /* RSS 文本组装 */ },
});
```

### 5.3 调度器

```typescript
// worker/scheduler.ts

async function tick(pool: Pool) {
  // 1. 查全部活跃 channel_source（JOIN sources WHERE status='active' AND enabled=true）
  const rows = await pool.query(`
    SELECT cs.*, s.type as source_type, s.config as source_config, ss.next_fetch_at
    FROM channel_sources cs
    JOIN sources s ON s.id = cs.source_id
    LEFT JOIN source_states ss ON ss.channel_source_id = cs.id
    WHERE cs.enabled = true AND s.status = 'active'
  `);

  const now = new Date();
  for (const row of rows) {
    // 2. 没到抓取时间，跳过
    if (row.next_fetch_at && row.next_fetch_at > now) continue;

    // 3. 已有排队中/执行中的 fetch task，跳过
    const exists = await pool.query(`
      SELECT 1 FROM tasks
      WHERE status IN ('queued','running') AND type='fetch' AND channel_source_id=$1
      LIMIT 1
    `, [row.channel_source_id]);
    if (exists.rows.length > 0) continue;

    // 4. 创建 fetch task，更新 next_fetch_at
    const everySeconds = row.fetch_policy?.schedule?.every_seconds || 600;
    await pool.query(`
      INSERT INTO tasks(type, channel_space_id, channel_source_id, status, priority, run_after)
      VALUES('fetch', $1, $2, 'queued', 0, now())
    `, [row.channel_space_id, row.channel_source_id]);

    await pool.query(`
      INSERT INTO source_states(channel_source_id, cursor, next_fetch_at, updated_at)
      VALUES($1, '{}', now() + interval '${everySeconds} seconds', now())
      ON CONFLICT (channel_source_id) DO UPDATE SET next_fetch_at=EXCLUDED.next_fetch_at
    `, [row.channel_source_id]);
  }
}

// 入口：每 5 秒执行一次
export function start(pool: Pool, stop: AbortSignal) {
  const loop = () => {
    if (stop.aborted) return;
    tick(pool).finally(() => setTimeout(loop, 5000));
  };
  loop();
}
```

**调度配置已经在表里，无需改 Schema**：

```json
// channel_sources.fetch_policy 字段（JSONB）
{
  "schedule": { "every_seconds": 600 },    // 每10分钟抓一次
  "budget":   { "max_items_per_run": 20 }  // 每次最多20条
}
```

每个 Source 独立配置频率和容量，互不干扰。

### 5.4 分发器

```typescript
// worker/dispatcher.ts

import { find } from './fetchers/registry';

async function execute(pool: Pool, task: Task) {
  // 1. 查出 source 信息
  const row = await pool.query(`
    SELECT cs.*, s.type, s.config
    FROM channel_sources cs
    JOIN sources s ON s.id = cs.source_id
    WHERE cs.id = $1
  `, [task.channel_source_id]);

  const { type, config, channel_space_id, source_id } = row.rows[0];

  // 2. 找对应 Fetcher
  const fetcher = find(type);
  if (!fetcher) throw new Error(`未注册的 Source 类型: ${type}`);

  // 3. 执行抓取
  const { items, cursorUpdates } = await fetcher.fetch(config, cursor, maxItems);

  // 4. 批量写 raw_items + 创建 process task
  for (const item of items) {
    const { id } = await pool.query(`
      INSERT INTO raw_items(...) VALUES(...) ON CONFLICT DO NOTHING RETURNING id
    `);
    if (id) {
      await pool.query(`
        INSERT INTO tasks(type, channel_space_id, raw_item_id, status, priority, run_after)
        VALUES('process', $1, $2, 'queued', 0, now())
      `, [channel_space_id, id]);
    }
  }

  // 5. 更新游标
  await pool.query(`UPDATE source_states SET cursor = cursor || $1 WHERE ...`, [cursorUpdates]);
}
```

**分发器不关心 source_type 是什么**——查 registry → 调 fetch() → 入库。新增数据源零改动。

### 5.5 处理器 + LLM

```typescript
// worker/processor.ts

import { find } from './fetchers/registry';
import { process } from './llm';

async function execute(pool: Pool, task: Task) {
  const row = await pool.query(`
    SELECT ri.*, s.type FROM raw_items ri
    JOIN sources s ON s.id = ri.source_id WHERE ri.id = $1
  `, [task.raw_item_id]);

  const { type, content, source_item_url } = row.rows[0];

  // 用 Fetcher 自带的 renderForLLM 组装文本
  const fetcher = find(type);
  const text = fetcher?.renderForLLM(content) ?? JSON.stringify(content);

  // LLM 处理
  const result = await process(text, source_item_url);

  // 写 processed_news
  await pool.query(`
    INSERT INTO processed_news(...) VALUES(...) ON CONFLICT (raw_item_id) DO NOTHING
  `);
}
```

## 7. 迁移策略

### 7.1 三步走

```
第1步：数据库层
  ├─ Drizzle Schema 定义（对齐现有 DDL）
  ├─ pg Pool 连接
  └─ 验证：基础 CRUD 测试脚本对 Supabase 执行

第2步：API 层
  ├─ Fastify + Zod 路由（11 个路由文件）
  ├─ 中间件（CORS、Admin IP、HTTP 日志）
  ├─ Winston 日志（JSON 格式 + 按天轮转）
  └─ 验证：前端指向新后端（不启动 Python），所有页面功能可用

第3步：Worker + 部署
  ├─ 5 个 Worker 文件（scheduler/dispatcher/processor/monitor/reclaim）
  ├─ 1 个 Fetcher（x_twitter.ts，从 Python 直译）
  ├─ LLM 模块（功能等价）
  ├─ Dockerfile → 单进程镜像
  └─ 验证：端到端——Source 按配置自动抓取 → LLM 处理 → 前端展示
```

### 7.2 不做的

- 不改数据库 Schema（DDL 原样保留）
- 不改 API 路径/参数/返回值（前端零改动）
- 不重写 LLM prompt（功能等价，独立迭代再做）
- 不引入 Redis / BullMQ / 消息队列

## 8. 风险与对策

| 风险 | 等级 | 对策 |
|------|------|------|
| Drizzle JSONB 读写边界（null vs {}） | 低 | 当前 Python `_as_dict()` 已做兜底，TS 侧同样封装一个 `safeJsonb()` |
| LLM 调用超时控制 | 低 | Node.js `AbortSignal.timeout(15000)` 等价 `asyncio.wait_for(15s)` |
| X API 限流/鉴权处理 | 低 | 当前逻辑是 `if 401/403 → NonRetryableError` + `if 429 → throw retryable`，fetch 版直接翻译 |
| 同进程 Worker 阻塞事件循环 | 极低 | LLM 调用和 X API 调用全是网络 IO，不占 CPU；监控/回收循环间隔长 |
| Supabase PgBouncer 连接池限制 | 低 | Drizzle 和 pg 均支持连接池模式，不强制独占连接 |

### #1 迁移验证策略（建议补充）

- 每个端点的请求/响应格式与 Python 版对比

- 排序、分页、筛选行为的等价性

这份清单在实现阶段由开发自检 + PM Review 逐条对照，避免"功能看着能用但角落 case 不一致"的回归。

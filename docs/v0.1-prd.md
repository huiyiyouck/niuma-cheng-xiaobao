## 1. 产品概述

### 1.1 产品名称

牛马程小报（News Aggregator）

### 1.2 产品定位

面向个人/小团队的新闻聚合阅读平台。从多个外部渠道（RSS、学术论文、社交媒体、开源社区）自动抓取原始素材，经 AI 处理后生成结构化中文新闻卡片，按频道空间和子频道组织展示。

### 1.3 目标用户

一人公司内部使用，暂不对外开放。产品所有者即唯一用户。

### 1.4 产品愿景

打造一个可扩展的多领域新闻聚合平台，当前以 AI 领域为 MVP 切入点，后续逐步扩展至财经、体育、历史、军事等方向。每个领域作为一个独立的频道空间，空间内可按子频道进一步分类。

### 1.5 当前状态

项目已完成 MVP 基础框架，包含：
- 后端 API（FastAPI + PostgreSQL/Supabase）
- Worker（抓取 + LLM 处理）
- 前端（Vue3 + TypeScript，浏览页 + 管理页）
- 已支持的 Source 类型：RSS、HuggingFace Daily Papers、HackerNews、GitHub Trending、Semantic Scholar

### 1.6 现有架构参考

参见项目根目录的 `docs/news-aggregator-design.md` 和 `docs/使用说明.md`。

> **与现有设计文档的关系**：`docs/news-aggregator-design.md` 中的 `processing_pipelines` 表、`channel_space_shares` 表、`processing_pipelines.prompt_version` 等概念，v0.1 不引入，留待后续迭代。本 PRD 以当前实际代码为基准，描述 v0.1 的增量变更。

## 4. 功能需求

### 4.1 子频道（Sub-Channel）

#### 4.1.1 数据模型

新增 `sub_channels` 表，挂在 `channel_spaces` 下：

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | UUID | PK, default gen_random_uuid() | |
| `channel_space_id` | UUID | FK → channel_spaces.id, ON DELETE CASCADE, NOT NULL | |
| `name` | VARCHAR(100) | NOT NULL | 同一空间下唯一 |
| `sort_order` | INT | DEFAULT 0 | 越小越靠前 |
| `created_at` | TIMESTAMPTZ | DEFAULT now(), NOT NULL | |

约束：`UNIQUE(channel_space_id, name)`

索引：`ix_sub_channels_space_sort(channel_space_id, sort_order)`

`channel_sources` 表新增字段：

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `sub_channel_id` | UUID | FK → sub_channels.id, ON DELETE SET NULL, NULLABLE | 可选，指定绑定的子频道 |

`processed_news` 表新增字段：

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `sub_channel_id` | UUID | FK → sub_channels.id, ON DELETE SET NULL, NULLABLE | 冗余字段，从 channel_sources 继承 |

新增索引：
```sql
CREATE INDEX ix_processed_news_sub_published
  ON processed_news(channel_space_id, sub_channel_id, published_at DESC);
```

**`processed_news.sub_channel_id` 数据流**（解决 #1）：

`process_one()` 在处理 raw_item 时，通过以下路径获取 `sub_channel_id`：

```
raw_item.source_id + raw_item.channel_space_id
  → channel_sources（WHERE source_id = ? AND channel_space_id = ?）
    → channel_sources.sub_channel_id
      → 写入 processed_news.sub_channel_id
```

在 `worker/main.py` 的 `process_one()` 函数中，需要新增一次查询：

```python
sub_channel_id = await conn.fetchval(
    """
    SELECT sub_channel_id FROM channel_sources
    WHERE source_id = $1 AND channel_space_id = $2
    LIMIT 1
    """,
    row["source_id"],
    row["channel_space_id"],
)
```

如果一个 source 绑定到同一空间的多个 channel_source（理论上不应出现，因为 `UNIQUE(channel_space_id, source_id)` 约束），取第一个即可。

#### 4.1.2 CRUD API

- `POST /v1/channel-spaces/{channel_space_id}/sub-channels` — 创建子频道
- `GET /v1/channel-spaces/{channel_space_id}/sub-channels` — 列出子频道（按 sort_order 排序）
- `PUT /v1/sub-channels/{sub_channel_id}` — 更新子频道（名称、排序）
- `DELETE /v1/sub-channels/{sub_channel_id}` — 删除子频道

#### 4.1.3 绑定变更

- `POST /v1/channel-spaces/{channel_space_id}/sources`（绑定）body 新增可选字段 `sub_channel_id`
- `PUT /v1/channel-sources/{channel_source_id}` body 新增可选字段 `sub_channel_id`
- 绑定后更改子频道时，已处理的新闻 `sub_channel_id` 不追溯更新（仅影响新入库的新闻）

#### 4.1.4 新闻查询

- `GET /v1/channel-spaces/{channel_space_id}/news` 新增查询参数 `sub_channel_id`（可选），按子频道筛选
- 不传 `sub_channel_id` 时返回该空间下所有新闻（行为不变）

### 4.2 X/Twitter 数据源

#### 4.2.1 Source 类型定义

`type` = `x_twitter`

#### 4.2.2 Source config 结构

v0.1 支持两种独立模式，每种模式创建一个独立的 Source。用户如需同时使用关键词搜索和账号追踪，创建两个 Source 即可。

**搜索模式：**
```json
{
  "mode": "search",
  "search_query": "LLM OR GPT OR Claude",
  "max_results_per_query": 20
}
```

**账号追踪模式：**
```json
{
  "mode": "user_timeline",
  "usernames": ["OpenAI", "AnthropicAI"],
  "max_results_per_user": 20
}
```

字段说明：
- `mode`：必填，`search` 或 `user_timeline`
- `search_query`：mode = search 时必填
- `usernames`：mode = user_timeline 时必填，数组
- `max_results_per_query`：搜索模式每次查询最大返回数，默认 20，上限 100
- `max_results_per_user`：账号追踪模式每个账号每次最大返回数，默认 20，上限 100

> v0.1 不引入 `combined` 模式。理由：用户可创建多个 Source 分别配置，合并模式增加游标管理复杂度且一个 API 限流会影响另一个。

> `tweet_fields` 和 `user_fields` 不暴露给用户配置，在 `worker/fetch_x_twitter.py` 中硬编码默认值。

#### 4.2.3 环境变量新增

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `X_BEARER_TOKEN` | 无 | X API v2 Bearer Token（App-Only 认证） |

v0.1 仅使用 Bearer Token（App-Only），满足 Basic 套餐的 recent search 和 user timeline 需求。OAuth 1.0a 凭证暂不引入，如需 User Context 认证时再扩展。

#### 4.2.4 游标（Source State Cursor）

搜索模式游标：
```json
{
  "since_id": "1760012345678900000"
}
```

账号追踪游标：
```json
{
  "user_cursors": {
    "OpenAI": "1760012345678900000",
    "AnthropicAI": "1759999999999900000"
  }
}
```

增量逻辑：
- 搜索模式：每次请求带上 `since_id`，只拉取该 ID 之后的新推文。首次拉取不设 `since_id`，拉取最近推文。
- 账号追踪：每个用户名维护独立的游标，逐用户增量拉取。

#### 4.2.5 Raw Item 内容结构

```json
{
  "tweet_id": "1760012345678900000",
  "text": "推文文本...",
  "author_username": "OpenAI",
  "author_name": "OpenAI",
  "created_at": "2026-05-20T10:30:00Z",
  "public_metrics": {
    "retweet_count": 120,
    "reply_count": 45,
    "like_count": 890,
    "quote_count": 30,
    "bookmark_count": 200,
    "impression_count": 50000
  },
  "entities": {
    "urls": [{"expanded_url": "...", "title": "..."}],
    "mentions": [{"username": "..."}],
    "hashtags": [{"tag": "..."}]
  },
  "referenced_tweets": [{"type": "quoted", "id": "..."}],
  "url": "https://x.com/OpenAI/status/1760012345678900000"
```

#### 4.2.6 LLM 渲染文本

`render_text_for_llm` 输出格式：

```
[Tweet by @OpenAI]
文本内容...

Likes: 890 | Retweets: 120 | Replies: 45

```

#### 4.2.7 Worker 实现文件

新增 `worker/fetch_x_twitter.py`，包含：
- `async def fetch_x_search(client, config, cursor, max_items)` — 关键词搜索
- `async def fetch_user_timeline(client, username, cursor, max_items)` — 单账号追踪
- `async def parse_x_twitter(config, cursor, max_items)` — 主入口，根据 mode 调度
- `def render_text_for_llm(content)` — 渲染给 LLM 的文本

X API 请求参数（tweet_fields、user_fields、expansions）在此文件中硬编码，不从 config 读取。

#### 4.2.8 限流与预算

- X Basic 套餐限制：recent search 每月 10,000 条，60 requests / 15 min
- `fetch_policy.budget.max_items_per_run` 控制每次抓取上限
- Worker 应在请求失败 429 时触发退避重试（已有 `_requeue_task` 机制）

#### 4.2.9 Worker 调度集成

在 `worker/main.py` 的 `fetch_and_ingest()` 函数中，新增 `x_twitter` 分支：

```python
elif source_type == "x_twitter":
    cursor_data = cursor  # 当前游标
    items = await parse_x_twitter(source_config, cursor_data, max_items)
    cursor_updates = items.cursor_updates  # parse_x_twitter 返回游标更新
```

`parse_x_twitter` 返回结构改为 `(items: list[dict], cursor_updates: dict)`，以便外部拿到更新后的游标。

### 4.3 LLM 处理增强

#### 4.3.1 输出结构变更

现有输出：
```json
{"title": "...", "summary": "...", "language": "zh"}
```

改为：
```json
{
  "title": "中文标题",
  "summary": "80-200字中文摘要",
  "bullets": ["要点1", "要点2", "要点3"],
  "tags": ["标签1", "标签2", "标签3"],
  "entities": [
    {"name": "实体名", "type": "person|org|product|tech"}
  ],
  "importance_score": 7.5,
  "language": "zh"
}
```

字段说明：
- `bullets`：3-5 个核心要点，每个不超过 50 字
- `tags`：3-5 个分类标签
- `entities`：识别的关键实体（人名、组织、产品、技术）
- `importance_score`：0-10 浮点数，综合考虑时效性、影响力、相关性
- `source_refs`：保持现状，由 `process_one` 从 `raw_items.source_item_url` + `source_id` 构造，不在 LLM 输出中

#### 4.3.2 Prompt 变更

`worker/llm.py` 中 `translate_and_summarize` 的 prompt 替换为：

```
把下面内容处理成中文新闻卡片，做翻译与深度摘要。
要求：
1) 如果原文是英文，翻译成中文
2) 输出中文标题(title)和中文摘要(summary)
3) summary 80-200 字
4) 提炼 3-5 个核心要点(bullets)，每个不超过 50 字
5) 生成 3-5 个分类标签(tags)
6) 识别关键实体(entities)，每个包含 name 和 type(person/org/product/tech)
7) 评估重要性评分(importance_score)，0-10 浮点数，考虑时效性、影响力、相关性
8) 只输出严格 JSON，不要输出多余文字
JSON 格式：{"title":"...","summary":"...","bullets":["..."],"tags":["..."],"entities":[{"name":"...","type":"..."}],"importance_score":7.5,"language":"zh"}
source_url: {source_url}

content:
{text}
```

#### 4.3.3 输出校验规则

LLM 输出解析后需进行字段校验和修正（在 `translate_and_summarize` 返回前执行）：

| 字段 | 校验规则 | 异常处理 |
|------|---------|---------|
| `title` | 必须为非空字符串 | 缺失时用 `"无标题"` 填充 |
| `summary` | 必须为非空字符串 | 缺失时用 `title` 填充 |
| `bullets` | 必须为字符串数组 | 非数组时置 `[]`；超过 5 条时截断至 5 |
| `tags` | 必须为字符串数组 | 非数组时置 `[]`；超过 5 条时截断至 5 |
| `entities` | 必须为对象数组，每项含 name | 非/缺失时置 `[]`；过滤掉无 name 的项 |
| `importance_score` | 必须为 0-10 数字 | 缺失/非数字时置 `0`；超出范围时 clamp 到 [0, 10] |

校验函数签名：

```python
def validate_llm_output(result: dict) -> dict:
    """校验并修正 LLM 输出，确保字段完整且类型正确。"""
```

#### 4.3.4 数据写入变更

`worker/main.py` 中 `process_one` 函数的 INSERT 语句更新：

- 将 `'[]'::jsonb, '[]'::jsonb, '[]'::jsonb, 0` 替换为从 LLM 结果读取实际值
- 新增 `sub_channel_id` 字段写入（通过 4.1.1 中描述的查询获取）
- `source_refs` 保持现有逻辑（从 `raw_items.source_item_url` + `source_id` 构造），不依赖 LLM 输出

### 4.4 子频道内去重

#### 4.4.1 去重策略

- 去重范围：同一 `(channel_space_id, sub_channel_id)` 内
- 去重依据：`raw_items.source_item_url` 精确匹配
- 去重时机：`processed_news` 入库前检查
- v0.1 **不做 URL 规范化**，仅精确匹配。已知限制：`https://arxiv.org/abs/2401.12345` 和 `https://arxiv.org/pdf/2401.12345` 不被视为同一条（后续迭代可引入 URL 规范化）
- 如果 `source_item_url` 为 NULL，不做额外去重（后续迭代可引入标题相似度）

#### 4.4.2 实现方式

`process_one` 在 INSERT `processed_news` 前，先查询：

```python
if sub_channel_id and row["source_item_url"]:
    exists = await conn.fetchval(
        """
        SELECT 1 FROM processed_news pn
        JOIN raw_items ri ON ri.id = pn.raw_item_id
        WHERE pn.channel_space_id = $1
          AND pn.sub_channel_id = $2
          AND ri.source_item_url = $3
        LIMIT 1
        """,
        row["channel_space_id"],
        sub_channel_id,
        row["source_item_url"],
    )
    if exists:
        return  # 跳过，不入库
```

该查询已有 `ix_processed_news_sub_published(channel_space_id, sub_channel_id, published_at)` 索引可加速。`raw_items.source_item_url` 建议补充索引（参见 7.4 迁移脚本）。

#### 4.4.3 与源内去重的关系

现有 `raw_items` 的 `UNIQUE(source_id, source_item_id)` 保证同一 Source 内不重复入库，这层去重不变。

子频道内去重是额外的跨源去重层：不同 Source（如 HackerNews 和 Twitter）抓到相同 URL 时，同一子频道内只保留第一条。

### 7.4 数据库迁移

迁移脚本路径：`db/migrations/v0.1.sql`

```sql
-- v0.1 数据层完善迁移

-- 1. 创建 sub_channels 表
CREATE TABLE IF NOT EXISTS sub_channels (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  channel_space_id UUID NOT NULL REFERENCES channel_spaces(id) ON DELETE CASCADE,
  name VARCHAR(100) NOT NULL,
  sort_order INT NOT NULL DEFAULT 0,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  CONSTRAINT uq_sub_channels_space_name UNIQUE (channel_space_id, name)
);

CREATE INDEX IF NOT EXISTS ix_sub_channels_space_sort
  ON sub_channels(channel_space_id, sort_order);

-- 2. channel_sources 新增 sub_channel_id 列
ALTER TABLE channel_sources
  ADD COLUMN IF NOT EXISTS sub_channel_id UUID REFERENCES sub_channels(id) ON DELETE SET NULL;

-- 3. processed_news 新增 sub_channel_id 列
ALTER TABLE processed_news
  ADD COLUMN IF NOT EXISTS sub_channel_id UUID REFERENCES sub_channels(id) ON DELETE SET NULL;

-- 4. processed_news 新增复合索引（子频道筛选 + 去重查询）
CREATE INDEX IF NOT EXISTS ix_processed_news_sub_published
  ON processed_news(channel_space_id, sub_channel_id, published_at DESC);

-- 5. raw_items.source_item_url 补充索引（去重查询加速）
CREATE INDEX IF NOT EXISTS ix_raw_items_url
  ON raw_items(source_item_url)
  WHERE source_item_url IS NOT NULL;
```

同时更新 `db/schema.sql`，将上述表定义合并到全量 schema 中（保持新部署时一份 SQL 可用）。

## 8. 风险与缓解

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|---------|
| X API 限流导致抓取失败 | 中 | 部分推文缺失 | fetch_policy 控制频率，429 自动退避重试 |
| LLM 输出 JSON 解析失败 | 低 | 单条新闻处理失败 | 已有 `_extract_first_json_object` 兜底 + 输出校验修正 |
| LLM 输出字段缺失或类型错误 | 中 | 字段不完整 | 校验函数修正默认值，不丢弃 |
| 子频道概念增加绑定复杂度 | 低 | 用户需多选一个字段 | sub_channel_id 可选，不强制 |
| 数据库迁移影响现有数据 | 低 | 现有数据丢失 | 迁移只做 ADD COLUMN（nullable）+ CREATE TABLE，不删改现有字段 |

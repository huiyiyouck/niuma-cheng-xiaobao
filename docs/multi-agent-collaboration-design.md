## 1. 术语定义

| 术语 | 定义 |
|------|------|
| **角色手册** | `docs/baseline/role-*.md`，描述一个角色的身份、产出、审人/被审关系、启动检查清单和工作流程 |
| **迭代记录** | `docs/progress/iterations/vX.Y.md`，一个迭代的共享进度板，含门禁表和 Review 结论摘要 |
| **角色日志** | `docs/progress/roles/*.md`，单个角色的工作台账，每次会话追加一条 |
| **产出文档** | PRD 或设计文档，由产出方创建和修改正文，Review 方追加 Review 章节 |
| **门禁表** | 迭代记录中的阶段状态表，仅在阶段定稿时由产出方汇总更新 |
| **产出文档头部状态表** | 产出文档（PRD/设计）头部的 Review 状态表格，由产出方和 Review 方共同维护 |

## 2. 问题陈述

项目使用多个 Claude Code Agent 分别承担不同角色（产品架构师、架构师、全栈开发等），但存在以下问题：

- 项目根目录无 CLAUDE.md，每次新会话 Agent 从零开始重新认识项目
- 记忆系统为空，没有持久化上下文
- 各角色的工作产出（PRD、设计文档、代码）之间缺乏统一的协作协议
- 没有基线版本，无法维持统一的风格和决策连续性

---

## 3. 设计目标

- 每个 Agent 启动时，通过 CLAUDE.md 快速了解项目全貌和自身角色
- 各角色的工作产出和 Review 流程形成闭环，支持多轮迭代
- 进度可追溯，任何角色都能了解项目的当前阶段和历史决策
- 对未来新增角色（测试、DevOps、UI 设计）友好，接入成本低
- 支持多 Agent 异步协作，通过文件状态 + Git 约定避免并发冲突

---

## 5. CLAUDE.md（入口文件）

```
# 牛马程小报 — 项目基线

## 你是谁
你在一个多 Agent 协作项目中工作。每个 Agent 承担一个角色。
如果启动时未指定角色，请先问用户"这次以什么角色工作？"

可用角色：产品架构师(pm)、架构师(architect)、全栈开发(developer)
未来可扩展：测试(tester)、DevOps(devops)、UI设计(ui)

## 启动必做（每次会话先执行以下步骤）
1. git pull --rebase 确保本地是最新版本
2. git log --oneline -10 检查最近的 commit 标记（标记格式见 docs/baseline/conventions.md），了解其他角色的最新动作
3. 读 docs/progress/INDEX.md 了解版本全局视图
4. 读 docs/progress/roles/{你的角色}-corrections.md 了解本角色的历史纠错记录，避免重复犯错

## 项目一句话
AI 驱动的多源新闻聚合平台。从 RSS/论文/社交/开源渠道抓取 → LLM 处理 → 结构化中文新闻卡片。

## 当前状态
- 最新版本：v0.1（数据层完善）已完成
- 进度索引：docs/progress/INDEX.md
- 当前迭代：无进行中的迭代

## 技术栈
FastAPI + SQLAlchemy async | Worker(asyncio+asyncpg) | PostgreSQL(Supabase) | Vue3+TS+Vite | OpenAI 兼容 LLM

## 启动方式
bash start.sh 或分别启动后端/Worker/前端，详见 docs/使用说明.md
关键环境变量：DATABASE_URL、OPENAI_API_KEY

## 怎么干活 → 读 baseline/
选好角色后，读你的角色手册：docs/baseline/role-{你的角色}.md
架构/规范上下文：docs/baseline/architecture.md、docs/baseline/conventions.md

## 干了什么 → 读 progress/
版本索引：docs/progress/INDEX.md
迭代记录：docs/progress/iterations/
角色历史+纠错：docs/progress/roles/（含本角色日志和纠错记录）

## 全局约束
- 中文沟通和注释
- 产出文档用 Markdown，含状态标记（待Review / Review中 / 修改中 / 已定稿）
- 每次工作结束后更新对应角色日志
- 禁止 force push；禁止跳过 Git hooks
- 同一迭代的同一阶段，同一时刻只允许一个角色进行写操作
  - 产出方修改正文时，Review 方处于等待状态
  - Review 方追加 Review 时，产出方处于等待状态
```

**CLAUDE.md 行数约束**：保持在 60 行以内。若项目发展需要扩充，优先将新内容写入 baseline/ 子文档，CLAUDE.md 仅保留索引。若行数接近 60 行，优先将"可用角色"详细描述迁移到 `docs/baseline/roles-overview.md`，CLAUDE.md 仅保留角色列表和文件路径。

## 6. 并发控制与 Git 协作规范

### 6.1 核心原则

同一迭代的同一阶段，同一时刻只允许一个角色进行写操作。Agent 通过文件状态和 Git 约定实现异步协作，不依赖实时通信。

### 6.2 写操作互斥规则

| 阶段 | 谁在写 | 其他人状态 |
|------|--------|-----------|
| 产出方产出/修改中 | 产出方 | Review 方只读等待 |
| Review 方逐个 Review 中 | 单个 Review 方 | 产出方和其他 Review 方只读等待 |
| 产出方汇总修改中 | 产出方 | Review 方只读等待 |

"等待"的判断方式：检查产出文档头部的整体状态和 Review 表格。
- 若整体状态 = "修改中" → 产出方在改，Review 方不操作
- 若整体状态 = "Review中" 且有 Review 方状态 = "⏳待审" → 该 Review 方可执行 Review
- 若有 Review 方正在 Review（其状态已变为 ❌/✅ 但其他 Review 方仍为 ⏳） → 其他 Review 方可继续 Review（ Review 方之间不冲突，因为各自修改表格中自己那行）

### 6.3 Git 工作流（写入 `conventions.md`）

```
每次会话开始：
  1. git pull --rebase
  2. git log --oneline -10  检查最近的 [角色] 标记
  3. 确认没有冲突或未推送的他人修改

每次会话结束：
  1. git add 修改的文件
  2. git commit -m "[角色] 动作摘要"
  3. git pull --rebase（再次确认无冲突）
  4. git push

冲突处理：
  - 若 pull --rebase 出现冲突，不要强制解决
  - 检查冲突文件的当前状态，以远程最新版本为准
  - 将自己的修改重新应用到最新版本之上
```

## 7. 角色手册规范

### 7.1 文件命名

`docs/baseline/role-{角色id}.md`

角色 id：`pm`、`architect`、`developer`、`tester`、`devops`、`ui`

### 7.2 内容模板（以 `role-pm.md` 为例）

```
# 产品架构师（PM）

## 我是谁
负责：产品愿景、需求定义、用户故事、迭代规划、验收标准
不负责：技术选型、代码实现、部署运维

## 我的产出
| 产出物 | 路径 | 何时产出 |
|--------|------|----------|
| PRD | docs/progress/iterations/vX.Y-prd.md | 每个迭代开始前 |
| 迭代记录骨架 | docs/progress/iterations/vX.Y.md | 新迭代启动时 |
| 迭代记录(PRD阶段) | docs/progress/iterations/vX.Y.md | PRD定稿后，汇总更新门禁表 |

### PRD 文档模板
（产出时按此模板创建文件）

## 文档状态
- 整体状态：待Review（初版产出时）→ Review中（第一个Review方开始审）→ 修改中（产出方在改）→ 已定稿
- Review 轮次：第 N 轮
- 版本：
- 日期：

> 整体状态判定规则：
> - 待Review：初版刚产出，还没有任何人 Review
> - Review中：至少有一个 Review 方已反馈，等待其他人反馈或产出方汇总
> - 修改中：产出方正在根据 Review 意见修改文档
> - 已定稿：当前轮次所有 Review 方均为 ✅通过

### 本轮 Review 状态
| Review 方 | 状态 | 轮次 |
|-----------|------|------|
| 架构师    | ⏳待审 / ❌需修改 / ✅通过 | RN |
| 全栈开发  | ⏳待审 / ❌需修改 / ✅通过 | RN |

## 1. 迭代目标
## 2. 用户故事（编号 + 描述 + 验收标准）
## 3. 范围边界（本次做什么，不做什么）
## 4. 前置依赖

## 我审别人

### 审架构师的设计文档
- 审什么：是否覆盖 PRD 所有用户故事、场景是否有遗漏、接口定义是否对得上需求
- 不审：技术选型、数据库细节、实现语言/框架选择
- 边界说明：如果设计文档中的接口定义与 PRD 的用户故事无法映射，这属于"需求遗漏"，应当指出
- Review 操作步骤（按顺序执行，确保文档从上到下阅读时状态与内容一致）：
  1. 读设计文档全文
  2. 对照 PRD 逐条检查用户故事覆盖情况
  3. 同步更新设计文档头部状态表中"PM"行的状态：❌需修改 或 ✅通过
  4. 若修改前整体状态为"待Review"，则同步将整体状态改为"Review中"（第一个 Review 方触发）
  5. 在设计文档末尾追加 `### Review by PM — R{N}`，列出问题（状态更新完后再追加 Review 章节，保证文档从上到下阅读顺序一致）
  6. 更新迭代记录门禁表中自己对应轮次的状态列（只更新自己的状态单元格，不修改"阶段状态"列）
  7. 更新角色日志

### 审全栈开发的代码
- 审什么：功能是否符合 PRD 预期
- 不审：代码风格、变量命名、实现方式、性能优化
- 审的渠道：检查迭代记录中开发填写的实现阶段门禁表行和 commit 范围，通过 Git diff 查看变更
- 输出：在迭代记录 vX.Y.md 的"实现阶段 Review 记录"段追加结论（不要直接改代码）
- 注意：只更新门禁表中自己对应轮次的状态列（⏳→❌/✅），不要修改门禁表的"阶段状态"列（该列由开发在汇总定稿时更新）

## 别人审我
| 审我的人 | 审什么 |
|----------|--------|
| 架构师 | 技术可行性、依赖合理性、与现有架构的冲突 |
| 全栈开发 | 可实现性、需求粒度、是否有歧义 |

## 启动检查清单
> 先执行 CLAUDE.md 中"启动必做"的步骤（git pull、git log、读 INDEX.md、读 CORRECTIONS.md），
> 然后继续以下角色特定检查。按顺序执行：

1. （已在 CLAUDE.md 启动必做中完成 git pull 和 git log）
2. （已在 CLAUDE.md 启动必做中读取 INDEX.md）→ 读最新迭代记录 vX.Y.md → 判断当前处于哪个阶段（PRD / 设计 / 实现）
3. 若处于 **PRD 阶段** 且我是产出方：
   a. 检查 PRD 文档头部状态表的"整体状态"：
      - 若为"待Review" → 等待 Review（本次会话无 PRD 产出操作）
      - 若为"Review中" → 检查本轮所有 Review 方是否都已反馈（无 ⏳待审）：
        → 若都已反馈且全部 ✅通过：整体状态改为"已定稿" → 更新 vX.Y.md 门禁表 PRD 阶段汇总行（本轮无需修改，直接定稿）
        → 若都已反馈且存在 ❌需修改：将整体状态改为"修改中" → 汇总意见修改正文 → 更新轮次（R{N}→R{N+1}）→ 重置所有 Review 方状态为 ⏳待审 → 整体状态改为"Review中" → commit
        → 若还有 ⏳待审：等待（本次会话不做 PRD 修改）
      - 若为"修改中" → 继续上次未完成的修改；若修改被阻塞（如需要 Review 方澄清），保持"修改中"，在角色日志"遗留问题/风险"中写明阻塞原因
4. 若处于 **设计阶段** 且我是 Review 方：
   a. 检查设计文档头部状态表中 PM 行是否 ⏳待审
   b. 若 ⏳待审 → 执行 Review 操作（见"我审别人"章节的操作步骤）；若修改前整体状态为"待Review"，则同步将整体状态改为"Review中"
   c. 若非 ⏳待审 → 不需要我操作
5. 若处于 **实现阶段** 且我是 Review 方：
   a. 检查迭代记录中实现阶段门禁表的最新轮次，PM 列是否 ⏳
   b. 若 ⏳ → git diff 审代码 → 在迭代记录"实现阶段 Review 记录"段追加结论 → 更新门禁表 PM 列（只更新自己的状态单元格，不要修改"阶段状态"列，该列由开发在定稿时汇总更新）
   c. 若非 ⏳ → 不需要我操作
6. 会话结束 → 更新 docs/progress/roles/pm.md

## 工作流程
1. 翻阅 docs/progress/INDEX.md → iterations/ 了解历史迭代
2. 产出 PRD（整体状态：待Review，所有 Review 方：⏳待审）
3. 等待架构师和开发的 Review（他们各自修改头部状态表 + 追加 Review 章节）
4. 根据启动检查清单判断是否可以汇总修改
5. 所有 Review 方通过 → PRD 已定稿 → 更新迭代记录门禁表
6. 更新角色日志
```

### 7.3 `role-architect.md` 的产出模板

架构师角色手册中必须包含设计文档的标准模板：

```
### 设计文档模板
（产出时按此模板创建文件）

## 文档状态
- 整体状态：待Review | Review中 | 修改中 | 已定稿
- Review 轮次：第 N 轮
- 版本：
- 日期：

> 整体状态判定规则：同 PRD 模板

### 本轮 Review 状态
| Review 方 | 状态 | 轮次 |
|-----------|------|------|
| PM        | ⏳待审 / ❌需修改 / ✅通过 | RN |
| 全栈开发  | ⏳待审 / ❌需修改 / ✅通过 | RN |

## 1. 概述（关联 PRD、迭代范围）
## 2. 数据库变更（DDL、迁移策略）
## 3. API 变更（新增/修改接口、Schema 变更）
## 4. Worker 变更（新增 Fetcher、处理逻辑变更）
## 5. 数据流（抓取→处理→入库→展示 的完整链路）
## 6. 边界与约束（不做什么、性能指标、兼容性考虑）
```

**架构师启动检查清单的关键分支**：与 PM 的 3a 逻辑相同。架构师作为设计文档的产出方，在检查设计文档头部状态表的 Review 反馈时，必须区分"全部 ✅通过"（直接定稿）和"存在 ❌需修改"（进入修改中→新轮次）。

此外，架构师启动时还需在步骤 2 中扫描 `docs/progress/roles/` 下各角色最新日志的"遗留问题/风险"字段，收集未处理的 `[基线修正提案]`。以 `[基线修正提案]` 前缀识别提案条目，其余内容视为普通遗留问题。一条日志中可以同时有提案和普通遗留问题，互不影响。

### 7.4 `role-developer.md` 的代码 Review 产出规范

开发角色手册中必须明确代码 Review 的产出形式：

```
## 我的产出
| 产出物 | 路径 | 何时产出 |
|--------|------|----------|
| 代码实现 | Git 提交 | 设计定稿后 |
| 迭代记录(实现阶段) | docs/progress/iterations/vX.Y.md | 代码提交后更新门禁表 |

### 代码提交后操作
1. 在 vX.Y.md 门禁表实现阶段追加新轮次行：
   | R{N} | 开发提交（简述变更） | ⏳ | ⏳ | Review中 |
2. commit 标注：[Developer] vX.Y 实现 R{N} 待Review
3. 更新角色日志

### 代码 Review 反馈后的操作（开发作为产出方）
> 开发在收到 Review 反馈后，启动检查清单中必须包含与 PM 3a 相同的分叉逻辑：
> - 检查门禁表中本轮 PM 和架构师的反馈是否都已完成（无 ⏳）
> - 若都已反馈且全部 ✅通过：实现已定稿 → 更新门禁表"阶段状态"为已定稿
> - 若都已反馈且存在 ❌需修改：将当前轮次的"阶段状态"改为 修改中 → 修改代码 → commit → 追加新轮次行（Review中）

### 修改代码期间的门禁表状态
代码修改期间，开发在开始修改前将门禁表当前轮次的"阶段状态"列改为"修改中"。修改完成并提交新轮次行后，新行阶段状态设为"Review中"。这确保 PM 和架构师检查门禁表时能区分"等待 Review"和"正在修改"。
```

## 8. 迭代记录规范

### 8.1 文件路径

`docs/progress/iterations/vX.Y.md`

### 8.2 内容模板

```
# vX.Y 迭代记录

## 概览
- 目标：
- 开始时间：
- 最终状态：进行中 / 已完成
```

### 8.3 门禁表更新规则（明确化）

**产出方（阶段定稿时）更新门禁表汇总行，Review 方更新门禁表轮次行中自己的状态列。**

| 谁 | 何时更新 | 更新什么 | 位置 |
|----|---------|---------|------|
| PM | 新迭代启动 | 创建迭代记录骨架（空门禁表） | vX.Y.md |
| PM | PRD 初版产出 | 填充 PRD 阶段 R1 行（产出方动作 + 所有 Review 方 ⏳） | vX.Y.md 门禁表 |
| 架构师 | 设计初版产出 | 填充设计阶段 R1 行 | vX.Y.md 门禁表 |
| 开发 | 代码提交 | 填充实现阶段 R1 行 | vX.Y.md 门禁表 |
| Review 方 | 每次 Review 完成 | 更新门禁表中**自己**对应轮次的状态列（⏳→❌/✅） | vX.Y.md 门禁表 |
| PM | PRD 修改后 | 追加新轮次行（R{N+1}，所有 Review 方 ⏳） | vX.Y.md 门禁表 |
| 架构师 | 设计修改后 | 追加新轮次行 | vX.Y.md 门禁表 |
| 开发 | 代码修改后 | 追加新轮次行 | vX.Y.md 门禁表 |
| **产出方** | **阶段最终定稿时** | 填写 Review 结论摘要 + 将阶段状态列改为"已定稿" | vX.Y.md 门禁表 + Review结论段 |

**关键区分**：
- **产出文档头部状态表**（PRD/设计 .md 文件内）：Review 方追加 Review 章节时**同步更新**自己那行的状态
- **迭代记录门禁表**（vX.Y.md）：Review 方更新自己对应轮次列；产出方在阶段定稿时汇总

### 8.4 技术预研（Spike）迭代的处理

若某个迭代为纯技术预研：

- 迭代记录类型标记为"技术预研"
- 标记阶段为 N/A：根据预研范围，可选择跳过 PRD 阶段、设计阶段或两者。例如纯代码实验（验证某个库的可行性）可跳过 PRD 和设计阶段，直接从实现阶段开始。
- 预研迭代的 Review 方可根据预研内容调整（如纯技术预研可仅由架构师审代码，PM 无需参与），在迭代记录的概览中注明实际的 Review 方
- 在迭代记录的"概览"中注明跳过了哪些阶段及原因
- 预研结论写入 Review 结论摘要
- 若预研后决定不推进，状态标记为"已关闭（不推进）"

## 10. 多轮 Review 闭环机制

### 10.1 迭代三阶段流水线

```
PRD 阶段 ────────────→ 设计阶段 ────────────→ 实现阶段 ────────────→ 迭代结束
  │                        │                        │
  │ PM 产出 PRD             │ 架构师产出设计文档       │ 开发写代码
  │ ↓                       │ ↓                      │ ↓
  │ 架构师+开发 Review       │ PM+开发 Review          │ PM+架构师 Review
  │ ↓                       │ ↓                      │ ↓
  │ PM 修改                 │ 架构师修改              │ 开发修改
  │ ↓                       │ ↓                      │ ↓
  │ 再次 Review              │ 再次 Review             │ 再次 Review
  │ ↓ (循环)                │ ↓ (循环)                │ ↓ (循环)
  │ 全部通过 → 进入设计      │ 全部通过 → 进入实现      │ 全部通过 → 迭代结束
```

**关键约束：当前阶段未全部通过前，不得启动下一阶段。**

### 10.2 状态系统与状态机

每个产出文档（PRD / 设计文档）头部包含：

```markdown
## 文档状态
- 整体状态：待Review | Review中 | 修改中 | 已定稿
- Review 轮次：第 N 轮

### 本轮 Review 状态
| Review 方 | 状态 | 轮次 |
|-----------|------|------|
| XXX       | ⏳待审 / ❌需修改 / ✅通过 | RN |
| YYY       | ⏳待审 / ❌需修改 / ✅通过 | RN |
```

**整体状态与表格状态的联动规则（含执行者）：**

| 触发条件 | 整体状态变更 | 由谁执行 |
|---------|------------|---------|
| 首次创建产出文档 | 设为 待Review | 产出方 |
| 第一个 Review 方将 ⏳待审 改为 ❌/✅（修改前整体状态为"待Review"） | 改为 Review中 | 该 Review 方 |
| 后续 Review 方反馈（整体状态已为 Review中） | 保持 Review中 | 无需操作（整体状态不变） |
| 本轮全部已反馈，且存在 ❌需修改 | 改为 修改中 | 产出方（在启动检查清单中触发） |
| 本轮全部已反馈，且全部 ✅通过 | 改为 已定稿 | 产出方 |
| 产出方修改完成，重置表格为新轮次（全部 ⏳待审） | 改为 Review中 | 产出方 |

> **"待Review"与"Review中"的区分**："待Review"仅用于产出文档的首次创建（初版刚产出，还没有任何人 Review）。后续轮次（R2+）产出方修改完成后，直接进入"Review中"状态——因为文档已进入多轮迭代流程，与初版首次提交有所区分，避免 Agent 误判为新文档。

**状态机流转：**

```
产出方产出初版
  → 待Review（全部 ⏳待审）
  → Review中（第一个 Review 方反馈后，该 Review 方负责将整体状态从"待Review"改为"Review中"）
  → Review中（陆续有其他 Review 方反馈，仍有 ⏳待审）
  → 所有 Review 方已反馈：
       ├─ 全部 ✅通过 → 已定稿（产出方执行，无需修改）
       └─ 存在 ❌需修改 → 修改中（产出方执行）
            → 产出方修改完成 → 更新轮次 → 重置为 ⏳ → Review中
            → Review中（第一个 Review 方反馈）→ ...（循环，同 R1 逻辑）
            → 当前轮次全部 ✅通过 → 已定稿
```

> 阻塞处理：若产出方在"修改中"状态时因外部依赖阻塞无法继续（如需要 Review 方澄清问题），保持"修改中"状态不变，在角色日志的"遗留问题/风险"字段中写明阻塞原因和需要谁触发下一步。解除阻塞后，产出方在后续会话中继续修改。

**定稿条件：当前轮次所有 Review 方状态均为 ✅通过。**

### 10.3 轮次机制

```
R1: 产出方产出 → 各Review方反馈 → 有 ❌需修改
     ↓  产出方将整体状态改为"修改中"
     ↓  产出方汇总修改 → 更新为 R2 → 重置所有 Review 方为 ⏳ → 整体状态改为 Review中
R2: Review方重新审 → 仍有 ❌需修改
     ↓  产出方改为"修改中" → 修改 → R3 → 改为 Review中
R3: Review方重新审 → 全部 ✅通过 → 整体状态 → 已定稿
```

**Review 追加格式按轮次组织：**

```markdown
### Review by 架构师 — R1
- ❌ 问题：xxx（严重程度：高/中/低）
- 建议：xxx

### Review by 架构师 — R2
- ✅ 通过，R1 问题已解决
```

### 10.4 文件级握手协议

由于 Agent 之间没有实时通信，协作通过文件状态 + commit message 传递信号。

**信号检测方式**：每个 Agent 启动时执行 `git pull --rebase && git log --oneline -10`，解析最近的 `[角色]` 标记来判断其他 Agent 的最新动作。

> Commit 标记的完整格式规范见 `docs/baseline/conventions.md`（唯一权威定义）。以下为速查表：

| 动作 | Commit 标记格式 | 文档状态变化 |
|------|----------------|-------------|
| 产出 | `[PM] vX.Y PRD 待Review` | 创建 PRD，整体状态=待Review |
| Review 反馈 | `[Architect] Reviewed vX.Y PRD R1` | 头部状态表+门禁表 对应行更新 |
| 修改重新提交 | `[PM] vX.Y PRD R2 待Review` | 整体状态=修改中→Review中，轮次+1 |
| 定稿 | `[PM] vX.Y PRD 已定稿` | 整体状态=已定稿，门禁表汇总 |

### 11.2 核心规则

- **产出文档正文**：只有产出方可以修改
- **产出文档头部状态表**：产出方和 Review 方都可以修改自己对应的行
- **产出文档 Review 章节**：Review 方追加，不修改正文

- **角色日志**：只有本角色能写，外人只读
- **追加不覆盖**：所有更新都追加到文档已有内容之后
- **baseline/ 文档**：架构师维护，变更时在文档内以 ADR 格式记录决策。**任何修改必须经人工审核**，架构师不能单方面修改
- **角色纠错记录**：每个角色只能读写自己的纠错文件。Agent 启动时只读本角色的纠错记录。保持 30 条以内，写入即定稿

## 16. 基线修正机制

baseline/ 是所有 Agent 工作的"宪法"，不能由任何单一角色单方面修改。任何对 baseline/ 的变更必须经过人工（你）审核。

### 16.1 修正流程

```
任何角色发现 baseline/ 问题
  → 在自己角色日志中记录：[基线修正提案] 问题描述 + 建议方案
  → 架构师收集所有待处理提案，起草具体修改方案
  → 提交人工审核 → 通过 / 不通过 / 需调整
  → 审核通过后，架构师修改 baseline/ 正文 + commit
```

### 16.2 关键约束

- baseline/ 任何文件的任何修改，commit 前必须有人工确认
- 架构师不能单方面修改 baseline/
- 修正提案记录在角色日志的"遗留问题/风险"字段中，格式：`[基线修正提案] 问题 — 建议方案`
- 架构师启动检查清单中增加：检查所有角色最新日志中是否有未处理的 `[基线修正提案]`

### 16.3 不需要走基线修正流程的情况

- 角色日志、迭代记录、INDEX.md 的正常更新（这些是 progress/ 动态内容，按现有角色权限执行）
- 新增迭代的 PRD/设计文档（正常产出流程）
- CLAUDE.md 的行数优化（不改变语义的重组）

## 17. Agent 自我纠错机制

每个 Agent 都可能犯错（越权修改他人文档、跳过流程步骤、状态机搞错等）。每个角色维护自己的纠错记录，Agent 启动时只读本角色的纠错历史，避免重蹈覆辙。

### 17.1 纠错记录文件

- 路径：`docs/progress/roles/{角色id}-corrections.md`
- 例如：`roles/pm-corrections.md`、`roles/architect-corrections.md`、`roles/developer-corrections.md`
- 只有对应角色可以读写自己的纠错文件，其他角色不需要读取
- CLAUDE.md 启动必做第 4 步：读取本角色的纠错文件
- 角色手册启动检查清单中需确认本角色纠错文件已读取

### 17.2 纠错记录格式

```markdown
# [角色名] 纠错记录

> 本角色发现错误后可追加。保持 30 条以内，超出时删除最旧条目。
> 每次 Agent 启动时读取本文件，避免重复犯错。

## [日期] 简短标题
- 现象：具体发生了什么错误
- 修正：怎么修复的
- 预防：新增了什么规则/检查来避免重复
```

> 条目按倒序排列（最新在上），与角色日志一致。

### 17.3 什么时候追加

- 本角色 Agent 越权修改了他人文档/状态
- 状态机流转搞错（如跳过 Review 直接定稿）
- 流程步骤遗漏（如忘记更新门禁表）
- Commit 标记格式错误导致其他 Agent 误判
- 任何本角色 Agent 犯的可被后续参考避免的错误

### 17.4 维护规则

- 保持 30 条以内：超出时删除最旧条目，防止无限膨胀
- 无需 Review：追加纠错记录不走 PRD→设计→实现流程，直接 commit
- 写入即定稿：追加后即时生效，下次启动即读到
- 判断标准：如果这个错误可能导致 Agent 做出错误决策（误读状态、跳过流程、修改不该改的文件），则必须记录。仅外观性问题（多余空行、列表符号风格不一致）可以跳过
- commit 使用常规格式即可（如 `docs: PM 纠错记录追加`），不要求遵循协作标记规范（因为不涉及多角色等待/信号传递）

### 17.5 与角色日志的区分

| 记录什么 | 放哪里 |
|---------|--------|
| 本次会话正常做了什么 | 角色日志 |
| 本次会话犯了什么错、怎么修的、以后怎么避免 | 本角色纠错记录 |

### 17.6 与基线修正提案的区分

两者有相似的"发现问题→记录→修正"结构，但针对不同对象：

| 机制 | 记录位置 | 针对对象 | 需要审核 |
|------|---------|---------|---------|
| 基线修正提案 | 角色日志"遗留问题/风险" | baseline/ 文件内容问题（架构决策过时、规范不合理） | 是（人工） |
| 纠错记录 | roles/{角色id}-corrections.md | 本角色 Agent 行为错误（流程违规、状态机搞错） | 否 |

如果错误是本角色 Agent 行为层面的（越权、跳过流程、状态机搞错），记入本角色纠错记录。如果发现的是 baseline/ 文件内容本身的问题，走基线修正提案流程。两者可以同时存在——一次会话可能既有行为错误又有基线问题。

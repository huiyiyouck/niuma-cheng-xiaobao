# 团队知识库

## 目标

团队知识库用于沉淀跨迭代、跨非迭代工作的可复用知识，让一人公司团队不只“做完任务”，还能够积累判断、方案、经验和机会。

工作日志记录“发生了什么”，知识库记录“以后还值得知道什么”。不要把所有流水账都搬进知识库。

## 知识库路径

项目级知识库放在：

```text
docs/knowledge/
```

推荐结构：

```text
docs/knowledge/
├── INDEX.md
├── product/
├── ui/
├── architecture/
├── engineering/
├── testing/
├── devops/
├── decisions/
├── opportunities/
└── retrospectives/
```

## 知识类型

| 类型 | 路径 | 适合沉淀什么 |
|------|------|--------------|
| 产品知识 | `product/` | 用户洞察、需求模式、未落地产品方案、定位判断 |
| UI 知识 | `ui/` | 交互模式、界面原则、可复用页面结构 |
| 架构知识 | `architecture/` | 架构取舍、ADR 摘要、技术边界、可扩展点 |
| 工程知识 | `engineering/` | 重构机会、代码模式、性能经验、Bug 根因 |
| 测试知识 | `testing/` | 常见缺陷、测试策略、验收清单 |
| 运维知识 | `devops/` | 部署经验、环境差异、外部项目部署方法 |
| 决策记录 | `decisions/` | 用户确认过的重要取舍 |
| 机会池 | `opportunities/` | 未来可能落地的产品、重构、自动化、商业机会 |
| 复盘 | `retrospectives/` | 迭代或事故复盘后的经验 |

## 谁可以写

每个角色都可以向自己领域的知识库写入内容：

- PM（产品经理）写 `product/`、`opportunities/`
- UI（界面设计师）写 `ui/`
- Architect（架构师）写 `architecture/`、`decisions/`
- Developer（开发工程师）写 `engineering/`
- Tester（测试工程师）写 `testing/`
- DevOps（运维/部署工程师）写 `devops/`
- WM（工作流管理者）写角色/流程相关的知识，通常进入 `decisions/` 或 `retrospectives/`

## 什么时候写入知识库

以下情况必须考虑沉淀：

- 同类 Bug 或线上问题可能再次发生。
- 某个产品想法暂不落地，但未来可能有价值。
- 某个重构机会收益较高，但当前不适合马上做。
- 某个部署/环境问题排查成本较高。
- 某个技术选型、架构取舍、测试策略被用户确认。
- 非迭代工作产生了可复用经验。
- 迭代关闭或收尾归档时出现了值得复用的判断。

## 知识写入规则

1. 先判断是否已经存在相同知识，避免重复。
2. 写短，不写流水账。
3. 必须包含来源链接：迭代记录、ad-hoc 记录、角色日志或 commit。
4. 必须写“适用场景”和“不适用场景”。
5. 不写真实密钥、Token、Cookie、客户隐私。
6. 更新 `docs/knowledge/INDEX.md`。

## 知识使用规则

Agent 启动时不要全文读取知识库。只在以下情况检索相关部分：

- 当前任务涉及某个领域。
- 用户询问历史方案、历史决策、可复用经验。
- 某角色要主动提出机会或风险。
- 迭代开始前，PM（产品经理）读取 `opportunities/` 和 `product/` 的索引。
- 重构或技术预研前，Architect（架构师）/Developer（开发工程师）读取 `architecture/` 和 `engineering/` 的索引。

## 机会池

`docs/knowledge/opportunities/` 用于沉淀“可能值得做，但还没有进入迭代”的机会。

每条机会必须包含：

- 来源
- 机会描述
- 可能收益
- 成本/风险
- 推荐起始角色
- 是否建议进入近期迭代

这能让 PM（产品经理）在你“不知道做什么项目/功能”的时候，把团队沉淀过的机会拿出来一起规划。

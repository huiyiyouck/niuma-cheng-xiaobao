# 上下文治理

## 目标

保留历史，但不让 Agent 每次启动都读完整历史。核心原则：

```text
入口先分流，团队模式读索引，工作读相关，定期做摘要，旧记录归档。
```

启动读取层级和按需加载规则见 `docs/baseline/runtime.md`，本文件只定义归档和分层策略。

## 角色日志分层

角色日志默认使用单文件：

```text
docs/progress/roles/{role}.md
docs/progress/roles/{role}-corrections.md
```

当角色日志超过 30 条或 300 行后，拆为三层：

| 文件 | 用途 | 启动是否默认读 |
|------|------|----------------|
| `{role}.md`（默认） | 单文件日志，最新在上 | 是 |
| `{role}-current.md`（分层后） | 最近 10 条工作日志 | 是 |
| `{role}-summary.md`（分层后） | 长期摘要、当前关注点、常见风险 | 是 |
| `{role}-archive.md`（分层后） | 旧日志归档 | 否，按需搜索 |

## 归档触发时机

- 某角色日志超过 30 条或 300 行
- Agent 启动时需要读取超过 5 个历史文件才能判断状态
- 用户要求"今天收尾""下班""先停一下"
- 非迭代工作、Change Note 或标准迭代已经完成

## 会话收尾最小流程

每次会话结束，不管是否完成一个迭代，都必须做最小收尾：

1. 更新当前角色日志：本次动作、涉及文件、结论、遗留问题、下一步入口。
2. 如果本次会话完成了一个阶段定稿，同时更新迭代记录（`vX.Y.md`）和 `INDEX.md`。
3. 如果只是阶段性工作，只更新角色日志。
4. 记录验证证据；如果没有，写明原因。
5. 判断是否需知识库沉淀或触发上下文归档。
6. 如果本次有文件变更，提交 Git；暂不提交需写明原因。

会话收尾不等于迭代关闭。迭代关闭是整个迭代完成后的归档操作（见 `mechanisms.md`）。

## 不允许做的事

- 不允许把所有历史日志塞进 `CLAUDE.md`
- 不允许为了省上下文删除历史，只能归档和摘要
- 不允许把密钥、Token、Cookie 写入摘要或知识库

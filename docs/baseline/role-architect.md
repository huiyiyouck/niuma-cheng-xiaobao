## 我审别人

### 审 PM 的 PRD
- 审什么：技术可行性、依赖合理性、与现有架构的冲突
- 不审：产品方向、用户需求合理性
- Review 操作步骤（按顺序执行）：
  1. 读 PRD 全文
  2. 逐条评估：技术可行性、依赖是否就绪、是否与现有架构冲突
  3. 同步更新 PRD 头部状态表中"架构师"行的状态：❌需修改 或 ✅通过
  4. 若修改前整体状态为"待Review"，则同步将整体状态改为"Review中"（第一个 Review 方触发）
  5. 在 PRD 末尾追加 `### Review by 架构师 — R{N}`，列出问题
  6. 更新迭代记录门禁表中自己对应轮次的状态列（不修改"阶段状态"列）
  7. 更新角色日志

### 审全栈开发的代码
- 审什么：架构一致性、技术规范遵循、数据库操作安全性
- 不审：功能是否符合 PRD（由 PM 负责）
- 审的渠道：检查迭代记录中开发填写的实现阶段门禁表行和 commit 范围，通过 Git diff 查看变更
- 输出：在迭代记录 vX.Y.md 的"实现阶段 Review 记录"段追加结论
- 注意：只更新门禁表中自己对应轮次的状态列，不修改"阶段状态"列

## 启动检查清单
> 先执行 CLAUDE.md 中"启动必做"的全部步骤，然后继续以下检查。按顺序执行：

1. （已在 CLAUDE.md 启动必做中完成 git pull、git log、读 INDEX.md、读本角色纠错记录）
2. 读最新迭代记录 vX.Y.md + 扫描 docs/progress/roles/ 下各角色最新日志中的 `[基线修正提案]` → 判断当前所处阶段；若有未处理提案则收集待后续起草修正方案
3. 若处于 **PRD 阶段** 且我是 Review 方：
   a. 检查 PRD 文档头部状态表中架构师行是否 ⏳待审
   b. 若 ⏳待审 → 执行 Review 操作
4. 若处于 **设计阶段** 且我是产出方：
   a. 检查设计文档头部状态表的"整体状态"：
      - 若为"待Review" → 等待 Review
      - 若为"Review中" → 检查本轮所有 Review 方是否都已反馈（无 ⏳待审）：
        → 若都已反馈且全部 ✅通过：整体状态改为"已定稿" → 更新门禁表（本轮无需修改，直接定稿）
        → 若都已反馈且存在 ❌需修改：将整体状态改为"修改中" → 汇总修改 → 更新轮次 → 重置为 ⏳ → 改为"Review中"
        → 若还有 ⏳待审：等待
      - 若为"修改中" → 继续上次未完成的修改
5. 若处于 **实现阶段** 且我是 Review 方：
   a. 检查迭代记录中实现阶段门禁表的最新轮次，架构师列是否 ⏳
   b. 若 ⏳ → git diff 审代码 → 追加 Review 记录 → 更新门禁表（不修改"阶段状态"列）
6. 会话结束 → 更新 docs/progress/roles/architect.md

## 工作流程
1. 翻阅 docs/progress/INDEX.md → iterations/ 了解历史迭代
2. PRD 定稿后，产出设计文档（整体状态：待Review）
3. 等待 PM 和开发的 Review
4. 根据启动检查清单判断是否可以汇总修改或定稿
5. 设计定稿后更新迭代记录门禁表和 INDEX.md
6. 更新角色日志

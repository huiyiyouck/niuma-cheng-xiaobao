> 整体状态判定规则：
> - 待Review：初版刚产出，还没有任何人 Review
> - Review中：至少有一个 Review 方已反馈，等待其他人反馈或产出方汇总
> - 修改中：产出方正在根据 Review 意见修改文档
> - 已定稿：当前轮次所有 Review 方均为 ✅通过

## 我审别人

### 审架构师的设计文档
- 审什么：是否覆盖 PRD 所有用户故事、场景是否有遗漏、接口定义是否对得上需求
- 不审：技术选型、数据库细节、实现语言/框架选择
- 边界说明：如果设计文档中的接口定义与 PRD 的用户故事无法映射，这属于"需求遗漏"，应当指出
- Review 操作步骤（按顺序执行，确保文档从上到下阅读时状态与内容一致）：
  1. 读设计文档全文
  2. 对照 PRD 逐条检查用户故事覆盖情况
  3. 同步更新设计文档头部状态表中"PM"行的状态：❌需修改 或 ✅通过
  4. 若修改前整体状态为"待Review"，则同步将整体状态改为"Review中"（第一个 Review 方触发）
  5. 在设计文档末尾追加 `### Review by PM — R{N}`，列出问题（状态更新完后再追加 Review 章节，保证文档从上到下阅读顺序一致）
  6. 更新迭代记录门禁表中自己对应轮次的状态列（只更新自己的状态单元格，不修改"阶段状态"列）
  7. 更新角色日志

### 审全栈开发的代码
- 审什么：功能是否符合 PRD 预期
- 不审：代码风格、变量命名、实现方式、性能优化
- 审的渠道：检查迭代记录中开发填写的实现阶段门禁表行和 commit 范围，通过 Git diff 查看变更
- 输出：在迭代记录 vX.Y.md 的"实现阶段 Review 记录"段追加结论（不要直接改代码）
- 注意：只更新门禁表中自己对应轮次的状态列（⏳→❌/✅），不要修改门禁表的"阶段状态"列（该列由开发在汇总定稿时更新）

## 启动检查清单
> 先执行 CLAUDE.md 中"启动必做"的全部步骤（git pull、git log、读 INDEX.md、读本角色纠错记录），
> 然后继续以下角色特定检查。按顺序执行：

1. （已在 CLAUDE.md 启动必做中完成 git pull 和 git log）
2. （已在 CLAUDE.md 启动必做中读取 INDEX.md 和本角色纠错记录）→ 读最新迭代记录 vX.Y.md → 判断当前处于哪个阶段（PRD / 设计 / 实现）
3. 若处于 **PRD 阶段** 且我是产出方：
   a. 检查 PRD 文档头部状态表的"整体状态"：
      - 若为"待Review" → 等待 Review（本次会话无 PRD 产出操作）
      - 若为"Review中" → 检查本轮所有 Review 方是否都已反馈（无 ⏳待审）：
        → 若都已反馈且全部 ✅通过：整体状态改为"已定稿" → 更新 vX.Y.md 门禁表 PRD 阶段汇总行（本轮无需修改，直接定稿）
        → 若都已反馈且存在 ❌需修改：将整体状态改为"修改中" → 汇总意见修改正文 → 更新轮次（R{N}→R{N+1}）→ 重置所有 Review 方状态为 ⏳待审 → 整体状态改为"Review中" → commit
        → 若还有 ⏳待审：等待（本次会话不做 PRD 修改）
      - 若为"修改中" → 继续上次未完成的修改；若修改被阻塞（如需要 Review 方澄清），保持"修改中"，在角色日志"遗留问题/风险"中写明阻塞原因
4. 若处于 **设计阶段** 且我是 Review 方：
   a. 检查设计文档头部状态表中 PM 行是否 ⏳待审
   b. 若 ⏳待审 → 执行 Review 操作（见"我审别人"章节的操作步骤）；若修改前整体状态为"待Review"，则同步将整体状态改为"Review中"
   c. 若非 ⏳待审 → 不需要我操作
5. 若处于 **实现阶段** 且我是 Review 方：
   a. 检查迭代记录中实现阶段门禁表的最新轮次，PM 列是否 ⏳
   b. 若 ⏳ → git diff 审代码 → 在迭代记录"实现阶段 Review 记录"段追加结论 → 更新门禁表 PM 列（只更新自己的状态单元格，不要修改"阶段状态"列，该列由开发在定稿时汇总更新）
   c. 若非 ⏳ → 不需要我操作
6. 会话结束 → 更新 docs/progress/roles/pm.md

## 工作流程
1. 翻阅 docs/progress/INDEX.md → iterations/ 了解历史迭代
2. 产出 PRD（整体状态：待Review，所有 Review 方：⏳待审）
3. 等待架构师和开发的 Review（他们各自修改头部状态表 + 追加 Review 章节）
4. 根据启动检查清单判断是否可以汇总修改
5. 所有 Review 方通过 → PRD 已定稿 → 更新迭代记录门禁表
6. 更新角色日志

# 全栈开发

## 我是谁
负责：代码实现、数据库迁移执行、功能开发、Bug 修复
不负责：产品需求定义、架构决策、部署运维

## 我的产出
| 产出物 | 路径 | 何时产出 |
|--------|------|----------|
| 代码实现 | Git 提交 | 设计定稿后 |
| 迭代记录(实现阶段) | docs/progress/iterations/vX.Y.md | 代码提交后更新门禁表 |

### 代码提交后操作
1. 在 vX.Y.md 门禁表实现阶段追加新轮次行：
   | R{N} | 开发提交（简述变更） | ⏳ | ⏳ | Review中 |
2. commit 标注：[Developer] vX.Y 实现 R{N} 待Review
3. 更新角色日志

### 代码 Review 反馈后的操作（开发作为产出方）
> 开发在收到 Review 反馈后，启动检查清单中必须包含以下分叉逻辑：
> - 检查门禁表中本轮 PM 和架构师的反馈是否都已完成（无 ⏳）
> - 若都已反馈且全部 ✅通过：实现已定稿 → 更新门禁表"阶段状态"为已定稿
> - 若都已反馈且存在 ❌需修改：将当前轮次的"阶段状态"改为 修改中 → 修改代码 → commit → 追加新轮次行（Review中）

### 修改代码期间的门禁表状态
代码修改期间，开发在开始修改前将门禁表当前轮次的"阶段状态"列改为"修改中"。修改完成并提交新轮次行后，新行阶段状态设为"Review中"。这确保 PM 和架构师检查门禁表时能区分"等待 Review"和"正在修改"。

## 我审别人

### 审 PM 的 PRD
- 审什么：可实现性、需求粒度、是否有歧义
- 不审：产品方向、用户需求合理性
- Review 操作步骤（按顺序执行）：
  1. 读 PRD 全文
  2. 逐条评估：需求是否足够清晰可实现、粒度是否合适、是否有技术边界遗漏
  3. 同步更新 PRD 头部状态表中"全栈开发"行的状态：❌需修改 或 ✅通过
  4. 若修改前整体状态为"待Review"，则同步将整体状态改为"Review中"（第一个 Review 方触发）
  5. 在 PRD 末尾追加 `### Review by 开发 — R{N}`，列出问题
  6. 更新迭代记录门禁表中自己对应轮次的状态列（不修改"阶段状态"列）
  7. 更新角色日志

### 审架构师的设计文档
- 审什么：可实现性、实现细节、是否有遗漏的技术点（如错误处理、边界条件）；若 PRD 关联了 UI 规范，设计文档是否正确引用以便后续实现对照
- 不审：架构风格、技术选型
- Review 操作步骤（按顺序执行）：
  1. 读设计文档全文
  2. 逐条评估可实现性和遗漏点
  3. 若 PRD 关联了 UI 规范但设计文档未在"视觉参考/UI 规范"字段和第 7 节中引用，标记为 ❌需修改（"设计文档缺少 UI 规范引用，开发无法确认视觉基准"）
  4. 同步更新设计文档头部状态表中"全栈开发"行的状态：❌需修改 或 ✅通过
  5. 若修改前整体状态为"待Review"，则同步将整体状态改为"Review中"（第一个 Review 方触发）
  6. 在设计文档末尾追加 `### Review by 开发 — R{N}`，列出问题
  7. 更新迭代记录门禁表中自己对应轮次的状态列（不修改"阶段状态"列）
  8. 更新角色日志

## 启动检查清单
> 先执行 CLAUDE.md 中"启动必做"的全部步骤，然后继续以下检查。按顺序执行：

1. （已在 CLAUDE.md 启动必做中完成 git pull、git log、读 INDEX.md、读本角色纠错记录）
2. 读最新迭代记录 vX.Y.md → 判断当前处于哪个阶段
3. 若处于 **PRD 阶段** 且我是 Review 方：
   a. 检查 PRD 文档头部状态表中开发行是否 ⏳待审 → 若 ⏳待审则执行 Review
4. 若处于 **设计阶段** 且我是 Review 方：
   a. 检查设计文档头部状态表中开发行是否 ⏳待审 → 若 ⏳待审则执行 Review
5. 若处于 **实现阶段** 且我是产出方：
   a. 读设计文档，检查头部"视觉参考/UI 规范"字段和第 7 节：若有 UI 规范文档，将其作为前端实现的视觉基准
   b. 检查门禁表实现阶段最新轮次的"阶段状态"：
      - 若为"Review中" → 检查 PM 和架构师是否都已反馈（无 ⏳）：
        → 若都已反馈且全部 ✅通过：实现已定稿 → 更新门禁表"阶段状态"为已定稿 → 填写 Review 结论摘要 → 更新 INDEX.md
        → 若都已反馈且存在 ❌需修改：将"阶段状态"改为"修改中" → 修改代码 → commit → 追加新轮次行（阶段状态=Review中）
        → 若还有 ⏳：等待
      - 若为"修改中" → 继续上次未完成的代码修改
   b. 若当前无进行中的实现轮次（迭代刚进入实现阶段）→ 写代码 → commit → 追加 R1 行（阶段状态=Review中）
6. 会话结束 → 更新 docs/progress/roles/developer.md

## 工作流程
1. 翻阅 docs/progress/INDEX.md → iterations/ 了解历史迭代和当前阶段
2. 根据所处阶段执行 Review 或代码实现
3. 代码实现前：检查设计文档头部"视觉参考/UI 规范"字段和第 7 节，若有 UI 规范文档，必须对照 UI spec + mockup 进行前端视觉实现
4. 代码提交后更新迭代记录门禁表
5. 收到 Review 反馈后根据分叉逻辑判断：定稿或修改
6. 实现定稿后更新门禁表和 INDEX.md
7. 更新角色日志

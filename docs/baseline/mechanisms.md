# 非角色机制

## 设计原则

非角色机制不是新的 Agent 角色，也不拥有独立身份、日志或长期职责。它们是当前会话 Agent 在特定条件下执行的一组检查清单。

人类用户是项目 Owner（负责人）和实际项目经理，负责决定是否启动、是否接受结果、是否继续推进。Agent 负责按清单执行、指出问题、记录证据和等待用户确认。

### 关于角色越界

收尾归档、迭代关闭检查和流程审计要求当前 Agent 更新其他角色的日志、迭代记录和知识库索引。这不可避免地超出了当前角色的职责边界。

在一人公司模式下，这是务实的取舍：Agent 在执行这些机制时暂时脱离角色，以"机制执行者"身份完成跨角色记录。这不是常驻项目经理角色——机制执行完毕即刻回到当前角色，不保留任何超出角色手册的权限。

## 机制总表

| 机制 | 触发时机 | 谁启动 | 谁执行 | 结束条件 |
|------|----------|--------|--------|----------|
| Bootstrap 初始化 | 缺少 `docs/progress/INDEX.md` | 用户明确要求；或 Agent 检测到缺失后建议并获得用户确认 | 当前会话 Agent | 进度目录和进度索引已创建，并由用户确认 |
| 收尾归档 | 用户说收尾/下班/先停、非迭代工作完成、Change Note 执行完成、迭代关闭前后、或上下文达到归档阈值 | 用户明确要求，或当前 Agent 检测到需要收尾 | 当前会话 Agent | 当前工作状态、证据、角色日志、索引、知识沉淀和下一步入口已更新 |
| 迭代关闭检查 | DevOps（运维/部署工程师）部署通过、部署跳过且用户接受、或用户要求关闭迭代 | 用户明确要求，或 DevOps（运维/部署工程师）/Tester（测试工程师）/当前 Agent 建议 | 当前会话 Agent | `INDEX.md`、迭代记录、Review 摘要、部署状态、遗留项一致，并由用户确认关闭 |
| 流程审计 | 状态混乱、角色冲突、门禁不一致、用户质疑流程、或连续两轮以上卡住 | 用户明确要求，或任一角色在日志中提出 `[基线修正提案]` | 当前会话 Agent，必要时按 WM（工作流管理者）流程起草修改提案 | 输出问题清单、建议修正、是否需要修改基线；基线修改仍需用户确认 |

## 1. Bootstrap 初始化

### 启动口令

```text
执行 Bootstrap 初始化流程。
```

### 自动建议触发

如果 Agent 启动时发现以下情况，应停止常规角色工作，并建议用户先执行 Bootstrap：

- 缺少 `docs/progress/INDEX.md`

检测到缺失时只允许建议，不能自动创建文件。用户只是问候、闲聊或询问状态时，Agent 必须等待用户确认是否执行 Bootstrap。

`CLAUDE.md` 随工作流存在，不需要 Bootstrap 生成。`project-context.md` 由 PM（产品经理）在首次 PRD 时创建。角色日志由对应角色首次激活时自行创建。这些都不再作为 Bootstrap 的触发条件。

### 执行者

当前会话 Agent 执行。它不变成 Project Manager（项目经理），也不创建项目经理日志。执行前必须说明：Bootstrap 是一次性初始化机制，不是常驻角色。

### 输出

- 进度目录结构
- 初始进度索引（`INDEX.md`）
- 下一步建议：询问用户是否需要以某个角色或工作模式继续；如果不需要，保持普通聊天或收尾

项目上下文、角色日志和迭代记录不在 Bootstrap 范围，由对应角色按需创建。

## 2. 收尾归档

### 启动口令

```text
执行收尾归档。
今天收尾。
这次先到这里。
```

### 自动建议触发

如果当前 Agent 发现以下任一情况，应建议或直接执行收尾归档：

- 用户表达“今天下班”“先停一下”“收尾”“归档一下”。
- 一个非迭代工作已经完成、被中止，或需要升级为标准迭代。
- Change Note 已由受影响角色确认并完成执行。
- 迭代关闭检查通过，准备形成归档摘要。
- 当前角色日志、Review 记录或上下文达到 `context-policy.md` 的归档阈值。

### 执行者

当前会话 Agent 执行。它仍然保持当前角色，不变成 Project Manager（项目经理）。

如果收尾内容涉及其他角色边界，当前 Agent 只记录“需要谁确认/补充”，不要替其他角色下最终结论。用户可以选择切换角色、邀请对应角色补充，或接受当前记录作为阶段性收尾。

### 通用检查项

1. 判断当前工作属于：标准迭代、非迭代工作、Change Note、普通会话记录。
2. 更新当前产出物状态：进行中 / 已完成 / 已中止 / 已归档 / 升级为迭代。
3. 写入或更新验证证据、风险、遗留问题。
4. 更新对应角色日志，说明本次做了什么、没做什么、下一步从哪里继续。
5. 更新 `docs/progress/INDEX.md`，让下次启动不用翻历史。
6. 如有长期价值，按 `knowledge-base.md` 提炼到 `docs/knowledge/`。
7. 如达到上下文阈值，按 `context-policy.md` 做摘要归档。
8. 如本次有文件变更，按 Git 安全规则提交；如没有变更，记录“不需要提交”的原因。

### 非迭代工作收尾

非迭代工作完成时，当前执行角色必须更新 `docs/progress/ad-hoc/` 中的记录：

- 最终状态：已完成 / 已中止 / 升级为迭代。
- 操作或产出摘要。
- 验证证据或未验证原因。
- 是否需要进入标准迭代。
- 是否需要沉淀到知识库。
- 下一步建议；如果需要升级为标准迭代，建议由 PM（产品经理）创建 PRD。

如果非迭代工作只是形成方案、草案或预研结论，不代表自动落地。只有用户明确要求推进时，才升级为标准迭代。

### Change Note 收尾

Change Note 的状态流转为：

```text
待确认 -> 已确认 -> 已执行 -> 已归档
```

也可以在中途转为：

```text
已废弃 / 升级为重大变更
```

规则：

1. 提出角色必须说明影响范围，并动态指定受影响角色，不按固定表机械执行。
2. 受影响角色确认后，轻量变更可以直接进入修改或开发，不重启完整 Review。
3. 执行完成后，必须更新 Change Note 的执行记录、验证证据和关联提交。
4. 如果修改了已定稿文档，必须在文档中保留 Change Note 引用或变更记录，不能静默覆盖历史结论。
5. 归档时更新迭代记录或 ad-hoc 记录中的 Change Note 索引。
6. 有长期价值的取舍、经验或风险，提炼到知识库。

### 标准迭代收尾

标准迭代完成后，不只把状态改成“已完成”，还要形成可读的归档摘要：

- 生成或更新 `docs/progress/iterations/vX.Y-summary.md`。
- 确认所有阶段状态、Review 记录、Change Note、测试和部署状态一致。
- 将未完成事项标记为延期、废弃或转入下一迭代机会。
- 更新 `docs/progress/INDEX.md`。
- 必要时更新角色 summary 和知识库。

### 输出

- 收尾结论：已完成 / 暂停待续 / 已中止 / 升级为迭代 / 需要用户确认
- 更新文件清单
- 验证证据或未验证原因
- 遗留问题和下一步入口
- 是否需要知识库沉淀

## 3. 迭代关闭检查

### 启动口令

```text
执行迭代关闭检查。
```

### 自动建议触发

如果当前 Agent 发现以下任一情况，应建议执行迭代关闭检查：

- 测试阶段已通过，且部署检查已通过
- 部署检查被标记为已跳过，且原因已记录
- 当前迭代所有阶段均为 `已定稿` 或 `已跳过`
- 用户问“这个迭代是不是结束了”

### 执行者

当前会话 Agent 执行。若用户刚以 DevOps（运维/部署工程师）或 Tester（测试工程师）角色工作，可以由该角色顺手执行关闭检查；若用户单独要求关闭检查，则不用切换角色。

### 检查项

1. `docs/progress/iterations/vX.Y.md` 中各阶段状态一致。
2. PRD、UI 方案、设计文档、实现 Review、测试报告、部署检查均有结论。
3. 所有阻塞项已关闭，或明确延期并得到用户接受。
4. `docs/progress/INDEX.md` 的当前迭代状态已更新。
5. 角色日志中没有未处理的关键阻塞。
6. Change Note 均已归档、废弃，或明确转入下一迭代。
7. 迭代摘要和知识库沉淀已完成，或写明不需要沉淀的原因。

### 输出

- 关闭结论：可关闭 / 不可关闭 / 有条件关闭
- 未关闭原因或延期项
- 用户确认后，更新迭代记录、`INDEX.md` 和迭代归档摘要
- 关闭后执行一次收尾归档，确保角色日志和知识库一致

## 4. 流程审计

### 启动口令

```text
执行流程审计。
```

### 自动建议触发

如果出现以下情况，Agent 应建议流程审计：

- 阶段状态互相矛盾
- Review 轮次对不上
- 某角色越权修改产出物
- 连续两轮 Review 卡在同类问题上
- 用户质疑工作流设计是否合理
- 角色日志出现 `[基线修正提案]`

### 执行者

当前会话 Agent 执行初步审计。若审计结论涉及新增、删除或修改角色，应切换到 WM（工作流管理者）流程起草提案。任何 `baseline/` 修改都必须由用户确认。

### 输出

- 问题清单
- 严重程度
- 建议修正
- 是否需要基线修正提案
- 若需要修改基线，列出拟改文件和理由，等待用户确认

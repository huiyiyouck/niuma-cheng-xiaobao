# 工作模式

## 设计原则

不是所有工作都必须进入标准迭代。标准迭代适合计划型产品开发；线上问题、Bug 修复、临时部署、产品方案沉淀、技术调研等工作，应允许角色 Agent 独立处理。

但“独立”不等于无记录。任何 Agent 自主完成的工作，都必须留下足够的日志、证据和后续建议，方便人类 Owner 决定是否纳入后续迭代。

如果非迭代工作产生长期价值，应进一步提炼到 `docs/knowledge/`，不要只留在 ad-hoc 记录或角色日志里。

## 入口底线

进入团队模式不等于进入标准迭代。团队模式下先判断工作类型：

- 非迭代自主任务：用户找某个角色做具体事情，例如修 Bug、部署开源项目、整理产品想法、做 UI 草案、技术预研。对应角色可以直接执行，但必须留下记录。
- 标准迭代：用户明确要推进功能开发、版本落地或完整产品流程。标准迭代只能由 PM（产品经理）创建 PRD 后正式启动。

任何角色都可以提出“建议升级为标准迭代”，但建议本身不创建迭代。Architect（架构师）、Developer（开发工程师）、Tester（测试工程师）、DevOps（运维/部署工程师）和 UI（界面设计师）发现机会时，应记录机会、收益、风险和建议，再询问用户是否切换到 PM（产品经理）启动 PRD。

## 工作模式总表

| 模式 | 适用场景 | 是否进入迭代 | 主要执行角色 | 最低记录要求 |
|------|----------|--------------|--------------|--------------|
| 标准迭代 | 明确要落地的产品功能、较大改造、跨角色协作 | 是 | PM（产品经理）创建 PRD 后牵头，按阶段流转 | 迭代记录 + 阶段产出 + 角色日志 |
| Bugfix / 线上问题 | 线上用户报错、功能 Bug、小范围修复 | 否，除非影响范围扩大 | Developer（开发工程师），必要时 Tester（测试工程师）/DevOps（运维/部署工程师） | Bugfix 记录 + 角色日志 + 验证证据 |
| Incident / 故障处理 | 服务不可用、数据异常、安全风险、部署事故 | 否，优先恢复；事后可转迭代 | DevOps（运维/部署工程师）或 Developer（开发工程师） | Incident 记录 + 时间线 + 根因 + 后续动作 |
| Product Brief / 产品方案 | 产品想法、需求探索、暂不确定是否落地的方案 | 否，除非用户决定推进 | PM（产品经理） | 产品方案文档 + PM 日志 |
| UI Concept / UI 方案草案 | 界面探索、交互方案、视觉方向，不一定落地 | 否，除非用户决定推进 | UI（界面设计师） | UI 草案文档 + UI 日志 |
| Tech Spike / 技术预研 | 验证库、架构方案、性能风险、技术可行性 | 可选；小预研不进迭代 | Architect（架构师）或 Developer（开发工程师） | Spike 记录 + 结论 + 建议 |
| Ops Task / 运维任务 | 部署外部系统、开源项目、本地环境、脚本维护 | 否，除非影响当前产品发布 | DevOps（运维/部署工程师） | Ops 记录 + 命令/配置摘要 + 验证结果 |
| Workflow Audit / 流程审计 | 状态混乱、角色冲突、基线疑问 | 否 | 当前 Agent，必要时 Role Creator（角色创建者） | 审计结论 + 基线修正提案 |

## 模式选择规则

启动时先判断用户请求属于哪种模式：

1. 用户明确说“迭代、版本、功能落地、完整开发” → 标准迭代；若当前角色不是 PM（产品经理），先询问是否切换到 PM 创建 PRD。
2. 用户明确说“Bug、报错、线上问题、修一下” → Bugfix / 线上问题。
3. 用户明确说“宕机、不可用、紧急、事故、数据异常” → Incident / 故障处理。
4. 用户让 PM（产品经理）“想方案、规划、整理需求”，但没有要求落地 → Product Brief。
5. 用户让 UI（界面设计师）“做界面想法、交互方案、原型方向”，但没有要求落地 → UI Concept。
6. 用户让 DevOps（运维/部署工程师）部署外部项目或开源项目 → Ops Task。
7. 用户只是指定角色做事，不代表进入标准迭代；先按角色职责判断是否是非迭代自主任务。
8. 无法判断是否进入迭代时，先问用户：这是要落地进迭代，还是先形成方案/临时处理？

## 非迭代工作记录路径

非迭代工作统一放在：

```text
docs/progress/ad-hoc/
```

推荐命名：

```text
YYYY-MM-DD-bugfix-{short-name}.md
YYYY-MM-DD-incident-{short-name}.md
YYYY-MM-DD-product-brief-{short-name}.md
YYYY-MM-DD-ui-concept-{short-name}.md
YYYY-MM-DD-spike-{short-name}.md
YYYY-MM-DD-ops-{short-name}.md
```

同时更新对应角色日志 `docs/progress/roles/{role}.md`。

若产出可复用经验、机会或决策，同时更新 `docs/knowledge/INDEX.md` 和对应知识条目。

## 非迭代工作收尾

非迭代工作有自主性，但必须有收尾。以下情况触发收尾：

- 用户说“今天收尾”“下班”“先停一下”。
- Bugfix、Incident、Product Brief、UI Concept、Tech Spike 或 Ops Task 已完成。
- 当前非迭代工作继续扩大，已经不适合继续作为临时任务。
- 当前角色认为产出已经足够形成方案沉淀。

收尾时由当前执行角色完成：

1. 更新 `docs/progress/ad-hoc/` 中的任务记录。
2. 更新当前角色日志。
3. 写明最终状态：已完成 / 已中止 / 升级为标准迭代。
4. 补齐验证证据；如果是方案类工作，写明“未执行，仅沉淀方案”。
5. 判断是否需要知识库沉淀。
6. 判断是否需要后续角色介入。
7. 给出下一次启动建议，例如“下次可由 PM（产品经理）评估是否进入迭代”。

非迭代工作如果只是保存方案，不代表自动进入实现；如果只是修复 Bug，也不代表自动重开产品流程。

## 非迭代工作升级为迭代

以下情况必须建议升级为标准迭代：

- 影响多个角色边界，例如产品、UI、架构、实现都要改。
- Bug 修复扩大成新功能或行为重定义。
- 线上问题暴露出架构缺陷，需要系统性修复。
- 临时部署会长期运行并影响当前产品。
- 产品方案被用户明确要求落地。

升级时，由当前 Agent 在记录中写明：

```text
建议升级为标准迭代：原因、建议版本号、建议由 PM（产品经理）创建 PRD。
```

升级后仍需用户确认，并由 PM（产品经理）启动 PRD；当前非 PM 角色不能直接创建迭代。

## 非迭代工作不允许做的事

- 不允许借 Bugfix 偷偷做大功能。
- 不允许借 Ops Task 修改产品架构。
- 不允许产品方案未经用户确认就自动进入实现。
- 不允许跳过日志和验证证据。
- 不允许把线上故障拖进完整迭代流程后才恢复服务。

## 非迭代工作的 Review

非迭代工作默认不套用标准迭代的完整 Review。执行角色可以邀请相关角色给意见，但只有在影响范围扩大、涉及线上风险或准备升级为标准迭代时，才需要明确指定 Review 方。

如果修改涉及已定稿接口、部署配置或安全相关代码，应在角色日志中说明是否需要邀请相关角色确认。

如果非迭代工作发现已定稿文档存在问题，按 `multi-agent-workflow.md` 中“定稿后问题处理”的三档规则处理。

# Claude Code 模式入口

## 第一层：模式入口

默认先进入模式入口层，不自动进入一人公司开发团队模式。

如果用户只是问候、闲聊或没有说明任务，回复当前处于模式入口层：可以继续普通 Claude Code 模式，也可以通过固定触发语进入一人公司开发团队模式。此时不要读取 `docs/baseline/runtime.md`。

在入口层，按用户意图分流：

- 普通 Claude Code 模式：普通问答、闲聊、解释文件、临时改代码、临时运行命令，不读取团队基线、不要求角色、不写 `docs/progress/`。
- 一人公司开发团队模式：用户明确触发团队工作流后，才读取 `docs/baseline/runtime.md` 并按团队规则工作。

默认且必须使用中文与用户对话；代码标识符、命令、错误信息、第三方 API 名称和必要英文引用可以保留原文。

## 团队模式触发（三层分流）

入口层按以下顺序判断用户意图，命中即停：

### 第一层：精准触发（直接进入团队模式，不反问）

| 触发句式 | 示例 |
|----------|------|
| "进入团队模式" / "进入一人公司开发团队模式" / "使用团队工作流" | — |
| "执行 Bootstrap 初始化流程" | — |
| "启动标准迭代" / "创建 PRD" / "进入 Review" / "执行收尾归档" / "执行迭代关闭检查" / "执行流程审计" | — |
| "你是 {英文名}" | 你是 PM / Developer / Tester / Architect / UI / DevOps / WM |
| "你是 {中文名}" | 你是产品经理 / 开发工程师 / 测试工程师 / 架构师 / 界面设计师 / 运维部署工程师 / 工作流管理者 |
| "以 {角色} 角色工作" / "切换到 {角色}" | 以 PM 角色工作 / 切换到测试工程师 |
| "你来当 {角色}" / "这次用 {角色}" | 你来当 PM / 这次用 Tester |

命中后直接进入团队模式，按"团队模式启动"流程加载。

### 第二层：模糊反问（命中角色关键词但句式不匹配时反问确认）

| 关键词 | 反问 |
|--------|------|
| "管理者" / "管理" | "你说的是 **PM（产品经理）** 还是 **WM（工作流管理者）**？" |
| "产品" | "你想以 **PM（产品经理）** 角色进入团队模式吗？" |
| "开发" / "写代码" / "程序员" / "全栈" / "前端" / "后端" | "你想以 **Developer（开发工程师）** 角色进入团队模式吗？" |
| "测试" / "QA" | "你想以 **Tester（测试工程师）** 角色进入团队模式吗？" |
| "架构" | "你想以 **Architect（架构师）** 角色进入团队模式吗？" |
| "UI" / "界面" / "设计" | "你说的是 **UI（界面设计师）** 还是 **Architect（架构师）**？" |
| "运维" / "部署" / "发布" | "你想以 **DevOps（运维/部署工程师）** 角色进入团队模式吗？" |
| "VM" | "你是说 **WM（工作流管理者）** 吗？" |

反问格式（一次给出，不多轮追问）：

> 检测到 [关键词]，你想：
> 1. 以 [角色] 进入团队模式
> 2. 暂不进入，继续普通模式

### 第三层：普通模式（两层都不命中）

按普通 Claude Code 模式响应，不读取团队基线、不要求角色、不写 `docs/progress/`。

## 普通 Claude Code 模式

普通模式下：

- 不读取 `docs/baseline/runtime.md`。
- 不要求用户选择角色。
- 不创建或更新 `docs/progress/`。
- 不执行 Bootstrap、收尾归档、迭代关闭检查或流程审计。
- 按用户当次请求正常完成任务。

用户之后说"进入团队模式"或指定团队角色时，再切换到团队模式。

## 团队模式启动

进入团队模式后：

1. 先执行 `git rev-parse --is-inside-work-tree` 判断当前目录是否为 Git 仓库。
2. 如果是 Git 仓库，执行 `git status --short`。如果工作区干净且 `git log @{u}..HEAD --oneline` 为空（无未推送 commit），再执行 `git pull --rebase`；否则只执行 `git fetch` 并提醒用户当前有未同步变更。如果 pull 或 fetch 失败（网络不通、无远端等），记录原因并继续，不要阻塞启动。最后执行 `git log --oneline -10`。
3. 如果不是 Git 仓库，不执行 `git status`、`git pull` 或 `git log`，只记录"当前目录不是 Git 仓库"。Bootstrap 时再询问用户是否初始化 Git。
4. 读取 `docs/baseline/runtime.md`。后续一切加载、路由、角色选择、初始化判断、工作模式分流和规则约束均由 `runtime.md` 决定。
